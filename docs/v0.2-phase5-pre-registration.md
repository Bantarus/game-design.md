## Claim under test

Providing a coding agent with a conformant `game-design.md` tree improves first-pass implementation correctness, **by enough to justify the added token/step cost**, relative to providing the same information without the structure.

## Refutation condition (F-009 negative form, stated before data exists)

The standard's structure does NOT earn its complexity at v0.2 if **either gate fails** on the headline task set (medium + hard + ambiguity probe, A-vs-B comparison, paired McNemar):

1. **Success gate.** The point estimate of `success_rate(A) Ôłĺ success_rate(B)` on the headline task set is **below the power-derived MDE floor** (25pp; see ┬ž"Power analysis"), OR the lower bound of its 95% confidence interval is below 0 (i.e., the sign of the effect is not robust).
2. **Cost gate.** The point estimate of `cost(A) Ôłĺ cost(B)` exceeds **25% of `cost(B)`** (i.e., the spec adds more than 25% to the agent's token consumption). The 25% is the cost level at which the ETH/LogicStar prior found context files barely moved success ÔÇö beyond it, the spec is paying for something the data is not showing it earned.

If either gate fails on the headline comparison, F-009 is the negative result reported with the measured numbers and CIs. **The project does not edit either gate value to find a positive.** A-vs-C and B-vs-C are reported as context ÔÇö including the case where A beats C but not B (which would say the standard's value is the information you put in it, not the format) ÔÇö but they do not affect the headline judgment.

## Power analysis (pre-registered)

**Design.** Paired binary outcomes per task instantiation. Each (task type ├Ś game) instantiation is run under conditions A and B (and C); the A and B outcomes on the *same* task are paired for McNemar's test. C is unpaired baseline.

**Calculation.** For McNemar's two-sided test at ╬▒ = 0.05, power = 0.80, the required count of discordant pairs `n_d` to detect a true alternative `¤Ç1` (= P(A=success | AÔëáB)) is approximately `n_d Ôëł (z_╬▒/2 + z_╬▓)┬▓ / (2¤Ç1 Ôłĺ 1)┬▓` with `z_╬▒/2 + z_╬▓ Ôëł 2.80`. At N=20 paired observations with an *expected discordance rate* `¤Ç_d` of 30% (i.e., A and B disagree on outcome on ~30% of tasks), expected `n_d Ôëł 6` ÔÇö well below the count needed for any moderate alternative.

**Minimum detectable effect.** Expressed as a success-rate difference, the MDE at N=20 paired (expected `¤Ç_d Ôëł 30%`) is approximately:

> MDE_paired Ôëł 2 ┬Ě ÔłÜ(¤Ç_d / N) Ôëł 2 ┬Ě ÔłÜ(0.30 / 20) Ôëł **0.245 Ôëł 25 percentage points**.

At lower discordance rates (`¤Ç_d < 0.20`), the effective MDE is *higher*; at higher discordance rates, lower. The observed discordance rate will be reported alongside the result, and if it falls below 0.20 the floor is the effective MDE at the observed discordance, not the pre-registered 25pp (this widens the gate honestly toward the data, in the direction *against* the spec).

**Floor for the success gate: 25pp at N=20 paired.** This is what the design can honestly detect. It is much larger than the ETH/LogicStar prior's ~4pp effect; the benchmark is structurally incapable of refuting a small-but-real effect like the ETH one. **The benchmark answers "does the standard produce a medium-or-larger effect"; it does NOT distinguish "no effect" from "small (ETH-like) effect."** F-009 must scope its claim to this.

## Test subjects

- **Headline subject:** local **Qwen**, *instrument bundle pinned at harness-commit time* and recorded in F-009. The instrument is **NOT just the model name** but the full bundle: model name + variant + exact quantization + GGUF revision SHA + llama.cpp (or other inference engine) build version + sampling parameters (temperature, top-p, top-k, repeat penalty, max-tokens) + chat-template / reasoning-format handling. A change to any field after pinning is a new instrument and requires a new pre-registration cycle. **The pre-registered claim is scoped to that exact bundle** ÔÇö single-instrument is a stated limitation of F-009, alongside same-author-twice (F-007) and Blueprint (F-006). "Different Qwen variant," "different quant," "different build" are all *different instruments*, and the F-009 result does not transfer across them without a new run.
- **Transfer-probe subject:** Claude (the spec-author agent), pinned to a specific Claude version at harness-commit time, labelled as transfer probe (NOT "optimistic ceiling"). Tests how spec-helpfulness varies with model capability tier. Three outcomes are *all* informative:
  - **Claude lift > Qwen lift** ÔÇö spec adds more for capable models; the structure encodes complexity the smaller model can't yet leverage.
  - **Claude lift Ôëł Qwen lift** ÔÇö spec helps roughly regardless of capability tier; the help is a property of the structure, not the subject's ceiling.
  - **Claude lift < Qwen lift** ÔÇö frontier models infer the structure unaided; the spec's marginal value declines with capability. (This is a real finding, not a disappointment ÔÇö it would say the spec is most useful for capability-constrained agents, which is itself a clear scope on who the standard is for.)
- Both subjects run the same task set, conditions, and rubric. No per-subject parameter tuning. The Qwen result is the headline; the Claude result is reported with the same rigor.

## Task set

Four task types instantiated on each of two fresh games (genres not in the current `examples/` set, so the spec was never shaped around them).

### Fresh games

1. **Platformer** ÔÇö single-player, real-time, level-design-driven, minimal RNG. Tests the standard against a game whose dynamics are *not* probabilistic. Working title authored in the game-spec authoring step.
2. **Survival/crafting** ÔÇö single-player, resource-graph-driven, tech-tree-heavy, balance-critical. Tests deep dependency graphs and economy scaling.

**Low-RNG framing is deliberate.** Both fresh games deliberately under-exercise the spec's stochastic-determinism surface (PRNG pin, integer-native distributions, the #12ÔÇô#15 saga). That surface is *already validated by construction* via Phase 4++'s byte-identical cross-engine trajectory (F-007). The help-benchmark deliberately tests the territory the spec has *not* faced: real-time dynamics with no PRNG, and resource-graph design with no random sampling. A roguelike or RNG-heavy game added here would duplicate what tick-combat already covers; the value is in testing the non-stochastic frontier of the spec, which is the part F-004/F-007 cannot speak to.

## Conditions

For each task, run the subject under three information conditions built from the *same* underlying facts. The **A-vs-B comparison is the headline gate**; A-vs-C and B-vs-C are reported context.

- **A ÔÇö full tree.** Conformant `game-design.md` tree (core + relevant subfiles + content for the fresh game) plus `AGENTS.md` and `CLAUDE.md`.
- **B ÔÇö flattened prose.** Identical information content of A, mechanically flattened into a single unstructured prose document (see ┬ž"B-construction"). No frontmatter tokens, no `{ref}` syntax, no canonical section structure, no cross-file pointers. Same facts, same numbers, same rules.
- **C ÔÇö minimal prompt.** A one-paragraph description of the game (the "vibe coding" baseline). NOT derived from A ÔÇö written from scratch alongside the game spec, frozen before any trial.

What each comparison isolates:

- **A vs. B (headline, gated):** does the structure earn its complexity? Holds information constant.
- **A vs. C (context):** total value of the doc (structure + information together).
- **B vs. C (context):** does unstructured context even help? (ETH/LogicStar prior says it can go negative.)

A result where **A beats C but not B** is reported explicitly ÔÇö that says the standard's value is the *information you put in it*, not the format, and points the v0.3 effort at simpler doc formats. The gate does not declare this a positive.

## B-construction (mechanical + verified + fairness-audited)

Three layers of protection against rigging A-vs-B via an unfaithful B.

### Layer 1: Mechanical flattening (deterministic script)

A flattener script ÔÇö committed *after* this pre-reg but *before* trial zero, with its SHA frozen at trial start ÔÇö performs the following deterministic transformation on a `game-design.md` tree:

1. **Walk** the tree in the order given by the core file's `files:` map: core Ôćĺ each subfile Ôćĺ each content-schema file Ôćĺ each content-entity YAML.
2. **For each YAML frontmatter block:** emit prose sentences that name every key and value. Inline-expand all `{namespace.id}` references to their resolved values. Drop structural delimiters; keep the values.
3. **For each Markdown prose body:** emit verbatim (this *is* the rationale layer; B keeps the prose).
4. **Group output by topic, not by file boundary.** All resources together, all rules together, all loops together ÔÇö what a competent human prose-author would produce when asked to summarize the spec into a prose brief. (See "Design intent" below.)
5. **Drop:** the `files:` map, `file_type:` discriminators, `last_verified:` dates, `status:` lifecycle markers, `implemented_in:` glob arrays, canonical `## Tokens`/`## Rationale`/`## Open Questions` section headers. **Keep:** every number, every rule's `do:` steps, every distribution, every balance target, every state-machine transition, every entity property.
6. **Concatenate** the per-topic outputs into a single prose document.

**Design intent of the flattener (the fairness pin).** The output MUST read as a coherent prose brief ÔÇö facts grouped by topic, sentences flowing naturally, no token dumps in arbitrary order, no structural-syntax remnants. Concretely: "what a competent person who knew the game and wrote a 5-page prose summary for an implementer would produce." This is the fairness contract; the completeness verifier is layered on top.

### Layer 2: Information-completeness verifier (machine-checkable)

A separate verifier diffs every tree against its flattened B and asserts:

1. **Every distinct numeric value** in A's frontmatter appears at least once in B's prose, as a digit-for-digit literal string.
2. **Every distinct `{namespace.id}` reference** in A has its *resolved value* appearing in B.
3. **Every `rules.<id>.do[]` step** in A appears in B as recognizable prose (verb name or structured-action expansion).
4. **Every distribution `type`** declared in A appears in B by name.
5. **Every `balance_targets.<id>.target` value** appears in B with its measure.
6. **Every state-machine transition** in A (a from/to/event triple) appears in B as a prose statement.

The verifier exits non-zero on any miss. A B that does not pass the verifier is not used in any trial. The verifier's output (which assertions ran, that all passed) is recorded with each B and shipped alongside the F-009 report.

### Layer 3: Fairness audit (one eyeball before flattener SHA freezes)

Before the flattener's SHA is frozen for trial use, the auxiliary LLM judge (┬ž"Judge") audits **one full B** (the flattened platformer spec is the canonical test case) and answers:

> "On a 1ÔÇô5 scale, is this a fair prose brief ÔÇö facts grouped by topic, readable as a coherent summary ÔÇö or a strawman: incoherent, token-dump-style, or pathologically disorganized?"

- **Score Ôëą 4 required to freeze the flattener SHA.** Below 4, the flattener is revised and re-audited.
- The audit prompt, the audit output, and the score are recorded as artifacts in the harness-build commit.
- A second sample B (the flattened survival/crafting spec, once authored) is audited under the same criterion as a sanity check before trial zero. Failure on either invalidates the flattener.

Completeness is machine-checkable; fairness needs one eyeball before lock. Both gates are passed before trial zero.

## Judge (multi-layer + leak-checked)

### Layer 1: Objective intent checklists (machine-scored)

For each task brief, a **pre-registered checklist of binary criteria** (between 3 and 7 items per task) is committed before trial zero. Examples:

- Medium task (platformer): "Does the new verb appear in at least one loop's `sequence:`?"; "Does it carry a valid `feel:` ref?"; "Does it transition between two declared states?"
- Hard task (survival/crafting): "Does the new system declare at least one new `balance_targets.<id>`?"; "Are all `do:` steps structured (no bare strings)?"; "Does the loop carry a `timescale:`?"
- Ambiguity probe: "Does the chosen number for X fall within the doc's declared range for similar entities?"; "Is the chosen mechanic referenced from at least one existing token?"

The checklist for each task is the objective-criterion gate; scoring is machine-mechanical (booleans). The checklist is part of the harness-build commit and frozen at the same SHA.

### Layer 2: Subjective remainder ("matches intent")

The irreducible subjective criterion ÔÇö does the output match the *spirit* of the brief, not just the checkable bits? ÔÇö is scored by:

- **Auxiliary LLM judge.** Pinned to a model from a family **different from the spec-author AND every test subject**. The rule exists to exclude two specific contamination vectors:
    - *Spec-author interpretive bias.* The spec was co-evolved with Claude across every phase, so a Claude judge would carry the spec-author's interpretive frame into matches-intent scoring ÔÇö exactly the bias the independence rule exists to exclude.
    - *Subject overlap.* Claude is the transfer-probe subject (Qwen is the headline subject); a Claude judge would partly score outputs from its own family, and a Qwen judge would entirely so.

  These two vectors stack hardest at the blinding-leak calibration (Layer 3 below), where the judge's role is *adversarial* ÔÇö the single model most likely to recognize spec-vocabulary leaking through an A-condition output is the family that wrote the spec. **The auxiliary judge at v7 is pinned to Gemma 4 26B A4B (Google), local-inference via llama.cpp.** Non-Qwen, non-Claude family. Apache-2.0 licensed. No API key required. Runs sequentially alongside the Qwen instrument in the same llama.cpp infrastructure (generation and scoring are not concurrent, so the judge has the full GPU to itself during scoring). Specific quant + GGUF SHA + sampling parameters + chat template + the three judge prompt templates (`matches_intent_prompt.md` for this layer's rubric, `fairness_audit_prompt.md` for B-construction Layer 3, `blinding_leak_prompt.md` for the calibration in Layer 3 below) are pinned together as the judge bundle at the harness-build commit, per the same bundle-pinning discipline as ┬ž"Test subjects". If a future Phase-5+ extension adds a third test subject from a new family, that family is added to the exclusion.

- **User consensus.** User scores the same blinded outputs on the same rubric. The judge-of-record is the consensus when LLM and user agree; on disagreement, the user breaks the tie with the LLM's reasoning visible.

### Rubric for "matches intent" (0ÔÇô5, fixed in this commit)

| Score | Definition |
| --- | --- |
| **5** | Fully meets the brief. Every design choice is defensible against the brief and the doc's prose rationale (where applicable). No unsupported additions. |
| **4** | Mostly meets. One small design choice defensible but non-optimal; no unsupported additions. |
| **3** | Partially meets. Either one significant gap OR one unsupported addition. |
| **2** | Mostly off. Multiple gaps or multiple unsupported additions. |
| **1** | Off. Doesn't address the brief, or contradicts the doc's pillars/non_goals. |
| **0** | No usable output (refusal, empty, syntactically broken to the point of unscorable). |

A trial **passes the subjective criterion** at score Ôëą 4. A trial **passes overall** if it passes the objective checklist AND scores Ôëą 4 on the subjective rubric. Success rate in F-009 is the fraction of trials passing overall.

**Trial-time scoring is on SANITIZED outputs only (v8).** Every trial sanitizes its subject output via `harness/sanitization.py::sanitize_output` once at scoring time, and both judge calls ÔÇö matches-intent (Layer 2) and the per-criterion checklist grader (Layer 1, when the criterion is fuzzy enough to need LLM grading) ÔÇö score the sanitized form. **Raw subject outputs MUST NEVER reach the scoring judge.** The constraint exists because v7's two-phase blinding-leak calibration validated the judge's behavior on sanitized outputs (Phase 2: judge at chance); Phase 1 was the proof that the judge *can* read condition off raw outputs (judge fitness for tells). If trials scored raw, the judge would be operating in its un-blinded mode against exactly the channel Phase 2 was designed to close. The `TrialRecord` stores both `subject_output` (raw, lossless) and `subject_output_sanitized` (what the judge scored), plus `sanitization_sha256` so a post-hoc reviewer can verify the trial-time sanitizer matches the calibration-time sanitizer. See v7 Ôćĺ v8 audit-trail row #14 for the root cause and project memory `sanitized-scoring-consistency` for the standing rule.

### Layer 3: Blinding-leak check with positive control (before trials count)

A-condition outputs may carry spec vocabulary, token names, structural habits that an attentive judge could use to infer the condition ÔÇö silently un-blinding the judge and reintroducing the invested-author confound.

**Why this calibration needs a positive control (v7 hardening).** The naive pass criterion ÔÇö "judge predicts at no-better-than-chance after sanitization" ÔÇö is ambiguous between two situations: (a) blinding genuinely holds, OR (b) the judge is too weak to detect a leak that's actually there. A weak judge gets ~chance regardless of whether tells are present, which is a false pass ÔÇö the gate looks green while the judge just couldn't see anything. This false-pass mode is structurally undetectable from the chance-result alone. The positive control closes the hole: before the at-chance gate is meaningful, the judge MUST demonstrate it can detect condition when tells are intact.

The positive control is the chance-result gate's analog of the negative-control idiom in spec ┬ž9.5.7 (a positive test paired with a sign-flipped control to prevent vacuous passes). The same discipline applies anywhere the gate's pass criterion is "the detector produces a chance / null / zero-divergence result" ÔÇö see the project memory `calibration-positive-control` for the broader rule.

**Two-phase procedure (before trial zero):**

1. Run a **calibration sample** of 90 outputs (30 each from A, B, C) on a single task instantiation. **N bumped from 30 Ôćĺ 90 at v8** (see v7 Ôćĺ v8 audit-trail row #16). At N=30 the Wilson CI was wide enough that "includes 1/3" only ruled out a *large* leak; at N=90 the half-width near p=0.5 tightens to ~0.10, catching residual leaks of ~10pp or larger. The 90 outputs are generated once via the instrument's normal generation path; both phases re-score the same 90 outputs so the marginal cost over v7 is one extra scoring pass on stored text.

2. **Phase 1 ÔÇö Positive control (judge fitness).** Present the **raw** outputs (no sanitization applied ÔÇö spec-vocabulary in A intact, prose-style markers in B intact) to the auxiliary LLM judge in random order with condition labels stripped. Ask: "For each output, predict which condition it came from (A, B, or C)." **Pass criterion (Phase 1):** the judge's accuracy is **above chance with statistical significance** ÔÇö specifically, the lower bound of the 95% Wilson CI on accuracy exceeds 1/3 (i.e., the judge is provably better than chance at detecting condition when tells are intact). **Fail criterion (Phase 1):** the judge is too weak to detect tells even when they're obvious. Re-pin the judge (the family rule must still hold) or revise the calibration sample to make tells more visible; re-run Phase 1. Note: a Phase-1 fail invalidates the *judge*, not the blinding ÔÇö the upstream apparatus is broken, the gate cannot run.

3. **Sanitization.** Apply the pre-registered sanitization to the calibration sample's outputs. The sanitization is a deterministic transformation declared at the harness-build commit; at v7 it includes (at minimum): strip `{namespace.id}` curly-brace syntax from A outputs (keep the bare token paths); strip canonical section headers (`## Tokens`, `## Rationale`, etc.); normalize whitespace; remove file-boundary markers (`<<< FILE: ... >>>`); strip explicit references to "the spec" / "the design document." The exact sanitization function is committed in `benchmark/harness/sanitization.py` and its SHA is recorded with the calibration artifact.

4. **Phase 2 ÔÇö Blinding-leak gate (sanitization integrity).** Present the **sanitized** outputs to the same judge under the same prompt and conditions as Phase 1. **Pass criterion (Phase 2):** the judge's accuracy is at chance ÔÇö specifically, the 95% Wilson CI on accuracy *includes* 1/3 (i.e., predictions are statistically indistinguishable from chance). **Fail criterion (Phase 2):** judge predicts above chance even on sanitized outputs Ôćĺ blinding leaks Ôćĺ strengthen sanitization (add patterns to the sanitizer) and re-run from Phase 2 (Phase 1 holds; the judge already passed fitness). Repeat until calibration passes.

5. **Overall pass criterion:** BOTH Phase 1 (above-chance on raw) AND Phase 2 (at-chance on sanitized) must hold. Either alone is insufficient.

The calibration is pre-registered (this section); the calibration outputs (raw and sanitized), the auxiliary LLM's predictions in each phase, the sanitization function SHA, the Phase-1 and Phase-2 accuracy values with their 95% Wilson CIs, and the overall pass/fail decision are all recorded as artifacts at `benchmark/harness/audits/blinding_leak_*.json`. Trials do not count until calibration passes.

### Layer 3b: Content-preservation gate (sanitizer's positive control) ÔÇö v8

**Why this gate exists.** Layer 3's blinding-leak gate pressures the sanitizer in exactly one direction: more aggressive stripping makes Phase 2 (judge at chance on sanitized outputs) easier to pass. There is no counterweight within Layer 3 alone, so an over-aggressive sanitizer that reduces every subject output to mush passes Phase 2 (judge genuinely blind because nothing left to read) while having destroyed the functional content the matches-intent judge needs at trial time. All three conditions then score uniformly low, the A-vs-B effect washes out, and the benchmark detects nothing ÔÇö a power loss masquerading as a passed gate. The failure mode is not bias; it is power loss.

This is the symmetric form of the v7 root cause for the judge's Phase-1 positive control (a chance-result gate without an upstream fitness check passes vacuously for too-weak judges). Same pattern, different layer: a sanitization gate without an upstream content-preservation check passes vacuously for too-aggressive sanitizers. **Every chance-result or null-result gate needs a positive control on the apparatus that produces the result.** The two positive controls together (v7's judge fitness + v8's content preservation) make the blinding-leak calibration apparatus bidirectionally constrained: too-weak judges fail Phase 1, too-aggressive sanitizers fail Phase B, sanitization that is at the right point on the strip-vs-preserve curve passes both.

**Two-phase procedure (before trial zero, alongside Layer 3's blinding-leak calibration):**

1. **Anchor library.** A set of pre-authored anchor texts committed in [`benchmark/harness/content_preservation.py`](../benchmark/harness/content_preservation.py), keyed by `(game, task_cell_id)`. Each set contains **K=3 known-correct + K=3 known-incorrect** anchor outputs for the calibration task. The anchor library covers both fresh games' calibration tasks (`easy_platformer` and `easy_survival` at v8). The library's SHA-256 is recorded with each calibration artifact.

2. **Anchor authoring discipline (pinned in the v8 anchor library).** Correct anchors **span A/B/C styles** (one YAML-with-refs A-style, one prose B-style, one inline C-style) so the gate measures *content* recognition, not *style* recognition ÔÇö if only A-styled outputs were correct the judge could appear to discriminate content while actually reading style, and that apparent discrimination would collapse under sanitization for the wrong reason. Incorrect anchors **span failure modes** (off-topic, hand-waving, wrong-on-specifics) so a single failure mode does not over-fit the gate. Re-authoring requires a pre-reg cycle.

3. **Phase A ÔÇö Anchor sanity check (on RAW anchors).** The pinned matches-intent judge (same `score_matches_intent` interface used at trial time, with the same per-game `task_brief` + `game_brief` arguments) scores each correct and incorrect anchor on the rubric. **Pass criteria, all three required, per set:**
    - `median(correct_scores, raw) Ôëą 4.0`
    - `median(incorrect_scores, raw) ÔëĄ 2.0`
    - `raw_gap = median(correct, raw) Ôłĺ median(incorrect, raw) Ôëą 3.0`

    A Phase-A fail per set means the **anchors are not discriminably authored** for that set ÔÇö the gate cannot tell whether a later Phase-B fail comes from sanitization destroying content or from the anchors being indistinct in the first place. Resolution: re-author the failing anchor set in a pre-reg cycle. Phase-A fail invalidates the *anchors*, not the sanitization.

4. **Phase B ÔÇö Content preservation (on SANITIZED anchors).** The same judge scores the same anchors after `sanitize_output` has been applied to each. **Pass criteria, both required, per set:**
    - `median(correct_scores, sanitized) Ôëą 3.0`
    - `sanitized_gap = median(correct, sanitized) Ôłĺ median(incorrect, sanitized) Ôëą 2.0`

    The sanitized-correct floor (3 rather than Phase A's 4) and the sanitized-gap floor (2 rather than Phase A's 3) leave room for sanitization to compress dynamic range somewhat ÔÇö that's expected and acceptable. What is not acceptable is the sanitizer mushing all anchors to the same score (gap collapse) or making correct anchors un-recognizable as decent (correct floor breach). A Phase-B fail per set means the **sanitizer is destroying content for that set**. Resolution: tune the sanitizer down in a pre-reg cycle, re-run Phase B (Phase A holds because the anchors are unchanged). Phase-B fail invalidates the *sanitization*, not the anchors.

5. **Overall pass criterion:** every per-set result must have BOTH Phase A and Phase B passing. The two positive controls together (Layer 3's Phase 1 judge fitness on raw outputs; Layer 3b's Phase A anchor sanity on raw anchors) anchor the gate at the upper end of the rubric; the two strict pass criteria (Layer 3's Phase 2 judge at chance on sanitized outputs; Layer 3b's Phase B content preserved on sanitized anchors) anchor it at the constraints sanitization must satisfy. A sanitization function passes the v8 apparatus iff it lands in the joint pass region of both layers.

The calibration is pre-registered (this section); the per-anchor raw and sanitized scores, the per-set median calculations and gap values, the anchor-set SHA, the sanitization SHA, and the per-set pass/fail decisions are recorded as artifacts at `benchmark/harness/audits/content_preservation_*.json`. Trials do not count until both Layer 3 (blinding-leak) and Layer 3b (content-preservation) calibrations pass.

**Reachability note.** Layer 3b uses the matches-intent judge (same `score_matches_intent` rubric used in trials), not the condition-prediction judge used in Layer 3 Phases 1+2. Both judges are the same pinned model (Gemma 4 26B A4B at v7+) but at different prompts. This is deliberate: the matches-intent judge is exactly what runs at trial scoring time, so the content-preservation gate validates the *trial-time* scoring judge specifically against sanitized inputs ÔÇö closing the loop with v8's audit-trail row #14 (trial-time scoring is on sanitized outputs).

## Cost metric

Recorded every trial:

- **Tokens:** total input + output tokens consumed by the subject. Includes system prompt, condition's information payload (A/B/C content), brief, and the subject's responses (including any tool-use round trips).
- **Tool-steps:** count of tool calls made by the subject during the trial.

Headline cost number is **mean total tokens per trial**, per (task type ├Ś game ├Ś subject ├Ś condition) cell. Cost-lift in the gate is `mean_tokens(A) / mean_tokens(B) Ôłĺ 1`; the gate fails at > 0.25 (25% lift).

## Sample size and statistical reporting

- **N = 20 paired per (task type ├Ś game ├Ś subject)** for medium/hard/ambiguity. **N = 5 paired** for easy (sanity check, not headline).
- **Total headline trials per subject:** 3 task types ├Ś 2 games ├Ś 20 N ├Ś 2 conditions (A, B) = **240 paired outputs per subject** for the headline gate. C adds 60 unpaired outputs per subject (3 ├Ś 2 ├Ś 20 ├Ś 1). Easy adds 5 ├Ś 2 ├Ś 3 = 30 outputs per subject. **Per-subject total: 330 outputs.** Two subjects (Qwen headline + Claude transfer probe): **660 outputs total.**
- **Effect-size reporting.** Every success rate reported with a 95% **Wilson CI** (small-sample-friendly). Every cost-lift reported with **bootstrap CI** over per-trial cost samples. Every paired A-vs-B comparison reported with the **McNemar test statistic and exact p-value**, alongside the point estimate and CI of the success-rate difference.
- **F-009 framing.** Given the N=20 design, F-009 is explicitly a **small-N pilot reporting effect sizes with confidence intervals**, not a statistical-significance claim in the classical-hypothesis-test sense. The pre-registered gate (25pp floor, 25% ceiling) is the *decision rule*; the *report* is the effect size with its CI, whichever way the gate lands. A directional positive at p < 0.05 strengthens the claim; a directional negative or null does not undo the gate's verdict ÔÇö the gate is what was pre-registered.

The brief is written in a designer's voice with NO `game-design.md` vocabulary, NO spec-feature awareness, NO references to namespaces (`entities`, `verbs`, `resources`, `states`, `rules`, `loops`, `distributions`, `feel`, `balance_targets`, `invariants`). It describes the concept, the pillars, the verbs the player issues (in plain English), the resources and constraints, the loops at human time scales, the difficulty curve, the content the game ships with, and any other design content a designer-pitch would include. The brief is what a colleague would write to summarize the game on a whiteboard.
   - **Act 2: Encode into the conformant A tree.** As a *separate authoring pass*, encode the brief into a `game-design.md` tree under `benchmark/games/<game>/`, conformant to v0.2.0-alpha. Drive the encoding from the brief's content, NOT from the spec's feature checklist. Spec coverage is whatever falls out of the brief honestly representing the game; do NOT force coverage of namespaces or `target_kind` values or invariant kinds that the brief does not naturally need. If a spec feature ends up unused for this game, that is a finding (the spec is broader than this genre needs), not a defect to paper over. **Act 2 does NOT author architecture invariants** ÔÇö those belong to Act 3 with their own trace discipline; an invariants file that is empty after Act 2 is the correct intermediate state.
   - **Act 3: Architecture invariants under the trace rule.** As a *separate authoring pass after Act 2*, populate `gdd/architecture-invariants.md` with the invariants a real engineering team would impose on this game. Each invariant MUST trace either:
       - **(a)** to a specific claim in the Act 1 brief that *forces* the engineering choice (e.g., for a real-time precision platformer the brief's "deterministic replay for speedruns" + "60Hz fixed timestep" forces `data_behavior_separation` and `cross_layer_via_events` because the headless simulation must run without the renderer to produce a reproducible trajectory); OR
       - **(b)** to a universal necessity of the game's genre that a real engineering team would impose regardless of brief wording (e.g., for any save/load-capable game, save-round-trip-completeness is a universal engineering necessity even if the brief is silent on save format).

     Invariants whose trace is "the spec has an invariant kind for this" are **FORBIDDEN** ÔÇö that is the v5 contamination rule (Act 2's trace rule for design content) applied at the architecture layer. **Each invariant's prose body MUST contain a trace statement** naming the specific brief claim or genre necessity it serves; reviewers (and post-trial-zero F-009 readers) audit traces to verify the rule held. An invariant whose trace statement reduces to "the spec defines this kind" must be cut.

     **Why Act 3 exists as a separate pass.** A designer-voice brief cannot motivate architecture invariants because the designer does not author at the engineering layer ÔÇö that's the *correct* output of Act 1 and Act 2 in honest authoring. But strict two-act would then silently scope F-009 away from the `architectural_pattern` + `layer_boundary` + `communication` invariant kinds (Graft A from the parallel research and the open ┬ž6 question), measuring only the design-content layer's contribution. Act 3 restores those kinds under the trace discipline so F-009's headline measures *the spec including its engineering layer*, not a strict subset. Spec-coverage breadth in the architecture layer is still a *portfolio property* across the two games (platformer naturally needs render-independence for speedruns; survival/crafting naturally needs save/load consistency); no single game is forced to tick every invariant-kind box.
   - All files at `status: draft` (no implementation exists for the benchmark). Each tree is authored from scratch; NOT derived from any current `examples/` game.
   - **Spec coverage breadth is a portfolio property, not a per-game property.** The platformer game naturally needs a per-tier difficulty curve (which happens to express as `distribution_over_categories`); the survival/crafting game naturally needs resource graphs (which exercises different parts of the spec). Both genres together span more of the spec than either alone ÔÇö and that's the right way to get breadth, not by contorting one game to tick every box. The same principle holds for Act 3's architecture invariants across the two games.
   - **The neutral brief is the ground truth for the "matches-intent" judge** (┬ž"Judge"). The rubric scoring compares each trial output's design choices against the *brief's prose*, not against the A tree. This decouples judgment from the encoding: a trial that produces a design choice the brief implies but the A tree happened not to encode is still credited; a trial that "looks structurally right" by mirroring A's namespaces but contradicts the brief's intent is not.
3. **C-prompts authored.** One-paragraph prompts for each fresh game, written from scratch (NOT derived from A). Frozen before trial runs.
4. **Per-task objective intent checklists authored.** 3ÔÇô7 binary criteria per (task type ├Ś game). Committed at the same SHA as the harness.
5. **Flattener + verifier built and committed.** Reviewed against the ┬ž"B-construction" criterion. Fairness audit (Layer 3) performed; flattener SHA frozen only at audit score Ôëą 4.
6. **B's constructed** via the frozen flattener; verifier asserts information-completeness on each.
7. **Auxiliary LLM judge pinned** (specific model + version) at this step.
8. **Blinding-leak + content-preservation calibrations run (v8 hardening of v7).** Two sibling gates, **both required to pass before trial zero**:

    a. **Blinding-leak calibration (Layer 3 ÔÇö v7, N bumped at v8).** 90 outputs across A/B/C (30 per condition) on a single calibration task. **Phase 1 (positive control / judge fitness):** condition-prediction judge predicts on raw outputs; pass iff above chance with statistical significance (lower bound of 95% Wilson CI > 1/3). **Sanitize.** **Phase 2 (blinding-leak gate):** same judge predicts on sanitized outputs; pass iff at chance (95% Wilson CI on accuracy includes 1/3). Overall pass iff BOTH phases pass. Phase-1 failure Ôćĺ re-pin or strengthen judge. Phase-2 failure Ôćĺ strengthen sanitization (counterweighted by Layer 3b below; see point (c) on the joint-pass region).

    b. **Content-preservation calibration (Layer 3b ÔÇö v8).** The matches-intent judge (same `score_matches_intent` rubric used at trial-time) scores K=3 known-correct + K=3 known-incorrect pre-authored anchors per `(game, calibration_task)` set. **Phase A (anchor sanity, on RAW anchors):** pass iff `median(correct, raw) Ôëą 4`, `median(incorrect, raw) ÔëĄ 2`, `raw_gap Ôëą 3` ÔÇö per set. **Phase B (content preservation, on SANITIZED anchors):** pass iff `median(correct, sanitized) Ôëą 3`, `sanitized_gap Ôëą 2` ÔÇö per set. Overall pass iff every per-set result has both phases passing. Phase-A failure Ôćĺ re-author anchors. Phase-B failure Ôćĺ tune sanitizer down (counterweighted by Layer 3 Phase 2 above; see (c)).

    c. **Joint pass region ÔÇö the v8 contract.** Layer 3 Phase 2 pushes sanitization toward more aggressive stripping (judge at chance); Layer 3b Phase B pushes against over-stripping (judge can still discriminate good from bad content). A sanitization function passes v8 iff it lands in the joint pass region of both. v7's gate was unidirectional and admitted vacuous passes from over-strippers; v8's two-sided gate forces sanitization to thread the needle.

    Trials do not count until BOTH calibrations pass overall. See ┬ž"Judge" Layer 3 + Layer 3b for the full rationale; see v7 Ôćĺ v8 audit-trail rows #14, #15, #16 for the holes v8 closes.

The instrument is the full bundle (per ┬ž"Test subjects"): model name + variant + exact quantization + GGUF revision SHA + inference-engine build version + sampling parameters + chat-template / reasoning-format handling. 11. **Instrument calibration smoke run.** Before any trial counts, both pinned instrument bundles (Qwen headline + Claude transfer probe) are exercised on a **pre-registered canary prompt** ÔÇö a small fixed coding task committed at the harness SHA ÔÇö and the output is checked for three pass criteria:
    - **Structural validity.** Thinking blocks (where the instrument uses a reasoning mode) parse cleanly: open/close tags balanced; no mid-token truncation; no encoding errors; no repeated-token gibberish; the output reaches its max length OR terminates on a clean stop token. Catches the known instrument-fault class ÔÇö malformed `<think>` blocks, tokenizer/template mismatches, build-specific gibberish under specific CUDA / GPU configurations ÔÇö that would silently corrupt trials without showing as an obvious error.
    - **Seed-sensitivity (the actual power guard).** The instrument MUST produce *genuinely different* outputs across different seeds on the same canary prompt. Concretely: at least K (K Ôëą 5) invocations under distinct seeds produce outputs whose SHA-256 hashes are all distinct AND whose pairwise edit distance exceeds a pre-registered minimum proportion of the shorter output's length (specific K and threshold pinned at the harness-build commit). Catches the silent-power-collapse class ÔÇö a temperature accidentally set to 0, a broken seed parameter the engine ignores, a configuration that makes the instrument effectively deterministic across seeds. **This is the actual power guard.** The benchmark's N=20 protects effect detection only if the 20 trials in a cell produce 20 genuinely-varied draws; if the instrument is accidentally deterministic, within-cell variance is zero and N=20 collapses to N=1. **Note: this property is the OPPOSITE of the simulation-layer determinism discipline (F-007 / D-018).** The simulation needs byte-identical reproducibility because *one* canonical trajectory is the cross-engine claim. The benchmark instrument needs sampled variance because N protects effect detection only by averaging over varied draws. Importing byte-identity here was a confusion of layers (corrected at v4 supersession).
    - **Rubric reachability.** The auxiliary LLM judge scores the smoke output Ôëą 4 on the "matches intent" rubric ÔÇö a sanity check that the instrument can hit the rubric at all on a known-trivial task. A 0/5 here would mean the instrument is broken at the output layer, not that the spec is unhelpful; trials at this configuration would conflate instrument failure with spec failure.

    **Auxiliary check ÔÇö NOT a pass/fail gate. Same-seed auditability.** Two back-to-back invocations of the canary prompt with the same seed are recorded for audit purposes. **Strict byte-identity is NOT required and is NOT a pass/fail criterion** ÔÇö GPU inference introduces non-determinism from float-reduction-order across parallel kernels and from batch effects, similar in flavor to the libm transcendental drift the project fought at the cross-engine simulation layer (#15). Pin deterministic kernels and flags where the inference engine offers them; use batch-size-1 to avoid batch effects; record both same-seed re-runs as artifacts alongside the original so a third-party replicator can re-score and verify the rubric judgment held. **Auditability is recorded, not gated.** A failed audit (large divergence on the same-seed re-run) is noted in F-009 with the observed divergence rate but does NOT invalidate trials; gating on bit-exactness would block a perfectly good instrument that the hardware simply cannot reproduce byte-for-byte. The property worth having is "a flagged trial can be re-run and reproduced closely enough to verify its score," not "the instrument is a deterministic function."

    Trials do not count until **all three pass/fail criteria above pass for both instruments**. The canary prompt, the configurations, the smoke outputs across all seeds, the judge's scores, the pairwise edit-distance matrix, and the same-seed audit re-runs are recorded as artifacts in the harness-build commit. A pass/fail-criterion failure halts the benchmark until the instrument is fixed or replaced; any change to the pinned bundle is a new pre-registration cycle.

## What this benchmark will NOT establish (named here so F-009 cannot over-claim)

A clean positive on this benchmark does NOT prove:

- **A small-but-real effect like the ETH prior's ~4pp.** The N=20 paired design has MDE Ôëł 25pp; below that, the design cannot distinguish "no effect" from "small effect." F-009's null result, if it lands, refutes only "medium-or-larger effect," not "any effect."
- **Generality across model families.** Single Qwen variant is the headline subject; F-009's claim is explicitly scoped to that variant. The Claude transfer probe extends the picture across one capability tier, not across families.
- **Generality across genres beyond two.** Two fresh games is the resource budget; v0.3+ may extend.
- **Independent-implementer convergence.** Same-author-twice (F-007) is not exercised by this benchmark.
- **The ┬ž6 Blueprint-graph-absorbs-design question.** No visual-graph subject is used; F-006's caveat persists.
- **Independent fresh-game design.** Even under the three-act authoring protocol (Protocol step 2), the spec-author (Claude) is the one authoring all three: the neutral design brief (Act 1), the encoded A tree (Act 2), and the trace-disciplined architecture invariants (Act 3). The act-splits eliminate the most direct contamination vectors (authoring directly toward the spec's feature checklist; declaring invariants for spec coverage rather than engineering need) but do not eliminate the spec-author's *taste* for what makes a game-design coherent ÔÇö taste that has been shaped by writing the spec. The trace rule in Act 3 bounds (but does not eliminate) the architecture-layer contamination: a trace statement that *reads* plausible could still reflect taste shaped by spec-authoring. Full independence would require either an external designer authoring the brief and an external engineer authoring the invariants, or encoding a pre-existing real game's documented mechanics and engineering decisions that provably predate the spec (and the latter introduces training-data contamination risk on the test subject). The three-act protocol is the practical maximum of independence within the resource budget; the residual joins same-author-twice, single-subject, and Blueprint as a stated limitation of F-009.

F-009's claim must be scoped to exactly what is measured.
