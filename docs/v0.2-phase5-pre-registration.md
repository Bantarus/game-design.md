# v0.2 Phase 5 — Pre-Registration (LOCKED)

> **The Phase 5 help-benchmark's integrity rests on this document being committed BEFORE any trial runs and never edited after the first trial.** Any change to thresholds, refutation condition, task set, B-construction method, judge selection, or rubric between this commit and trial-zero — or after trials begin — invalidates the benchmark. The companion scope doc ([v0.2-phase5-help-benchmark-scope.md](v0.2-phase5-help-benchmark-scope.md)) frames the *why*; this doc fixes the *what*. The scope can evolve. This cannot.
>
> This pre-registration covers five integrity vectors: (1) the **threshold** against which success is measured, (2) the **refutation condition** stated in negative form, (3) the **B-construction method** that prevents an unfaithful flattening from rigging A vs. B, (4) the **judge** that prevents the test subject from grading itself, and (5) the **task set** on games the spec was never shaped around. Each was identified as a class of risk before any data existed.

## Claim under test

Providing a coding agent with a conformant `game-design.md` tree improves first-pass implementation correctness, **by enough to justify the added token/step cost**, relative to providing the same information without the structure.

## Refutation condition (F-009 negative form, stated before data exists)

The standard's structure does NOT earn its complexity at v0.2 if **either** of the following holds, evaluated on the *headline task set* (medium + hard + ambiguity probe):

1. `(success_rate(A) − success_rate(B)) < 0.10` — i.e., A does not beat B by at least **10 percentage points absolute** on success rate.
2. `(success_rate(A) − success_rate(B)) / (cost(A) − cost(B)) < 1.0` — i.e., the **success-lift-per-cost-lift ratio is below 1.0** (every 1% increase in cost must buy at least 1% increase in success rate).

If either condition holds, F-009 is reported as the negative result *with the measured numbers and confidence intervals*. The negative is as important as the positive; the project does not edit this threshold to find a positive.

**Calibration against the ETH/LogicStar prior.** The cited prior measured developer-written context files at +4% success / +20% cost (ratio ≈ 0.2) and LLM-written context files at −3% success / +20% cost (ratio ≈ −0.15). The 1.0 ratio bar is **five times stricter than the developer-written prior** — chosen because byte-identical determinism is a high constructive bar and the helpfulness claim deserves a similarly high empirical bar. The +10pp absolute floor prevents a Pyrrhic positive where a tiny lift (e.g., +1%) at a tiny cost increase (e.g., +0.5%) yields a high ratio without practical significance.

## Test subjects

- **Headline subject:** local **Qwen** (different model family from the spec-author agent; "helps" must not collapse to "helps the model the spec was tuned to"). Exact Qwen variant pinned in the harness commit and recorded in the F-009 report.
- **Optimistic-ceiling secondary:** Claude (the spec-author agent), **explicitly labelled, not reported as the headline number**. Used only to detect "the spec helps the model it was tuned to" — a positive here while the headline is negative would tell us the spec is over-fitted.
- **Both subjects use the same task set, conditions, and rubric.** No per-subject parameter tuning.

## Task set

Four task types instantiated on each of two fresh games (genres not in the current `examples/` set, so the spec was never shaped around them):

### Fresh games (NEW specs authored for the benchmark; not in `examples/`)

1. **Platformer** — single-player, real-time, level-design-driven, minimal RNG. Tests the standard against a game whose dynamics are *not* probabilistic. Working title chosen in the game-spec authoring step.
2. **Survival/crafting** — single-player, resource-graph-driven, tech-tree-heavy, balance-critical. Tests the standard against deep dependency graphs and economy scaling.

Two games (not three) to keep N tractable at 20 per cell; a third game can be added in a follow-up benchmark pass if the v0.2 result is ambiguous.

### Task types (each instantiated once per fresh game; 4 task types × 2 games = 8 task instances)

| Task type | Brief | Scoring axis | Headline? |
| --- | --- | --- | --- |
| **Easy — content authoring** | Author a single new content entity from the schema + one example. (§11.1 benchmark task.) | Schema-valid · lints-clean · refs-resolve · matches intent | **No** (excluded from headline; reported separately) |
| **Medium — verb/rule integration** | Implement a new verb (or rule) and wire it through the affected loop and state machine. | All of the above + builds + state-machine totality preserved | **Yes** |
| **Hard — system spanning** | Add a new system that touches `mechanics`, `loops`, `economy-balance`, and at least one state machine; honor declared architecture invariants; produce a deterministic result if the loop is `timescale: moment`. | All of the above + invariants honored + (for deterministic loops) reproducible trajectory at fixed seed | **Yes** |
| **Ambiguity probe — prose fallback** | A deliberately under-specified brief (e.g., "design a card that synergizes with X" or "add a status effect that feels punishing without being unfair") with **no numbers in the brief**. The agent must choose numbers and design choices that are *defensible given the prose rationale* of the game spec. | Matches intent + design choices are defensible against the prose rationale + no unsupported additions (e.g., importing a mechanic not in the brief or doc) | **Yes** |

**Why Easy is excluded from the headline.** Easy is format-mimicry-given-an-example, which frontier models ace regardless of the doc. Including it in the headline number inflates A's apparent lift over C (minimal prompt) and dilutes the medium/hard/ambiguity signal that the headline is supposed to isolate. Easy is reported separately to detect the "the doc helps with everything including format mimicry" case.

**Why ambiguity probe carries weight.** The standard's "prose is the fallback when no token covers a case" principle is the direct claim that prose rationale > structured tokens for under-specified design. The ambiguity probe is the one task type where A's prose layer should most clearly beat B's flat text and C's nothing. If A doesn't beat B on this task, the prose layer isn't doing what the spec claims.

## Conditions (three, built from the same underlying facts)

For each task, run the subject under three information conditions built from the *same* underlying facts. The **A-vs-B comparison is the headline** — it isolates whether the *structure* earns its complexity, holding information constant.

- **A — full tree.** Conformant `game-design.md` tree (core + relevant subfiles + content for the fresh game) plus `AGENTS.md` and `CLAUDE.md`.
- **B — flattened prose.** Identical information content of A, **mechanically** flattened into a single unstructured prose document. No frontmatter tokens, no `{ref}` syntax, no canonical section structure, no cross-file pointers. Same facts, same numbers, same rules.
- **C — minimal prompt.** A one-paragraph description of the game (the "vibe coding" baseline). NOT derived from A — written from scratch alongside the game spec, frozen before any trial.

What each comparison isolates:

- **A vs. B (headline):** does the structure earn its complexity? → success-lift-per-cost-lift on med/hard/ambiguity.
- **A vs. C:** total value of the doc (structure + information together).
- **B vs. C:** does unstructured context even help? (The ETH/LogicStar prior says it can go negative.)

If A beats C but not B, the standard's value is the *information you put in it*, not the format — a result worth being able to see, and one that would point the v0.3 effort at simpler doc formats.

## B-construction (the third integrity vector)

B must be a *faithful* flattening of A, not a strawman. Conscious or unconscious rigging vectors closed here: B authored by the party invested in A; B authored after seeing A's performance; B lossy in convenient places (rules dropped, numbers softened, references silently un-resolved).

### Construction method: mechanical flattening (deterministic script)

A flattener script — committed *after* this pre-reg but reviewed against the criterion below — performs the following deterministic transformation on a `game-design.md` tree:

1. **Walk** the tree in the order given by the core file's `files:` map: core → each subfile → each content-schema file → each content-entity YAML.
2. **For each YAML frontmatter block:** emit prose sentences that name every key and value. Inline-expand all `{namespace.id}` references to their resolved values (using the same resolution logic as `gdmd lint`). Drop the structural delimiters (`---`, `:`, `-`); keep the values.
3. **For each Markdown prose body:** emit verbatim (this *is* the rationale layer; B keeps the prose).
4. **Drop:** the `files:` map (replaced by simple concatenation), the `file_type:` discriminators, the `last_verified:` dates, the `status:` lifecycle markers, the `implemented_in:` glob arrays, the canonical `## Tokens`/`## Rationale`/`## Open Questions` section headers. **Keep:** every number, every rule's `do:` steps, every distribution, every balance target, every state-machine transition, every entity property.
5. **Concatenate** the per-file outputs into a single prose document. No section headers, no table of contents, no navigation.

The flattener is deterministic: running it twice on the same tree produces byte-identical output. It is committed *before* any B is constructed for trial use.

### Verifier criterion: information-completeness

A separate verifier — also committed *after* this pre-reg, *before* any trial — diffs every tree against its flattened B and asserts:

1. **Every distinct numeric value** in A's frontmatter (resources' min/max, balance_targets' numbers, distribution parameters, etc.) appears at least once in B's prose, as a digit-for-digit literal string.
2. **Every distinct `{namespace.id}` reference** in A has its *resolved value* (the target of the ref) appearing in B.
3. **Every `rules.<id>.do[]` step** in A appears in B as recognizable prose (either as a verb name or a structured-action expansion).
4. **Every distribution `type`** declared in A appears in B by name.
5. **Every `balance_targets.<id>.target` value** appears in B with its measure.
6. **Every state-machine transition** in A (a from/to/event triple) appears in B as a prose statement.

The verifier exits non-zero on any miss. **A B that does not pass the verifier is not used in any trial.** The verifier's output (the list of which assertions it ran, and that all passed) is recorded with each B and shipped alongside the F-009 report.

### What this guards against

- **Unfaithful flattening.** Verifier catches it: any number, rule, or distribution dropped from B is a verifier-fail.
- **Adversarial B authoring.** Eliminated by construction: the flattener is deterministic; no human chooses what to drop.
- **Late editing.** The flattener script's commit SHA is frozen at the start of the trial run. Any change to the flattener invalidates all already-collected B trials.

## Judge

- **Objective criteria (machine-scored).** schema-valid · lints-clean (0 errors) · refs-resolve · integer-domain honored · for code tasks: builds + passes `gdmd verify` trajectory where applicable + honors declared invariants. These are bit-checkable; no judgment involved.
- **Subjective criterion: "matches intent."** Scored by the **user (human judge)** against the rubric below. **NOT the test subject (Qwen) and NOT the spec-author agent (Claude).** The user is the design authority for the fresh games' specs; they know what intent means for those briefs.
- **Blinding.** Each trial output is presented to the judge without the condition label (A/B/C) attached — the harness shuffles condition order per task and removes any harness-recognizable artifacts before showing the output. Trial subject identity (Qwen vs. Claude) is also blinded where feasible.
- **Tie-breaker on uncertainty.** When the judge cannot decisively rate "matches intent" at the upper score, **score lower** (bias against false-positive success). The benchmark's integrity is in calling negatives accurately.

### Rubric for "matches intent" (subjective, 0–5 scale, fixed in this commit)

| Score | Definition |
| --- | --- |
| **5** | Fully meets the brief. Every design choice is defensible against the brief and the doc's prose rationale (where applicable). No unsupported additions. |
| **4** | Mostly meets the brief. One small design choice that is defensible but non-optimal; no unsupported additions. |
| **3** | Partially meets the brief. Either one significant gap (a piece of the brief not addressed) OR one unsupported addition (mechanic/rule not in the brief or doc). |
| **2** | Mostly off. Multiple gaps or multiple unsupported additions. |
| **1** | Off. Doesn't address the brief, or addresses it in a way that contradicts the doc's stated pillars or non_goals. |
| **0** | No usable output (refusal, empty, syntactically broken to the point of unscorable). |

A trial **passes the subjective criterion** at score ≥ 4. A trial **passes overall** if it passes every objective criterion AND scores ≥ 4 on subjective. **Success rate** in F-009 is the fraction of trials passing overall.

## Cost metric

Recorded every trial:

- **Tokens:** total input + output tokens consumed by the subject. Includes system prompt, condition's information payload (A/B/C content), brief, and the subject's responses.
- **Tool-steps:** count of tool calls made by the subject during the trial (file reads, file edits, shell commands, etc.). Tool-call results are included in the token count above.

The headline cost number is **mean total tokens per trial**, reported per (condition × task × subject) cell.

## Sample size and statistical reporting

- **N = 20 per (condition × task × subject) cell.** Total: 3 conditions × 4 task types × 2 games × 2 subjects = **48 cells × 20 = 960 trials**. (Optimistic-ceiling Claude subject doubles the cell count but is non-headline.)
- **Headline cells** (medium + hard + ambiguity, both games, headline subject Qwen): 3 task types × 2 games × 1 subject = 6 cells × 20 = **120 headline trials**.
- **Confidence intervals.** Every success rate reported with a 95% CI computed via Wilson score (small-sample-friendly). Every cost delta reported with bootstrap CI over the per-trial cost samples.
- **Significance call.** A vs. B success-rate-lift is called significant when the lower bound of A's 95% Wilson CI exceeds the upper bound of B's. Cost-lift significance computed analogously on the bootstrap.

## Protocol

1. **This pre-reg commit lands.** Sets the gate. SHA recorded inline in this doc after commit (see footer).
2. **Fresh games authored.** Two `game-design.md` trees (platformer, survival/crafting), conformant to v0.2.0-alpha. Authored fresh; NOT derived from any current `examples/` game.
3. **Flattener + verifier built and committed.** Reviewed against the §"B-construction" criterion above. SHA frozen.
4. **C-prompts authored.** One-paragraph prompts for each fresh game, written from scratch (NOT derived from A). Frozen before trial runs.
5. **B's constructed.** Flattener run against each fresh game's A; verifier asserts information-completeness. Failure here halts the benchmark until the flattener is fixed (and the flattener-fix invalidates any already-run trials).
6. **Trial harness built.** Parametrizes (subject, condition, task, game). Records token count, tool-step count, and output blob per trial. Blinds outputs for judge review.
7. **Trials run.** N=20 per cell; deterministic seeds where applicable; subject parameters frozen.
8. **Judge reviews blinded outputs.** Scores per rubric. Ties broken conservatively (lower).
9. **F-009 reported.** Computed success rates with CI per cell; A-vs-B headline; A-vs-C and B-vs-C secondary; per-task and per-game breakdown; cost-lift; success-lift-per-cost-lift. Pre-reg refutation condition evaluated against the measured numbers. F-009 published whichever way the numbers land.

## What this benchmark will NOT establish (named here so F-009 does not over-claim)

Even a clean positive on this benchmark does NOT prove:

- **Independent-implementer convergence** (the same-author-twice caveat recorded in F-007's "what's NOT closed"). This benchmark measures help-to-an-agent, not convergence-of-two-implementers.
- **The §6 Blueprint-graph-absorbs-design question.** Godot is text-authored, Qwen and Claude are text-authored; the Blueprint failure mode is not exercised by any of them. F-006's caveat persists.
- **Generality beyond two fresh genres.** Two games is the minimum; a third (and a fourth) would strengthen any claim of generality. v0.2's resource budget is two; v0.3+ may extend.

F-009's claim must be scoped to exactly what is measured.
