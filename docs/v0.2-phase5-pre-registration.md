## Claim under test

Providing a coding agent with a conformant `game-design.md` tree improves first-pass implementation correctness, **by enough to justify the added token/step cost**, relative to providing the same information without the structure.

## Refutation condition (F-009 negative form, stated before data exists)

The standard's structure does NOT earn its complexity at v0.2 if **either gate fails** on the headline task set (medium + hard + ambiguity probe, A-vs-B comparison, paired McNemar):

1. **Success gate.** The point estimate of `success_rate(A) ‚àí success_rate(B)` on the headline task set is **below the power-derived MDE floor** (25pp; see ¬ß"Power analysis"), OR the lower bound of its 95% confidence interval is below 0 (i.e., the sign of the effect is not robust).
2. **Cost gate.** The point estimate of `cost(A) ‚àí cost(B)` exceeds **25% of `cost(B)`** (i.e., the spec adds more than 25% to the agent's token consumption). The 25% is the cost level at which the ETH/LogicStar prior found context files barely moved success ‚Äî beyond it, the spec is paying for something the data is not showing it earned.

If either gate fails on the headline comparison, F-009 is the negative result reported with the measured numbers and CIs. **The project does not edit either gate value to find a positive.** A-vs-C and B-vs-C are reported as context ‚Äî including the case where A beats C but not B (which would say the standard's value is the information you put in it, not the format) ‚Äî but they do not affect the headline judgment.

## Power analysis (pre-registered)

**Design.** Paired binary outcomes per task instantiation. Each (task type √ó game) instantiation is run under conditions A and B (and C); the A and B outcomes on the *same* task are paired for McNemar's test. C is unpaired baseline.

**Calculation.** For McNemar's two-sided test at Œ± = 0.05, power = 0.80, the required count of discordant pairs `n_d` to detect a true alternative `œÄ1` (= P(A=success | A‚âÝB)) is approximately `n_d ‚âà (z_Œ±/2 + z_Œ≤)¬≤ / (2œÄ1 ‚àí 1)¬≤` with `z_Œ±/2 + z_Œ≤ ‚âà 2.80`. At N=20 paired observations with an *expected discordance rate* `œÄ_d` of 30% (i.e., A and B disagree on outcome on ~30% of tasks), expected `n_d ‚âà 6` ‚Äî well below the count needed for any moderate alternative.

**Minimum detectable effect.** Expressed as a success-rate difference, the MDE at N=20 paired (expected `œÄ_d ‚âà 30%`) is approximately:

> MDE_paired ‚âà 2 ¬∑ ‚àö(œÄ_d / N) ‚âà 2 ¬∑ ‚àö(0.30 / 20) ‚âà **0.245 ‚âà 25 percentage points**.

At lower discordance rates (`œÄ_d < 0.20`), the effective MDE is *higher*; at higher discordance rates, lower. The observed discordance rate will be reported alongside the result, and if it falls below 0.20 the floor is the effective MDE at the observed discordance, not the pre-registered 25pp (this widens the gate honestly toward the data, in the direction *against* the spec).

**Floor for the success gate: 25pp at N=20 paired.** This is what the design can honestly detect. It is much larger than the ETH/LogicStar prior's ~4pp effect; the benchmark is structurally incapable of refuting a small-but-real effect like the ETH one. **The benchmark answers "does the standard produce a medium-or-larger effect"; it does NOT distinguish "no effect" from "small (ETH-like) effect."** F-009 must scope its claim to this.

## Test subjects

- **Headline subject:** local **Qwen**, *instrument bundle pinned at harness-commit time* and recorded in F-009. A change to any field after pinning is a new instrument and requires a new pre-registration cycle. **The pre-registered claim is scoped to that exact bundle** ‚Äî single-instrument is a stated limitation of F-009, alongside same-author-twice (F-007) and Blueprint (F-006). - **Transfer-probe subject:** Claude (the spec-author agent), pinned to a specific Claude version at harness-commit time, labelled as transfer probe (NOT "optimistic ceiling").   - **Claude lift > Qwen lift** ‚Äî spec adds more for capable models; the structure encodes complexity the smaller model can't yet leverage.

  - **Claude lift < Qwen lift** ‚Äî frontier models infer the structure unaided; the spec's marginal value declines with capability. (This is a real finding, not a disappointment ‚Äî it would say the spec is most useful for capability-constrained agents, which is itself a clear scope on who the standard is for.)
- Both subjects run the same task set, conditions, and rubric. No per-subject parameter tuning. The Qwen result is the headline; the Claude result is reported with the same rigor.

## Conditions

For each task, run the subject under three information conditions built from the *same* underlying facts. The **A-vs-B comparison is the headline gate**; A-vs-C and B-vs-C are reported context.

- **A ‚Äî full tree.** Conformant `game-design.md` tree (core + relevant subfiles + content for the fresh game) plus `AGENTS.md` and `CLAUDE.md`.
- **B ‚Äî flattened prose.** Identical information content of A, mechanically flattened into a single unstructured prose document (see ¬ß"B-construction"). No frontmatter tokens, no `{ref}` syntax, no canonical section structure, no cross-file pointers. Same facts, same numbers, same rules.
- **C ‚Äî minimal prompt.** A one-paragraph description of the game (the "vibe coding" baseline). NOT derived from A ‚Äî written from scratch alongside the game spec, frozen before any trial.

What each comparison isolates:

- **A vs. B (headline, gated):** does the structure earn its complexity? Holds information constant.
- **A vs. C (context):** total value of the doc (structure + information together).
- **B vs. C (context):** does unstructured context even help? (ETH/LogicStar prior says it can go negative.)

A result where **A beats C but not B** is reported explicitly ‚Äî that says the standard's value is the *information you put in it*, not the format, and points the v0.3 effort at simpler doc formats. The gate does not declare this a positive.

## B-construction (mechanical + verified + fairness-audited)

Three layers of protection against rigging A-vs-B via an unfaithful B.

### Layer 1: Mechanical flattening (deterministic script)

A flattener script ‚Äî committed *after* this pre-reg but *before* trial zero, with its SHA frozen at trial start ‚Äî performs the following deterministic transformation on a `game-design.md` tree:

1. **Walk** the tree in the order given by the core file's `files:` map: core ‚Üí each subfile ‚Üí each content-schema file ‚Üí each content-entity YAML.
2. **For each YAML frontmatter block:** emit prose sentences that name every key and value. Inline-expand all `{namespace.id}` references to their resolved values. Drop structural delimiters; keep the values.
3. **For each Markdown prose body:** emit verbatim (this *is* the rationale layer; B keeps the prose).
4. **Group output by topic, not by file boundary.** All resources together, all rules together, all loops together ‚Äî what a competent human prose-author would produce when asked to summarize the spec into a prose brief. (See "Design intent" below.)
5. **Drop:** the `files:` map, `file_type:` discriminators, `last_verified:` dates, `status:` lifecycle markers, `implemented_in:` glob arrays, canonical `## Tokens`/`## Rationale`/`## Open Questions` section headers. **Keep:** every number, every rule's `do:` steps, every distribution, every balance target, every state-machine transition, every entity property.
6. **Concatenate** the per-topic outputs into a single prose document.

**Design intent of the flattener (the fairness pin).** The output MUST read as a coherent prose brief ‚Äî facts grouped by topic, sentences flowing naturally, no token dumps in arbitrary order, no structural-syntax remnants. Concretely: "what a competent person who knew the game and wrote a 5-page prose summary for an implementer would produce." This is the fairness contract; the completeness verifier is layered on top.

### Layer 2: Information-completeness verifier (machine-checkable)

A separate verifier diffs every tree against its flattened B and asserts:

1. **Every distinct numeric value** in A's frontmatter appears at least once in B's prose, as a digit-for-digit literal string.
2. **Every distinct `{namespace.id}` reference** in A has its *resolved value* appearing in B.
3. **Every `rules.<id>.do[]` step** in A appears in B as recognizable prose (verb name or structured-action expansion).
4. **Every distribution `type`** declared in A appears in B by name.
5. **Every `balance_targets.<id>.target` value** appears in B with its measure.
6. **Every state-machine transition** in A (a from/to/event triple) appears in B as a prose statement.

The verifier exits non-zero on any miss. A B that does not pass the verifier is not used in any trial. The verifier's output (which assertions ran, that all passed) is recorded with each B and shipped alongside the F-009 report.

### Layer 3: Fairness audit (one eyeball before flattener SHA freezes)

Before the flattener's SHA is frozen for trial use, the auxiliary LLM judge (¬ß"Judge") audits **one full B** (the flattened platformer spec is the canonical test case) and answers:

> "On a 1‚Äì5 scale, is this a fair prose brief ‚Äî facts grouped by topic, readable as a coherent summary ‚Äî or a strawman: incoherent, token-dump-style, or pathologically disorganized?"

- **Score ‚â• 4 required to freeze the flattener SHA.** Below 4, the flattener is revised and re-audited.
- The audit prompt, the audit output, and the score are recorded as artifacts in the harness-build commit.
- A second sample B (the flattened survival/crafting spec, once authored) is audited under the same criterion as a sanity check before trial zero. Failure on either invalidates the flattener.

Completeness is machine-checkable; fairness needs one eyeball before lock. Both gates are passed before trial zero.

## Judge (multi-layer + leak-checked)

### Layer 1: Objective intent checklists (machine-scored)

For each task brief, a **pre-registered checklist of binary criteria** (between 3 and 7 items per task) is committed before trial zero. Examples:

- Medium task (platformer): "Does the new verb appear in at least one loop's `sequence:`?"; "Does it carry a valid `feel:` ref?"; "Does it transition between two declared states?"
- Hard task (survival/crafting): "Does the new system declare at least one new `balance_targets.<id>`?"; "Are all `do:` steps structured (no bare strings)?"; "Does the loop carry a `timescale:`?"
- Ambiguity probe: "Does the chosen number for X fall within the doc's declared range for similar entities?"; "Is the chosen mechanic referenced from at least one existing token?"

The checklist for each task is the objective-criterion gate; scoring is machine-mechanical (booleans). The checklist is part of the harness-build commit and frozen at the same SHA.

### Layer 2: Subjective remainder ("matches intent")

The irreducible subjective criterion ‚Äî does the output match the *spirit* of the brief, not just the checkable bits? ‚Äî is scored by:

- **Auxiliary LLM judge.** Pinned to a model from a family **different from the spec-author AND every test subject**. The rule exists to exclude two specific contamination vectors:
    - *Spec-author interpretive bias.* The spec was co-evolved with Claude across every phase, so a Claude judge would carry the spec-author's interpretive frame into matches-intent scoring ‚Äî exactly the bias the independence rule exists to exclude.
    - *Subject overlap.* Claude is the transfer-probe subject (Qwen is the headline subject); a Claude judge would partly score outputs from its own family, and a Qwen judge would entirely so.

  These two vectors stack hardest at the blinding-leak calibration (Layer 3 below), where the judge's role is *adversarial* ‚Äî the single model most likely to recognize spec-vocabulary leaking through an A-condition output is the family that wrote the spec. **The auxiliary judge at v7 is pinned to Gemma 4 26B A4B (Google), local-inference via llama.cpp.** Non-Qwen, non-Claude family. Apache-2.0 licensed. No API key required. Runs sequentially alongside the Qwen instrument in the same llama.cpp infrastructure (generation and scoring are not concurrent, so the judge has the full GPU to itself during scoring). Specific quant + GGUF SHA + sampling parameters + chat template + the three judge prompt templates (`matches_intent_prompt.md` for this layer's rubric, `fairness_audit_prompt.md` for B-construction Layer 3, `blinding_leak_prompt.md` for the calibration in Layer 3 below) are pinned together as the judge bundle at the harness-build commit, per the same bundle-pinning discipline as ¬ß"Test subjects". If a future Phase-5+ extension adds a third test subject from a new family, that family is added to the exclusion.

- **User consensus.** User scores the same blinded outputs on the same rubric. The judge-of-record is the consensus when LLM and user agree; on disagreement, the user breaks the tie with the LLM's reasoning visible.

### Rubric for "matches intent" (0‚Äì5, fixed in this commit)

| Score | Definition |
| --- | --- |
| **5** | Fully meets the brief. Every design choice is defensible against the brief and the doc's prose rationale (where applicable). No unsupported additions. |
| **4** | Mostly meets. One small design choice defensible but non-optimal; no unsupported additions. |
| **3** | Partially meets. Either one significant gap OR one unsupported addition. |
| **2** | Mostly off. Multiple gaps or multiple unsupported additions. |
| **1** | Off. Doesn't address the brief, or contradicts the doc's pillars/non_goals. |
| **0** | No usable output (refusal, empty, syntactically broken to the point of unscorable). |

A trial **passes the subjective criterion** at score ‚â• 4. A trial **passes overall** if it passes the objective checklist AND scores ‚â• 4 on the subjective rubric. Success rate in F-009 is the fraction of trials passing overall.

**Trial-time scoring is on SANITIZED outputs only (v8).** Every trial sanitizes its subject output via `harness/sanitization.py::sanitize_output` once at scoring time, and both judge calls ‚Äî matches-intent (Layer 2) and the per-criterion checklist grader (Layer 1, when the criterion is fuzzy enough to need LLM grading) ‚Äî score the sanitized form. **Raw subject outputs MUST NEVER reach the scoring judge.** The constraint exists because v7's two-phase blinding-leak calibration validated the judge's behavior on sanitized outputs (Phase 2: judge at chance); Phase 1 was the proof that the judge *can* read condition off raw outputs (judge fitness for tells). If trials scored raw, the judge would be operating in its un-blinded mode against exactly the channel Phase 2 was designed to close. The `TrialRecord` stores both `subject_output` (raw, lossless) and `subject_output_sanitized` (what the judge scored), plus `sanitization_sha256` so a post-hoc reviewer can verify the trial-time sanitizer matches the calibration-time sanitizer. See v7 ‚Üí v8 audit-trail row #14 for the root cause and project memory `sanitized-scoring-consistency` for the standing rule.

### Layer 3: Blinding-leak check with positive control (before trials count)

A-condition outputs may carry spec vocabulary, token names, structural habits that an attentive judge could use to infer the condition ‚Äî silently un-blinding the judge and reintroducing the invested-author confound.

**Why this calibration needs a positive control (v7 hardening).** The naive pass criterion ‚Äî "judge predicts at no-better-than-chance after sanitization" ‚Äî is ambiguous between two situations: (a) blinding genuinely holds, OR (b) the judge is too weak to detect a leak that's actually there. A weak judge gets ~chance regardless of whether tells are present, which is a false pass ‚Äî the gate looks green while the judge just couldn't see anything. This false-pass mode is structurally undetectable from the chance-result alone. The positive control closes the hole: before the at-chance gate is meaningful, the judge MUST demonstrate it can detect condition when tells are intact.

The positive control is the chance-result gate's analog of the negative-control idiom in spec ¬ß9.5.7 (a positive test paired with a sign-flipped control to prevent vacuous passes). The same discipline applies anywhere the gate's pass criterion is "the detector produces a chance / null / zero-divergence result" ‚Äî see the project memory `calibration-positive-control` for the broader rule.

**Two-phase procedure (before trial zero):**

1. Run a **calibration sample** of 90 outputs (30 each from A, B, C) on a single task instantiation. **N bumped from 30 ‚Üí 90 at v8** (see v7 ‚Üí v8 audit-trail row #16). At N=30 the Wilson CI was wide enough that "includes 1/3" only ruled out a *large* leak; at N=90 the half-width near p=0.5 tightens to ~0.10, catching residual leaks of ~10pp or larger. The 90 outputs are generated once via the instrument's normal generation path; both phases re-score the same 90 outputs so the marginal cost over v7 is one extra scoring pass on stored text.

2. **Phase 1 ‚Äî Positive control (judge fitness).** Present the **raw** outputs (no sanitization applied ‚Äî spec-vocabulary in A intact, prose-style markers in B intact) to the auxiliary LLM judge in random order with condition labels stripped. Ask: "For each output, predict which condition it came from (A, B, or C)." **Pass criterion (Phase 1):** the judge's accuracy is **above chance with statistical significance** ‚Äî specifically, the lower bound of the 95% Wilson CI on accuracy exceeds 1/3 (i.e., the judge is provably better than chance at detecting condition when tells are intact). **Fail criterion (Phase 1):** the judge is too weak to detect tells even when they're obvious. Re-pin the judge (the family rule must still hold) or revise the calibration sample to make tells more visible; re-run Phase 1. Note: a Phase-1 fail invalidates the *judge*, not the blinding ‚Äî the upstream apparatus is broken, the gate cannot run.

3. **Sanitization.** Apply the pre-registered sanitization to the calibration sample's outputs. The sanitization is a deterministic transformation declared at the harness-build commit; the **v9 ruleset is committed in [`benchmark/harness/sanitization.py`](../benchmark/harness/sanitization.py)** with each rule explicitly labeled either **(S)** structural pattern or **(E)** closed enumeration drawn from the spec's declared vocabulary (NOT from observed calibration samples). The discipline exists because the sanitizer is applied to the ~660-trial population but is only validated by this calibration on the 90-output sample; a denylist drawn from calibration samples would be `train_set`-shaped and would leak tells appearing in trials but not in calibration. The closed-enumeration discipline makes the sanitizer content-general by construction (see v8 ‚Üí v9 audit-trail row #17). v9's nine rules: namespace-prefix strip for both brace-form and bare-form refs (closed enumeration from spec ¬ß3); full-line strip for the closed enumeration of canonical section headers (spec ¬ß5.2 + ¬ß7.1); structural strip for remaining `##`/`###`/`####` heading prefixes; structural strip for `<<< FILE: ... >>>` markers and `---...---` frontmatter blocks; closed-enumeration strip for "per the spec" / spec-noun reference phrases; structural whitespace normalization. The sanitization SHA is recorded with the calibration artifact.

4. **Phase 2 ‚Äî Blinding-leak gate (sanitization integrity).** Present the **sanitized** outputs to the same judge under the same prompt and conditions as Phase 1. **Pass criterion (Phase 2):** the judge's accuracy is at chance ‚Äî specifically, the 95% Wilson CI on accuracy *includes* 1/3 (i.e., predictions are statistically indistinguishable from chance). **Fail criterion (Phase 2):** judge predicts above chance even on sanitized outputs ‚Üí blinding leaks ‚Üí strengthen sanitization (add patterns to the sanitizer) and re-run from Phase 2 (Phase 1 holds; the judge already passed fitness). Repeat until calibration passes.

5. **Overall pass criterion:** BOTH Phase 1 (above-chance on raw) AND Phase 2 (at-chance on sanitized) must hold. Either alone is insufficient.

The calibration is pre-registered (this section); the calibration outputs (raw and sanitized), the auxiliary LLM's predictions in each phase, the sanitization function SHA, the Phase-1 and Phase-2 accuracy values with their 95% Wilson CIs, and the overall pass/fail decision are all recorded as artifacts at `benchmark/harness/audits/blinding_leak_*.json`. Trials do not count until calibration passes.

### Layer 3b: Content-preservation gate (sanitizer's positive control) ‚Äî v8

**Why this gate exists.** Layer 3's blinding-leak gate pressures the sanitizer in exactly one direction: more aggressive stripping makes Phase 2 (judge at chance on sanitized outputs) easier to pass. There is no counterweight within Layer 3 alone, so an over-aggressive sanitizer that reduces every subject output to mush passes Phase 2 (judge genuinely blind because nothing left to read) while having destroyed the functional content the matches-intent judge needs at trial time. All three conditions then score uniformly low, the A-vs-B effect washes out, and the benchmark detects nothing ‚Äî a power loss masquerading as a passed gate. The failure mode is not bias; it is power loss.

This is the symmetric form of the v7 root cause for the judge's Phase-1 positive control (a chance-result gate without an upstream fitness check passes vacuously for too-weak judges). Same pattern, different layer: a sanitization gate without an upstream content-preservation check passes vacuously for too-aggressive sanitizers. **Every chance-result or null-result gate needs a positive control on the apparatus that produces the result.** The two positive controls together (v7's judge fitness + v8's content preservation) make the blinding-leak calibration apparatus bidirectionally constrained: too-weak judges fail Phase 1, too-aggressive sanitizers fail Phase B, sanitization that is at the right point on the strip-vs-preserve curve passes both.

**Two-phase procedure (before trial zero, alongside Layer 3's blinding-leak calibration):**

1. **Anchor library.** A set of pre-authored anchor texts committed in [`benchmark/harness/content_preservation.py`](../benchmark/harness/content_preservation.py), keyed by `(game, task_cell_id)`. Each set contains **K=3 known-correct + K=3 known-incorrect** anchor outputs for the calibration task. The anchor library covers both fresh games' calibration tasks (`easy_platformer` and `easy_survival` at v8). The library's SHA-256 is recorded with each calibration artifact.

2. **Anchor authoring discipline (pinned in the v8 anchor library).** Correct anchors **span A/B/C styles** (one YAML-with-refs A-style, one prose B-style, one inline C-style) so the gate measures *content* recognition, not *style* recognition ‚Äî if only A-styled outputs were correct the judge could appear to discriminate content while actually reading style, and that apparent discrimination would collapse under sanitization for the wrong reason. Incorrect anchors **span failure modes** (off-topic, hand-waving, wrong-on-specifics) so a single failure mode does not over-fit the gate. Re-authoring requires a pre-reg cycle.

3. **Phase A ‚Äî Anchor sanity check (on RAW anchors).** The pinned matches-intent judge (same `score_matches_intent` interface used at trial time, with the same per-game `task_brief` + `game_brief` arguments) scores each correct and incorrect anchor on the rubric. **Pass criteria, all three required, per set:**
    - `median(correct_scores, raw) ‚â• 4.0`
    - `median(incorrect_scores, raw) ‚â§ 2.0`
    - `raw_gap = median(correct, raw) ‚àí median(incorrect, raw) ‚â• 3.0`

    A Phase-A fail per set means the **anchors are not discriminably authored** for that set ‚Äî the gate cannot tell whether a later Phase-B fail comes from sanitization destroying content or from the anchors being indistinct in the first place. Resolution: re-author the failing anchor set in a pre-reg cycle. Phase-A fail invalidates the *anchors*, not the sanitization.

4. **Phase B ‚Äî Content preservation (on SANITIZED anchors).** The same judge scores the same anchors after `sanitize_output` has been applied to each. **Pass criteria, both required, per set:**
    - `median(correct_scores, sanitized) ‚â• 3.0`
    - `sanitized_gap = median(correct, sanitized) ‚àí median(incorrect, sanitized) ‚â• 2.0`

    The sanitized-correct floor (3 rather than Phase A's 4) and the sanitized-gap floor (2 rather than Phase A's 3) leave room for sanitization to compress dynamic range somewhat ‚Äî that's expected and acceptable. What is not acceptable is the sanitizer mushing all anchors to the same score (gap collapse) or making correct anchors un-recognizable as decent (correct floor breach). A Phase-B fail per set means the **sanitizer is destroying content for that set**. Resolution: tune the sanitizer down in a pre-reg cycle, re-run Phase B (Phase A holds because the anchors are unchanged). Phase-B fail invalidates the *sanitization*, not the anchors.

5. **Overall pass criterion:** every per-set result must have BOTH Phase A and Phase B passing. The two positive controls together (Layer 3's Phase 1 judge fitness on raw outputs; Layer 3b's Phase A anchor sanity on raw anchors) anchor the gate at the upper end of the rubric; the two strict pass criteria (Layer 3's Phase 2 judge at chance on sanitized outputs; Layer 3b's Phase B content preserved on sanitized anchors) anchor it at the constraints sanitization must satisfy. A sanitization function passes the v8 apparatus iff it lands in the joint pass region of both layers.

The calibration is pre-registered (this section); the per-anchor raw and sanitized scores, the per-set median calculations and gap values, the anchor-set SHA, the sanitization SHA, and the per-set pass/fail decisions are recorded as artifacts at `benchmark/harness/audits/content_preservation_*.json`. Trials do not count until both Layer 3 (blinding-leak) and Layer 3b (content-preservation) calibrations pass.

**Reachability note.** Layer 3b uses the matches-intent judge (same `score_matches_intent` rubric used in trials), not the condition-prediction judge used in Layer 3 Phases 1+2. Both judges are the same pinned model (Gemma 4 26B A4B at v7+) but at different prompts. This is deliberate: the matches-intent judge is exactly what runs at trial scoring time, so the content-preservation gate validates the *trial-time* scoring judge specifically against sanitized inputs ‚Äî closing the loop with v8's audit-trail row #14 (trial-time scoring is on sanitized outputs).

### Layer 3c: Sanitizer-generalization check on real trial outputs ‚Äî v9

**Why this check exists.** Layer 3 validates the sanitizer on a 90-output calibration sample; Layer 3b validates content preservation on 12 pre-authored anchor texts. The sanitizer is then applied to the ~660-trial production population, **which neither calibration ever saw**. If the sanitizer is denylist-shaped (strips the specific spec-vocabulary tells observed in calibration), then a trial output containing a spec term that didn't appear in the calibration sample leaks that tell straight into a scored trial and the calibration had no opportunity to catch it. This is the train/test distribution-shift problem applied to the sanitization apparatus. The matches-intent component is the contaminated layer specifically ‚Äî the objective checklist is immune (machine-scored on game behavior, indifferent to style).

v9 addresses the gap two ways. **First**, the sanitizer is made content-general by construction (every rule (S) structural or (E) closed enumeration from spec vocabulary, NOT from calibration samples ‚Äî see Layer 3 step 3 above and the v8 ‚Üí v9 audit-trail row #17). A content-general sanitizer is the structural defense; v9 expands the rules and pins the discipline. **Second**, a positive verification that the sanitizer DID generalize to the trial population, run at two timepoints (this Layer 3c).

**Two-timepoint procedure:**

1. **Pre-sweep small-batch check (before the full trial sweep).** After instrument calibration (Protocol step 11) has passed and BEFORE committing to the full ~9-hour trial sweep, generate a **small batch of real trials** ‚Äî typically a handful per condition per game, total N=12‚Äì30. Apply the SHA-locked sanitization function. Run the same condition-prediction judge from Layer 3 on the sanitized outputs. **Pass criterion: identical to Phase 2 of Layer 3** ‚Äî 95% Wilson CI on judge accuracy includes 1/3 (predictions at chance). **Wilson CI is wide at small N ‚Äî this is an early-warning flag, not a statistical guarantee.** An above-chance accuracy at small-batch is a strong signal of generalization failure that catches the problem after ~20 trials rather than after ~660. A pre-sweep fail ‚Üí strengthen the sanitizer in a pre-reg cycle (re-runs Layer 3 Phase 2 + Layer 3b Phase B + this check at pre-sweep); do NOT proceed to the full trial sweep.

2. **Post-hoc full-sweep sample check (recorded alongside F-009).** After the full trial sweep completes, draw a **uniform sample** from the ~660 trial records ‚Äî typically N=60‚Äì120, distributed across conditions √ó games √ó tasks. Apply the same SHA-locked sanitization. Run the same condition-prediction judge. **Pass criterion: identical to Phase 2** ‚Äî 95% Wilson CI on judge accuracy includes 1/3. At N=100, the CI half-width near p=0.5 tightens to ~0.10, providing real statistical power on the question "did the sanitizer generalize to the trial population we actually ran?" A post-hoc fail is recorded as a stated F-009 limitation: "blinding-leak generalization passed at pre-sweep N=X with CI [...,...] but failed at post-hoc N=Y with CI [...,...]; observed leak magnitude ~Zpp." F-009 reports the leak magnitude alongside the headline so readers can calibrate the result for the residual leak.

The check is implemented in [`harness/calibration.py::check_blinding_generalization`](../benchmark/harness/calibration.py). Both pre-sweep and post-hoc results are recorded as artifacts at `benchmark/harness/audits/generalization_check_*.json` with the sanitization SHA, the judge bundle ID, and the per-prediction accuracy. The whole apparatus is "judge predictions on stored text" so the marginal cost of both checks together is small relative to the full sweep ‚Äî the same economics as v8's N=30 ‚Üí 90 bump.

**Generalization-of-the-rule.** Within-sample validation can be exquisitely correct yet mute on out-of-sample failure modes. Every calibration whose sample is smaller than its production population ‚Äî i.e., basically every calibration ‚Äî needs an explicit generalization check on the production data. v9 makes this an apparatus-level discipline for the help-benchmark; the broader rule is recorded as memory `train-test-distribution-shift-sanitizer`.

**Blinding-only generalization (a forced asymmetry ‚Äî stated explicitly).** Layer 3c verifies *blinding* generalization on real trial outputs but does NOT verify *content-preservation* generalization on them. The asymmetry is forced by data availability ‚Äî blinding-generalization needs each output's condition label (which trials carry); content-preservation-generalization needs each output's ground-truth correctness label (which is exactly what the benchmark is measuring, so it does not exist for arbitrary trial outputs the way it exists for the pre-authored anchors). The check can therefore only live on anchors. Failure-mode is benign: an over-stripping sanitizer on the trial population depresses all three conditions equally (cost: power loss, not bias). Named in ¬ß"What this benchmark will NOT establish" as a stated limitation; no gate.

## Cost metric

Recorded every trial:

- **Tokens:** total input + output tokens consumed by the subject. Includes system prompt, condition's information payload (A/B/C content), brief, and the subject's responses (including any tool-use round trips).
- **Tool-steps:** count of tool calls made by the subject during the trial.

Headline cost number is **mean total tokens per trial**, per (task type √ó game √ó subject √ó condition) cell. Cost-lift in the gate is `mean_tokens(A) / mean_tokens(B) ‚àí 1`; the gate fails at > 0.25 (25% lift).

## Sample size and statistical reporting

- **N = 20 paired per (task type √ó game √ó subject)** for medium/hard/ambiguity. **N = 5 paired** for easy (sanity check, not headline).
- **Total headline trials per subject:** 3 task types √ó 2 games √ó 20 N √ó 2 conditions (A, B) = **240 paired outputs per subject** for the headline gate. C adds 60 unpaired outputs per subject (3 √ó 2 √ó **10** √ó 1 ‚Äî C runs at N=10 for medium/hard/ambiguity, the explicit-arithmetic reconciliation of the v9-era "60" figure; see [`benchmark/harness/sweep_plan.py::C_N_PER_CELL_OVERRIDES`](../benchmark/harness/sweep_plan.py) for the planner contract). **Methodological rationale, locked pre-trial:** C is unpaired and feeds only the secondary A-vs-C and B-vs-C comparisons, so **its N does not bear on the headline** (the headline is the per-subject paired McNemar A-vs-B at N=20; C's N has zero effect on that gate). C is halved to N=10 to **limit the total Opus-xhigh cost/usage exposure** on the most-reasoning-heavy condition (a sparse one-paragraph C prompt drives the most internal extended-thinking and the most denied-tool-call flailing under xhigh, so halving N is where this cost concentration is cheapest to release; this is TOTAL cost/usage, NOT the cost-lift gate ‚Äî cost-lift is the within-Claude A-vs-B gate at ‚â§25% and C is not in it). The secondary A-vs-C/B-vs-C comparisons retain enough power for the large effects expected there ‚Äî any real design information beats a minimal one-paragraph prompt ‚Äî and N=10 can detect a large effect adequately. F-009 does NOT promote A-vs-C or B-vs-C to headline-grade powered claims; they are context for where A and B sit above the floor, not load-bearing. Easy adds 5 √ó 2 √ó 3 = 30 outputs per subject (full N=5 across all three conditions). **Per-subject total: 330 outputs.** Two subjects (Qwen headline + Claude transfer probe): **660 outputs total.**

### Execution-order discipline (pairing integrity, harness-build commit)

The headline gate is paired McNemar; pair statistical validity requires the two members of every pair (A_i, B_i) seeing the same nuisance conditions in real time. The harness MUST run pairs interleaved ‚Äî never blocked-by-condition (all A first, all B later) ‚Äî to defeat time-correlated nuisance:

- **Claude arm:** `claude-opus-4-7 --effort xhigh` runs against subscription usage limits; a long sweep can wall on rate limits mid-run, biasing late-running conditions toward `error_max_turns` failures.
- **Qwen arm:** local llama.cpp drifts over a multi-hour run (thermal throttling, memory fragmentation, sampling-state).

The planner at [`benchmark/harness/sweep_plan.py`](../benchmark/harness/sweep_plan.py) enforces this structurally: per-pair interleaving (A_i and B_i run back-to-back as a single execution unit), seeded within-pair coin (A-first vs B-first per unit, defeats first-call effects), and seeded inter-unit shuffle (defeats monotonic time effects). C cells are emitted as their own single-cell units and shuffled into the sweep order independently. The `shuffle_seed` used at trial zero is recorded in F-009 alongside the bundle SHAs. Re-runs against the same harness-build SHA + same shuffle_seed reproduce the same execution order exactly. The two subjects (Qwen, Claude) run in contiguous per-subject sweeps; within-subject pairing integrity is what the headline depends on, cross-subject lift comparisons see a per-subject time delta named in F-009 transfer-probe caveats. The same interleaving discipline applies at the smaller-N **blinding-leak calibration** generation step (`harness/calibration.py`); the calibration generation order is shuffled too.

`count-as-failure` for `error_max_turns` is unbiased ONLY under interleaved ordering ‚Äî otherwise rate-limit failures correlate with whichever condition ran late. Pre-sweep batch usage on the Claude arm should be watched before committing to the full sweep: a usage wall at N=20 is recoverable, a usage wall at N=300 is not.

### Scored field excludes reasoning traces (Opus xhigh)

Every judge call (checklist + matches-intent) scores `response.text` ‚Äî the FINAL implementation text returned by the subject. On the Claude side, `response.text` is the `result` field from Claude Code's `--print --output-format json` envelope, which contains *only* the final assistant text. Opus 4.7 at `--effort xhigh` produces a large internal extended-thinking trace, but that trace is NOT surfaced in `--print` output and therefore does NOT reach the scoring stack. This is the correct behavior ‚Äî the benchmark scores what the subject *produces*, not how it *deliberates* ‚Äî but it is named here so the property is explicit, not assumed. Verified in [`benchmark/harness/run_trial.py`](../benchmark/harness/run_trial.py) at the comment preceding `sanitize_output(response.text)`.
- **Effect-size reporting.** Every success rate reported with a 95% **Wilson CI** (small-sample-friendly). Every cost-lift reported with **bootstrap CI** over per-trial cost samples. Every paired A-vs-B comparison reported with the **McNemar test statistic and exact p-value**, alongside the point estimate and CI of the success-rate difference.
- **F-009 framing.** Given the N=20 design, F-009 is explicitly a **small-N pilot reporting effect sizes with confidence intervals**, not a statistical-significance claim in the classical-hypothesis-test sense. The pre-registered gate (25pp floor, 25% ceiling) is the *decision rule*; the *report* is the effect size with its CI, whichever way the gate lands. A directional positive at p < 0.05 strengthens the claim; a directional negative or null does not undo the gate's verdict ‚Äî the gate is what was pre-registered.

The benchmark instrument needs sampled variance because N protects effect detection only by averaging over varied draws. Importing byte-identity here was a confusion of layers (corrected at v4 supersession).

The property worth having is "a flagged trial can be re-run and reproduced closely enough to verify its score," not "the instrument is a deterministic function."

## What this benchmark will NOT establish (named here so F-009 cannot over-claim)

A clean positive on this benchmark does NOT prove:

- **A small-but-real effect like the ETH prior's ~4pp.** The N=20 paired design has MDE ‚âà 25pp; below that, the design cannot distinguish "no effect" from "small effect." F-009's null result, if it lands, refutes only "medium-or-larger effect," not "any effect."
- **Generality across model families.** Single Qwen variant is the headline subject; F-009's claim is explicitly scoped to that variant. The Claude transfer probe extends the picture across one capability tier, not across families.
- **Generality across genres beyond two.** Two fresh games is the resource budget; v0.3+ may extend.
- **Independent-implementer convergence.** Same-author-twice (F-007) is not exercised by this benchmark.
- **The ¬ß6 Blueprint-graph-absorbs-design question.** No visual-graph subject is used; F-006's caveat persists.
- **Independent fresh-game design.** Even under the three-act authoring protocol (Protocol step 2), the spec-author (Claude) is the one authoring all three: the neutral design brief (Act 1), the encoded A tree (Act 2), and the trace-disciplined architecture invariants (Act 3). The act-splits eliminate the most direct contamination vectors (authoring directly toward the spec's feature checklist; declaring invariants for spec coverage rather than engineering need) but do not eliminate the spec-author's *taste* for what makes a game-design coherent ‚Äî taste that has been shaped by writing the spec. The trace rule in Act 3 bounds (but does not eliminate) the architecture-layer contamination: a trace statement that *reads* plausible could still reflect taste shaped by spec-authoring. Full independence would require either an external designer authoring the brief and an external engineer authoring the invariants, or encoding a pre-existing real game's documented mechanics and engineering decisions that provably predate the spec (and the latter introduces training-data contamination risk on the test subject). The three-act protocol is the practical maximum of independence within the resource budget; the residual joins same-author-twice, single-subject, and Blueprint as a stated limitation of F-009.
- **Sanitizer content-preservation on the trial population specifically (asymmetry with the generalization check).** Layer 3c's generalization check verifies the sanitizer's *blinding* property on real trial outputs (pre-sweep + post-hoc), but there is **no symmetric generalization check on the sanitizer's *content-preservation* property** on real trial outputs. The asymmetry is forced by data availability: blinding is checkable on trials because each trial carries a known condition label (A/B/C); content-preservation needs each output's ground-truth correctness label ‚Äî which is exactly what the benchmark is measuring, so it is not available for arbitrary trial outputs the way it is for the pre-authored anchor library (Layer 3b). The check can therefore only live on the anchor library by construction. Failure-mode asymmetry makes this *benign rather than dangerous*: if the sanitizer over-strips content on real trial outputs beyond what the anchor library revealed, it depresses all three conditions roughly equally (sanitization runs identically across A/B/C), which costs power (a real A-vs-B effect may be under-detected) but cannot manufacture a false positive or any per-condition bias. In an already-conservative small-N pilot whose MDE is ~25pp, an under-detection risk is the safe direction to err. F-009 notes this as a stated limitation; it does not warrant a gate.
- **Cross-subject lift comparisons carry a harness-overhead delta (Claude transfer probe specifically) ‚Äî split into two kinds with different consequences.** The Claude transfer-probe instrument is invoked via the Claude Code CLI in headless mode (see `harness/instrument.py::ClaudeInstrument`); the Qwen headline instrument runs bare llama.cpp. Two named confounds:
    - **STATIC overhead.** Claude Code's intrinsic system context ‚Äî constitution, safety guidance, available-tool descriptions for the disabled built-in toolset, MCP server enumeration ‚Äî totals roughly **~200 input tokens** above the user-supplied prompt per call, re-measured at the Opus-4.7 `--effort xhigh` model-swap on this instrument (`cache_creation_input_tokens` and `cache_read_input_tokens` both 0 in the per-call accounting Claude Code surfaces ‚Äî the earlier 2k‚Äì4.5k Haiku-era figure was a cache-creation artifact of a prior Claude Code version and does NOT transfer). Prompt-independent and therefore **constant across A/B/C within the Claude arm**. The per-subject paired McNemar headline (A-vs-B success-lift for Claude alone) is unaffected; the within-Claude cost-lift gate has its denominator padded by a small constant (an order of magnitude smaller than was assumed pre-swap, so the constant-padding caveat is structurally weaker, not stronger). Cross-subject lift-magnitude comparisons (Qwen lift vs Claude lift) carry this static-overhead delta; F-009 reports it as a transfer-probe caveat. No mitigation needed; named is sufficient.
    - **DYNAMIC overhead.** Claude is heavily agentic-trained and may attempt a tool call even with `--tools ""`, burning a turn before delivering a text response. The flailing rate is **prompt-dependent**: a context-poor C prompt may trigger more "I'm missing information, let me look" reflexes than a context-rich A tree. If A and B trigger materially different flailing rates within the Claude arm, the within-Claude **cost-lift gate** (the second of two headline gates: cost-lift ‚â§ 25%) is regime-distorted rather than measuring what the spec format actually costs ‚Äî a confound on a headline gate, not a stated limitation. Mitigation, three parts: **(1)** a Claude-specific anti-flailing system-prompt suffix (`ClaudeInstrument.CLAUDE_ANTI_FLAILING_SUFFIX`: "tools are disabled in this environment. Respond with the implementation directly as text; do not attempt tool calls.") appended to every Claude call's system prompt ‚Äî cuts the flailing at the source; constant within Claude across A/B/C, so the within-subject headline remains safe; named as an instrument-level intervention on the cross-subject comparison. **(2)** `--max-turns 5` (not 1) lets Claude recover with a text-only response in the residual cases where the steer doesn't fully suppress a tool attempt. **(3)** a **post-trial regime-constancy check** ([`harness/regime_constancy.py`](../benchmark/harness/regime_constancy.py)) that computes per-(subject, condition) distributions of `num_turns`, `tokens_input`, `tokens_output`, `claude_code_total_cost_usd`, `stop_reason`, and error rate ‚Äî turning the within-Claude constancy assumption into a measured fact. Advisory flags fire when max/min cell-mean ratios on any single metric ‚â• 2x or per-condition error-rate spread ‚â• 10pp within a subject; F-009 reports the cell distributions and any flags as evidence the within-Claude cost-lift gate was measuring spec cost rather than a flailing-rate confound. Run on the **pre-sweep batch** (catches a regime problem on N=12‚Äì30 before committing to the full 660) AND post-hoc on the full sweep (recorded alongside F-009).
    - **error_max_turns handling.** Even with the anti-flailing steer + `--max-turns 5`, occasional trials may hit `subtype: error_max_turns`. **Pre-registered rule (v9.1): these trials COUNT AS FAILURES** (the apparatus produced a degraded output; the implementation didn't get delivered; per the help-benchmark scoring philosophy a non-implementation is a fail). **No exclusion; no retry.** Both moves could be condition-dependent (a higher exclusion rate on C would bias the success-lift headline; retries could differ in mean cost between conditions and bias the cost-lift headline). The per-condition error rate is recorded in `instrument_extra` on each `TrialRecord` and reported alongside F-009 as part of the regime-constancy artifact.
    - **Project-memory contamination** ‚Äî the catastrophic version of the harness confound, where Claude Code would auto-load `~/.claude/projects/<repo-hash>/memory/MEMORY.md` and hand the subject the spec ‚Äî is structurally prevented by the isolated-tempdir-cwd discipline in `ClaudeInstrument.complete()` (verified by [`harness/verify_claude_isolation.py`](../benchmark/harness/verify_claude_isolation.py), required PASS before trials count; this is a wiring gate, not a stated limitation). What remains is the static + dynamic overhead deltas above, structural and inseparable from "Claude is invoked through Claude Code, not bare API."
- **Cross-subject lift comparisons also carry a model-specialization delta (coder-vs-general).** The Qwen headline subject pinned at the harness-build commit is **Qwen3-Coder-30B-A3B-Instruct** ‚Äî a code-specialized variant of Qwen3. The Claude transfer-probe is **Opus 4.7** ‚Äî a general-purpose frontier model. The specialization difference is *ecologically apt* (a code-specialized model IS representative of the real consumer of a `game-design.md` tree, since the spec's outputs target a coding agent), AND the within-subject A-vs-B paired McNemar headline is unaffected by it (the comparison is each subject against itself). But the **cross-subject lift comparison** (Qwen lift vs Claude lift) now differs along two axes ‚Äî capability tier AND specialization ‚Äî so a difference in lifts could reflect either. F-009 names "coder-vs-general specialization" alongside the static/dynamic harness-overhead deltas as a transfer-probe caveat; the cross-subject comparison remains informative but should not be read as a pure capability-tier finding when the headline subject is a coder variant.

F-009's claim must be scoped to exactly what is measured.
