## Claim under test

Providing a coding agent with a conformant `game-design.md` tree improves first-pass implementation correctness, **by enough to justify the added token/step cost**, relative to providing the same information without the structure.

## Refutation condition (F-009 negative form, stated before data exists)

The standard's structure does NOT earn its complexity at v0.2 if **either gate fails** on the headline task set (medium + hard + ambiguity probe, A-vs-B comparison, paired McNemar):

1. **Success gate.** The point estimate of `success_rate(A) − success_rate(B)` on the headline task set is **below the power-derived MDE floor** (25pp; see §"Power analysis"), OR the lower bound of its 95% confidence interval is below 0 (i.e., the sign of the effect is not robust).
2. **Cost gate.** The point estimate of `cost(A) − cost(B)` exceeds **25% of `cost(B)`** (i.e., the spec adds more than 25% to the agent's token consumption). The 25% is the cost level at which the ETH/LogicStar prior found context files barely moved success — beyond it, the spec is paying for something the data is not showing it earned.

If either gate fails on the headline comparison, F-009 is the negative result reported with the measured numbers and CIs. **The project does not edit either gate value to find a positive.** A-vs-C and B-vs-C are reported as context — including the case where A beats C but not B (which would say the standard's value is the information you put in it, not the format) — but they do not affect the headline judgment.

## Power analysis (pre-registered)

**Design.** Paired binary outcomes per task instantiation. Each (task type × game) instantiation is run under conditions A and B (and C); the A and B outcomes on the *same* task are paired for McNemar's test. C is unpaired baseline.

**Calculation.** For McNemar's two-sided test at α = 0.05, power = 0.80, the required count of discordant pairs `n_d` to detect a true alternative `π1` (= P(A=success | A≠B)) is approximately `n_d ≈ (z_α/2 + z_β)² / (2π1 − 1)²` with `z_α/2 + z_β ≈ 2.80`. At N=20 paired observations with an *expected discordance rate* `π_d` of 30% (i.e., A and B disagree on outcome on ~30% of tasks), expected `n_d ≈ 6` — well below the count needed for any moderate alternative.

**Minimum detectable effect.** Expressed as a success-rate difference, the MDE at N=20 paired (expected `π_d ≈ 30%`) is approximately:

> MDE_paired ≈ 2 · √(π_d / N) ≈ 2 · √(0.30 / 20) ≈ **0.245 ≈ 25 percentage points**.

At lower discordance rates (`π_d < 0.20`), the effective MDE is *higher*; at higher discordance rates, lower. The observed discordance rate will be reported alongside the result, and if it falls below 0.20 the floor is the effective MDE at the observed discordance, not the pre-registered 25pp (this widens the gate honestly toward the data, in the direction *against* the spec).

**Floor for the success gate: 25pp at N=20 paired.** This is what the design can honestly detect. It is much larger than the ETH/LogicStar prior's ~4pp effect; the benchmark is structurally incapable of refuting a small-but-real effect like the ETH one. **The benchmark answers "does the standard produce a medium-or-larger effect"; it does NOT distinguish "no effect" from "small (ETH-like) effect."** F-009 must scope its claim to this.

## Test subjects

- **Headline subject:** local **Qwen**, *instrument bundle pinned at harness-commit time* and recorded in F-009. The instrument is **NOT just the model name** but the full bundle: model name + variant + exact quantization + GGUF revision SHA + llama.cpp (or other inference engine) build version + sampling parameters (temperature, top-p, top-k, repeat penalty, max-tokens) + chat-template / reasoning-format handling. A change to any field after pinning is a new instrument and requires a new pre-registration cycle. **The pre-registered claim is scoped to that exact bundle** — single-instrument is a stated limitation of F-009, alongside same-author-twice (F-007) and Blueprint (F-006). "Different Qwen variant," "different quant," "different build" are all *different instruments*, and the F-009 result does not transfer across them without a new run.
- **Transfer-probe subject:** Claude (the spec-author agent), pinned to a specific Claude version at harness-commit time, labelled as transfer probe (NOT "optimistic ceiling"). Tests how spec-helpfulness varies with model capability tier. Three outcomes are *all* informative:
  - **Claude lift > Qwen lift** — spec adds more for capable models; the structure encodes complexity the smaller model can't yet leverage.
  - **Claude lift ≈ Qwen lift** — spec helps roughly regardless of capability tier; the help is a property of the structure, not the subject's ceiling.
  - **Claude lift < Qwen lift** — frontier models infer the structure unaided; the spec's marginal value declines with capability. (This is a real finding, not a disappointment — it would say the spec is most useful for capability-constrained agents, which is itself a clear scope on who the standard is for.)
- Both subjects run the same task set, conditions, and rubric. No per-subject parameter tuning. The Qwen result is the headline; the Claude result is reported with the same rigor.

## Task set

Four task types instantiated on each of two fresh games (genres not in the current `examples/` set, so the spec was never shaped around them).

### Fresh games

1. **Platformer** — single-player, real-time, level-design-driven, minimal RNG. Tests the standard against a game whose dynamics are *not* probabilistic. Working title authored in the game-spec authoring step.
2. **Survival/crafting** — single-player, resource-graph-driven, tech-tree-heavy, balance-critical. Tests deep dependency graphs and economy scaling.

**Low-RNG framing is deliberate.** Both fresh games deliberately under-exercise the spec's stochastic-determinism surface (PRNG pin, integer-native distributions, the #12–#15 saga). That surface is *already validated by construction* via Phase 4++'s byte-identical cross-engine trajectory (F-007). The help-benchmark deliberately tests the territory the spec has *not* faced: real-time dynamics with no PRNG, and resource-graph design with no random sampling. A roguelike or RNG-heavy game added here would duplicate what tick-combat already covers; the value is in testing the non-stochastic frontier of the spec, which is the part F-004/F-007 cannot speak to.

## Conditions

For each task, run the subject under three information conditions built from the *same* underlying facts. The **A-vs-B comparison is the headline gate**; A-vs-C and B-vs-C are reported context.

- **A — full tree.** Conformant `game-design.md` tree (core + relevant subfiles + content for the fresh game) plus `AGENTS.md` and `CLAUDE.md`.
- **B — flattened prose.** Identical information content of A, mechanically flattened into a single unstructured prose document (see §"B-construction"). No frontmatter tokens, no `{ref}` syntax, no canonical section structure, no cross-file pointers. Same facts, same numbers, same rules.
- **C — minimal prompt.** A one-paragraph description of the game (the "vibe coding" baseline). NOT derived from A — written from scratch alongside the game spec, frozen before any trial.

What each comparison isolates:

- **A vs. B (headline, gated):** does the structure earn its complexity? Holds information constant.
- **A vs. C (context):** total value of the doc (structure + information together).
- **B vs. C (context):** does unstructured context even help? (ETH/LogicStar prior says it can go negative.)

A result where **A beats C but not B** is reported explicitly — that says the standard's value is the *information you put in it*, not the format, and points the v0.3 effort at simpler doc formats. The gate does not declare this a positive.

## B-construction (mechanical + verified + fairness-audited)

Three layers of protection against rigging A-vs-B via an unfaithful B.

### Layer 1: Mechanical flattening (deterministic script)

A flattener script — committed *after* this pre-reg but *before* trial zero, with its SHA frozen at trial start — performs the following deterministic transformation on a `game-design.md` tree:

1. **Walk** the tree in the order given by the core file's `files:` map: core → each subfile → each content-schema file → each content-entity YAML.
2. **For each YAML frontmatter block:** emit prose sentences that name every key and value. Inline-expand all `{namespace.id}` references to their resolved values. Drop structural delimiters; keep the values.
3. **For each Markdown prose body:** emit verbatim (this *is* the rationale layer; B keeps the prose).
4. **Group output by topic, not by file boundary.** All resources together, all rules together, all loops together — what a competent human prose-author would produce when asked to summarize the spec into a prose brief. (See "Design intent" below.)
5. **Drop:** the `files:` map, `file_type:` discriminators, `last_verified:` dates, `status:` lifecycle markers, `implemented_in:` glob arrays, canonical `## Tokens`/`## Rationale`/`## Open Questions` section headers. **Keep:** every number, every rule's `do:` steps, every distribution, every balance target, every state-machine transition, every entity property.
6. **Concatenate** the per-topic outputs into a single prose document.

**Design intent of the flattener (the fairness pin).** The output MUST read as a coherent prose brief — facts grouped by topic, sentences flowing naturally, no token dumps in arbitrary order, no structural-syntax remnants. Concretely: "what a competent person who knew the game and wrote a 5-page prose summary for an implementer would produce." This is the fairness contract; the completeness verifier is layered on top.

### Layer 2: Information-completeness verifier (machine-checkable)

A separate verifier diffs every tree against its flattened B and asserts:

1. **Every distinct numeric value** in A's frontmatter appears at least once in B's prose, as a digit-for-digit literal string.
2. **Every distinct `{namespace.id}` reference** in A has its *resolved value* appearing in B.
3. **Every `rules.<id>.do[]` step** in A appears in B as recognizable prose (verb name or structured-action expansion).
4. **Every distribution `type`** declared in A appears in B by name.
5. **Every `balance_targets.<id>.target` value** appears in B with its measure.
6. **Every state-machine transition** in A (a from/to/event triple) appears in B as a prose statement.

The verifier exits non-zero on any miss. A B that does not pass the verifier is not used in any trial. The verifier's output (which assertions ran, that all passed) is recorded with each B and shipped alongside the F-009 report.

### Layer 3: Fairness audit (one eyeball before flattener SHA freezes)

Before the flattener's SHA is frozen for trial use, the auxiliary LLM judge (§"Judge") audits **one full B** (the flattened platformer spec is the canonical test case) and answers:

> "On a 1–5 scale, is this a fair prose brief — facts grouped by topic, readable as a coherent summary — or a strawman: incoherent, token-dump-style, or pathologically disorganized?"

- **Score ≥ 4 required to freeze the flattener SHA.** Below 4, the flattener is revised and re-audited.
- The audit prompt, the audit output, and the score are recorded as artifacts in the harness-build commit.
- A second sample B (the flattened survival/crafting spec, once authored) is audited under the same criterion as a sanity check before trial zero. Failure on either invalidates the flattener.

Completeness is machine-checkable; fairness needs one eyeball before lock. Both gates are passed before trial zero.

## Judge (multi-layer + leak-checked)

### Layer 1: Objective intent checklists (machine-scored)

For each task brief, a **pre-registered checklist of binary criteria** (between 3 and 7 items per task) is committed before trial zero. Examples:

- Medium task (platformer): "Does the new verb appear in at least one loop's `sequence:`?"; "Does it carry a valid `feel:` ref?"; "Does it transition between two declared states?"
- Hard task (survival/crafting): "Does the new system declare at least one new `balance_targets.<id>`?"; "Are all `do:` steps structured (no bare strings)?"; "Does the loop carry a `timescale:`?"
- Ambiguity probe: "Does the chosen number for X fall within the doc's declared range for similar entities?"; "Is the chosen mechanic referenced from at least one existing token?"

The checklist for each task is the objective-criterion gate; scoring is machine-mechanical (booleans). The checklist is part of the harness-build commit and frozen at the same SHA.

### Layer 2: Subjective remainder ("matches intent")

The irreducible subjective criterion — does the output match the *spirit* of the brief, not just the checkable bits? — is scored by:

- **Auxiliary LLM judge.** Pinned to a non-Qwen, non-Claude-family model (Gemini, GPT-4o, or local Llama variant; specific model pinned at harness-build time). Scores each blinded output on the 0–5 rubric below.
- **User consensus.** User scores the same blinded outputs on the same rubric. The judge-of-record is the consensus when LLM and user agree; on disagreement, the user breaks the tie with the LLM's reasoning visible.

### Rubric for "matches intent" (0–5, fixed in this commit)

| Score | Definition |
| --- | --- |
| **5** | Fully meets the brief. Every design choice is defensible against the brief and the doc's prose rationale (where applicable). No unsupported additions. |
| **4** | Mostly meets. One small design choice defensible but non-optimal; no unsupported additions. |
| **3** | Partially meets. Either one significant gap OR one unsupported addition. |
| **2** | Mostly off. Multiple gaps or multiple unsupported additions. |
| **1** | Off. Doesn't address the brief, or contradicts the doc's pillars/non_goals. |
| **0** | No usable output (refusal, empty, syntactically broken to the point of unscorable). |

A trial **passes the subjective criterion** at score ≥ 4. A trial **passes overall** if it passes the objective checklist AND scores ≥ 4 on the subjective rubric. Success rate in F-009 is the fraction of trials passing overall.

### Layer 3: Blinding-leak check (before trials count)

A-condition outputs may carry spec vocabulary, token names, structural habits that an attentive judge could use to infer the condition — silently un-blinding the judge and reintroducing the invested-author confound.

**Procedure (before trial zero):**

1. Run a **calibration sample** of 30 outputs (10 each from A, B, C) on a single task instantiation.
2. Present them to the auxiliary LLM judge in random order with all condition labels stripped.
3. Ask: "For each output, predict which condition it came from (A, B, or C)."
4. **Pass criterion:** the auxiliary LLM's predictions are no better than chance (33% expected; 95% CI on accuracy includes 33%).
5. **Fail criterion:** predictions exceed chance → blinding leaks → outputs must be sanitized (e.g., strip spec-token-syntax from A outputs, strip prose-style markers from B outputs) before trials begin. Repeat until calibration passes.

The calibration is pre-registered (this section); the calibration outputs and the auxiliary LLM's predictions are recorded as artifacts. Trials do not count until calibration passes.

## Cost metric

Recorded every trial:

- **Tokens:** total input + output tokens consumed by the subject. Includes system prompt, condition's information payload (A/B/C content), brief, and the subject's responses (including any tool-use round trips).
- **Tool-steps:** count of tool calls made by the subject during the trial.

Headline cost number is **mean total tokens per trial**, per (task type × game × subject × condition) cell. Cost-lift in the gate is `mean_tokens(A) / mean_tokens(B) − 1`; the gate fails at > 0.25 (25% lift).

## Sample size and statistical reporting

- **N = 20 paired per (task type × game × subject)** for medium/hard/ambiguity. **N = 5 paired** for easy (sanity check, not headline).
- **Total headline trials per subject:** 3 task types × 2 games × 20 N × 2 conditions (A, B) = **240 paired outputs per subject** for the headline gate. C adds 60 unpaired outputs per subject (3 × 2 × 20 × 1). Easy adds 5 × 2 × 3 = 30 outputs per subject. **Per-subject total: 330 outputs.** Two subjects (Qwen headline + Claude transfer probe): **660 outputs total.**
- **Effect-size reporting.** Every success rate reported with a 95% **Wilson CI** (small-sample-friendly). Every cost-lift reported with **bootstrap CI** over per-trial cost samples. Every paired A-vs-B comparison reported with the **McNemar test statistic and exact p-value**, alongside the point estimate and CI of the success-rate difference.
- **F-009 framing.** Given the N=20 design, F-009 is explicitly a **small-N pilot reporting effect sizes with confidence intervals**, not a statistical-significance claim in the classical-hypothesis-test sense. The pre-registered gate (25pp floor, 25% ceiling) is the *decision rule*; the *report* is the effect size with its CI, whichever way the gate lands. A directional positive at p < 0.05 strengthens the claim; a directional negative or null does not undo the gate's verdict — the gate is what was pre-registered.

## Protocol

1. **This pre-reg commit lands.** Supersedes `f76f4c2`. Sets the new gate.
2. **Fresh games authored.** Two `game-design.md` trees (platformer, survival/crafting), conformant to v0.2.0-alpha. Authored from scratch; NOT derived from any current `examples/` game. Each carries `status: draft` (no implementation exists for the benchmark).
3. **C-prompts authored.** One-paragraph prompts for each fresh game, written from scratch (NOT derived from A). Frozen before trial runs.
4. **Per-task objective intent checklists authored.** 3–7 binary criteria per (task type × game). Committed at the same SHA as the harness.
5. **Flattener + verifier built and committed.** Reviewed against the §"B-construction" criterion. Fairness audit (Layer 3) performed; flattener SHA frozen only at audit score ≥ 4.
6. **B's constructed** via the frozen flattener; verifier asserts information-completeness on each.
7. **Auxiliary LLM judge pinned** (specific model + version) at this step.
8. **Blinding-leak calibration run.** 30 outputs across A/B/C on a single calibration task; auxiliary LLM predicts conditions; pass criterion is no-better-than-chance. Sanitization until pass.
9. **Trial harness built.** Parametrizes (subject, condition, task, game). Records tokens, tool-steps, output per trial. Shuffles condition order; blinds outputs for review.
10. **Qwen instrument pinned** at harness-build time. The instrument is the full bundle (per §"Test subjects"): model name + variant + exact quantization + GGUF revision SHA + inference-engine build version + sampling parameters + chat-template / reasoning-format handling. Same step pins the Claude transfer-probe version. All fields recorded together in F-009.
11. **Instrument calibration smoke run.** Before any trial counts, both pinned instrument bundles (Qwen headline + Claude transfer probe) are exercised on a **pre-registered canary prompt** — a small fixed coding task committed at the harness SHA — and the output is checked for three pass criteria:
    - **Structural validity.** Thinking blocks (where the instrument uses a reasoning mode) parse cleanly: open/close tags balanced; no mid-token truncation; no encoding errors; no repeated-token gibberish; the output reaches its max length OR terminates on a clean stop token. Catches the known instrument-fault class — malformed `<think>` blocks, tokenizer/template mismatches, build-specific gibberish under specific CUDA / GPU configurations — that would silently corrupt trials without showing as an obvious error.
    - **Within-instrument determinism.** Two back-to-back invocations of the same prompt with the same seed produce byte-identical outputs. (Mirrors the cross-engine byte-identity discipline at the simulation layer.) A non-deterministic instrument means trial-level noise the design has no power to detect.
    - **Rubric reachability.** The auxiliary LLM judge scores the smoke output ≥ 4 on the "matches intent" rubric — a sanity check that the instrument can hit the rubric at all on a known-trivial task. A 0/5 here would mean the instrument is broken at the output layer, not that the spec is unhelpful; trials at this configuration would conflate instrument failure with spec failure.

    Trials do not count until **all three criteria pass for both instruments**. The canary prompt, the configurations, the smoke outputs, the judge's scores, and the determinism diff are recorded as artifacts in the harness-build commit. A criterion failure halts the benchmark until the instrument is fixed or replaced; any change to the pinned bundle is a new pre-registration cycle.

12. **Trials run.** N=20 paired (5 easy) per cell; deterministic seeds where applicable; subject parameters frozen.
13. **Judge reviews blinded outputs.** Auxiliary LLM + user consensus on subjective; objective checklist machine-scored.
14. **F-009 reported.** Computed success rates with Wilson CI per cell; A-vs-B headline McNemar + paired CI; A-vs-C and B-vs-C context; per-task / per-game breakdowns; cost-lift with bootstrap CI; observed discordance rate. The pinned instrument bundles are reported in full alongside the result. Pre-reg gates evaluated against the measured numbers. F-009 published whichever way the numbers land.

## What this benchmark will NOT establish (named here so F-009 cannot over-claim)

A clean positive on this benchmark does NOT prove:

- **A small-but-real effect like the ETH prior's ~4pp.** The N=20 paired design has MDE ≈ 25pp; below that, the design cannot distinguish "no effect" from "small effect." F-009's null result, if it lands, refutes only "medium-or-larger effect," not "any effect."
- **Generality across model families.** Single Qwen variant is the headline subject; F-009's claim is explicitly scoped to that variant. The Claude transfer probe extends the picture across one capability tier, not across families.
- **Generality across genres beyond two.** Two fresh games is the resource budget; v0.3+ may extend.
- **Independent-implementer convergence.** Same-author-twice (F-007) is not exercised by this benchmark.
- **The §6 Blueprint-graph-absorbs-design question.** No visual-graph subject is used; F-006's caveat persists.

F-009's claim must be scoped to exactly what is measured.
