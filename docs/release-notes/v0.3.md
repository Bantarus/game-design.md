# `game-design.md` v0.3 — Release Notes

> **The living development surface release.** v0.3 closes the two expressiveness gaps surfaced by Phase 5 (F-008 per-instance state, F-010 verb-centric friction in non-turn-based games), ships the lifecycle-state expansion, anti-staleness lint rules, pre-commit hook + atomic `touch`, project dashboard view, six per-genre starter templates, and the three-mode agent companion, and records the validation-bar reframe (premise-correction discipline) as decision-of-record D-021. Released 2026-05-29. Pre-stable (v1.0 is the planned stable lock).

## What v0.3 is

v0.3 is the release that *moves the project from "developing the format" to "the format exists, positioned for live adoption to inform v0.4."* That is a real state transition. Most of the v0.4 questions — the longitudinal living-doc property, cross-agent transfer of the three-mode operating lens, the M1/M2 watch-items from the AGENTS.md adversarial probe, the spec→code anti-drift direction, the deferred status states — are blocked on live use feeding back, not on more in-repo work. v0.3 ships the vocabulary and apparatus those questions would test against; the test itself awaits the first live adopter.

The arc of v0.3 is a methodological one as much as a vocabulary one. It started under one stated validation bar ("at least one live project") and ships under a corrected one (three claims validated from in-repo evidence; one claim queued for v0.4+). The correction is recorded as decision-of-record [D-021](../../DECISIONS.md) and surfaced in the spec text at [§11.2](../spec.md). The lineage from the original bar through the corrected bar is reconstructible from those artifacts — that is what audit-lineage discipline does.

## The three validation claims v0.3 ships under (from in-repo evidence)

Spec §11.2 names these explicitly; this section restates them for the release audience.

### 1. Vocabulary closure

The closed-vocabulary additions land cleanly across the 6 in-repo trees without local invention. Specifically:

- **`{clocks.<id>}` namespace** (§4.7) — first-class time-passage primitive distinct from player verbs. Closes F-010. Three independent fresh-game trees (tick-combat, Embergrave, Driftwood) had been modeling time-passage as a synthetic verb (`verbs.advance_tick`, `verbs.advance_world_time`, etc.) that existed only to satisfy the spec's verb-triggers-rule pattern. v0.3's `clocks` namespace closes the gap with two modes — `continuous` (fixed-rate; tick-combat's tick, Embergrave's 60 Hz physics) and `per_verb_delta` (advances after each player verb; Driftwood's world clock) — and three retro-touches replace the synthetic verbs.

- **`instance_container` entity type** (§4.1) — N owned instances each carrying per-instance runtime state. Closes F-008. The v0.2.0-alpha three-layer vocabulary (`actor` + `content_collection` + `resources`) had no way to express "N owned instances each carrying per-instance runtime state" — the gap forced authoring workarounds in survival inventories (Driftwood), RPG parties (party-rpg), TCG board states (tcg), and tick-combat's deployed units. v0.3's `instance_container` closes the gap with `capacity:` + `holds_template_from:` + `per_instance_state:` sub-schema; four trees were retro-touched to use it.

- **Addressing DSL for instance_container** (§3, §4.5, [D-019](../../DECISIONS.md)) — binding semantics over existing `{actor.<field>}` / `{target.<field>}` refs through a documented lookup order (`per_instance_state` → template → container properties). Zero new syntax; a spec-level normative declaration of binding semantics. Writes restricted to `per_instance_state` fields; `write-to-template-field` lint rule enforces.

- **`experimental` and `deferred` lifecycle states** ([§8.1](../spec.md), [D-020](../../DECISIONS.md)) — lateral non-canonical-path states. `experimental`: code exists, design under active evaluation. `deferred`: lifecycle progression paused. Closes the vocabulary around four trees' observed need; the watch-for-v0.4 candidate `blocked` is deferred pending live evidence.

**Validation:** all 12 in-repo trees lint clean at v0.3 — 0 errors, 0 warnings — under default thresholds. The retro-touches of tick-combat, party-rpg, tcg, Embergrave, Driftwood landed without local syntax invention; the spec vocabulary holds.

### 2. Cross-engine determinism preserved through the additions

**Validation:** tick-combat's `gdmd verify` adapter gate (§9.5) clears byte-identical to the v0.2 golden trajectory at `seed=12345`, with negative-control divergence at `seed=99999`, after BOTH F-008 (retro-touch with the addressing DSL fully spec'd) AND F-010 (clocks namespace replaces synthetic verbs) land. The hardest gate the spec has runs green across the v0.3 additions.

The PASS is **descriptive-not-prescriptive evidence** that the new vocabulary *names* observable shape engines already have. xtreme's ECS components carried per-instance `hp` / `lifecycle` before F-008; the retro-touch was structural-rename, not behavioral-change. F-010's `clocks.tick` named what xtreme's tick-scheduler was already doing. The PASS would have been a surprise only if v0.3 had imposed shape engines must now conform to — the disciplined choice was to name observable reality, and the verify-adapter PASS is what that discipline looks like when it works.

### 3. Session-level maintenance

**Validation:** the agent performs the anti-drift ritual ([AGENTS.md](../../AGENTS.md)) end-to-end on the in-repo trees when actively prompted. The three-mode operating lens (authoring / operating / maintenance) was developed empirically across the v0.3 commit chain itself — the agent's behavior on the commits that produced v0.3 IS the population the three modes were extracted from. Every v0.3 commit interleaved all three modes; the lens names the modes already operating.

v0.3 also ships the **maintenance apparatus that makes the ritual mechanical**:

- `gdmd hook install` + `gdmd hook check` — pre-commit-framework registration; informational hook surfaces affected spec sections at commit time (not a gate).
- `gdmd touch` — atomically bumps `last_verified:` while preserving author quoting/formatting (regex-on-frontmatter, not pyyaml round-trip; one-line diffs).
- `gdmd status` — project dashboard projecting status counts, stale-sections, shipped-stale, active-without-impl.
- Anti-staleness lint rule family — `stale-section` (status-aware, configurable thresholds), `prototyped-without-pointer`, `shipped-stale-doc`. Defaults grounded in reasonable maintenance-cadence assumptions, not in-repo `last_verified` distribution (avoids circular calibration).

The maintenance ritual is now operationally complete in-repo: hook surfaces sections at commit time → touch bumps `last_verified:` atomically → stale-section lint catches drift if the bump was skipped → `gdmd status` projects the aggregate health snapshot.

## What's queued for v0.4+ (and why)

These are the claims and surfaces v0.3 explicitly does NOT validate. Each is a scope statement, not a failure; v0.3 ships the vocabulary and apparatus the claim would test against. The test itself is gated on live adoption feeding back.

### Longitudinal living-doc property

**The claim.** The spec stays current across weeks/months of game development, where the spec is one tool among many and the human isn't continuously reviewing it.

**Why v0.3 cannot validate it.** The 6 in-repo trees are *spec-illustrations and benchmark targets, not games-in-development*. The `gdmd status` snapshot shows the shape: most files at `status: draft`, a handful at `prototyped`, exactly one tree (tick-combat's xtreme reference impl) with `implemented` entries. That distribution is consistent with their role as spec evidence; it is not the population that would validate longitudinal maintenance.

**What would validate it.** A real adopter authoring a `game-design.md` tree alongside a real game-in-development, where the spec sits as a tool in a multi-tool workflow and the maintenance ritual is exercised under non-ceremonial pressure. The vocabulary and apparatus v0.4+ would test against — anti-staleness lint rules with configurable thresholds, the status lifecycle states (including `experimental` and `deferred`), the transition graph, `gdmd status`'s drift detection — all land at v0.3.

### Cross-agent transfer of the three-mode operating lens

**The claim.** The three-mode operating lens transfers to other coding agents (and to humans operating without agent assistance).

**Why v0.3 cannot validate it.** All operational work was performed by Claude Code instances. The 12-probe adversarial survey of AGENTS.md was authored by the agent it audits.

**What would validate it.** Cross-agent runs against the AGENTS.md prohibitions. v0.4+ candidate; the apparatus is the lens itself, which v0.3 ships.

### M1 and M2 watch-items

**The claim.** The two adversarial probes documented at AGENTS.md Task 5 as *legitimate template edges* (rather than defects the template failed to catch) are genuinely edges of the template's load-bearing scope, not gaps to be filled.

**Why v0.3 cannot decide.** The 10/12 probe result is methodologically stronger than 12/12 would have been; documenting M1 and M2 as legitimate edges is the honest reading at v0.3. Whether they remain edges or surface as defects under live use is the empirical question.

**What would resolve it.** Live cases that hit M1 or M2 patterns and either confirm them as edges (the template is right to be silent there) or surface them as defects (the template should have caught them; a new prohibition is justified). v0.4+ candidate.

### Spec → code anti-drift direction

**The claim.** A workflow shape where editing the spec automatically surfaces implementation areas that may need updating, inverse of v0.3's commit-side hook.

**Why v0.3 does not ship it.** Task 4 ships the code → spec direction via pre-commit hook + `stale-section` lint. The inverse direction is a different workflow shape (closer to design-doc-driven-development than to anti-drift) and adds significant complexity. The minimum-extension discipline that closed F-010's mode enum and D-019's addressing DSL applies here too: v0.3 ships what's demanded by the observed problem; the inverse direction is naturally a v0.4 follow-on if real adoption surfaces the need.

### Status vocabulary expansion (`blocked` and others)

**The claim.** Further lateral states would close gaps observed in real projects.

**Why v0.3 does not ship them.** D-020 closed the v0.3 vocabulary around observed need from 4 trees (`experimental` for "code exists, design under evaluation"; `deferred` for "lifecycle paused, returning later"). `blocked` ("waiting on external dependency") is a candidate but had no in-repo case to ground it. The closed-vocabulary-grows-by-observed-use discipline applies recursively: extending the lifecycle vocabulary without an observed case would be the symmetric form of the gate-loosening trap.

**What would surface a need.** A live adopter's tree that exhibits the friction the current states can't express.

### `scheduled` clock mode

**The claim.** A clock mode that fires at declared in-game-time points or intervals (day/night cycles, wave timers, scripted-event clocks) would close gaps in strategy / survival / event-driven games.

**Why v0.3 ships only `continuous` + `per_verb_delta`.** Those are the modes the project's three real cases (tick-combat, Embergrave, Driftwood) demonstrably needed. `scheduled` is the watch-for-v0.4 candidate — likely to surface in strategy or survival games with explicit waves or daily cadence, but speculative without an observed case.

**What would surface a need.** A live adopter's tree that needs scheduled time-passage and cannot express it cleanly under `continuous` or `per_verb_delta`.

## The validation-bar lineage (the reframe, recorded)

The v0.3 release ships under a validation surface different from the one the v0.3 kickoff set. This section makes the lineage explicit, per the audit-lineage discipline.

### What the kickoff said

The v0.3 kickoff named "at least one live project" as the validation bar. The working assumption was that named external projects (Xenogrid / Mirrorbind / Rêverie) had `game-design.md` spec trees the v0.3 vocabulary would be deployed into. The bar was set against that factual premise.

### What the premise turned out to be

Mid-development, the user clarified those projects don't have spec trees — they were placeholder names for future live adoption, not extant validation targets. The bar was set against a factual premise that wasn't true; correcting the premise necessarily changes what the bar can mean.

### What the corrected bar says

v0.3 ships under three claims validated from in-repo evidence (vocabulary closure, cross-engine determinism preserved, session-level maintenance) plus one claim queued for v0.4+ (longitudinal living-doc property). The three in-repo claims are honestly the three claims the 6 in-repo trees can carry. The longitudinal claim is *queued*, not silently dropped; it is named in §11.2 and in D-021 as the v0.4+-pending bar that awaits the first live adopter.

### Why the distinction is gate correction, not gate loosening

D-021's text states this in full; the short form:

- **Premise-genuine.** The factual correction is objectively verifiable: the named projects don't have spec trees in the repo or anywhere visible. Not a convenient reading.
- **Re-scoping-honest.** The longitudinal claim is explicitly queued for v0.4+, named as a future-resolvable limitation, distinguished from the three claims the in-repo surface does carry. Not silently swapped for weaker evidence.
- **Audit-lineage-preserved.** The premise correction is recorded in DECISIONS.md (D-021), surfaced in the spec text (§11.2), explicated in the F-009 case study, and named in this release note. A future reader can trace the bar's movement from kickoff through correction.

This is the third sister in a discipline family with two prior instances in project history: result-driven gate widening (counterfactual-adoption test) and constraint-driven scope reduction (Phase 5 v12-D, pre-reg commit `2b5d9a6`). The triangulation is documented at [`docs/methodology/README.md`](../methodology/README.md). All three forms name *legitimate* causes for a stated bar to move, with diagnostic tests that prevent that legitimacy from being claimed in cases where the bar just got embarrassing.

## F-009, in case anyone is wondering

The v0.2 help-benchmark trial zero (F-009) reported NULL on success-lift (−5.8pp, McNemar p=0.42, N=120 paired) and FAIL on cost-lift (37.1%, lower-bounded by output-cap truncation) on a single-subject Qwen-Coder configuration, under the locked rule that had been pre-registered before the trial fired. The result was reported as the rule called for, whichever way it landed. The apparatus limitations were named (post-hoc sanitizer leak ~19pp, cap-truncation on hard tasks, bounded-blinding ceiling, single-subject scope).

F-009 is documented in full at [`docs/v0.2-findings.md`](../v0.2-findings.md#f-009--phase-5-help-benchmark-trial-zero-qwen-coder-single-subject-under-v12-d-null-on-success-lift-fail-on-cost-lift-reported-by-the-locked-rule); the case study at [`docs/case-studies/F-009.md`](../case-studies/F-009.md) walks through the supersession chain, the result, the reframe, and the transferable lessons. The case study is *not an apology* — it is a worked example of how to operationalize honestly even when the operationalization returns null.

The F-009 result fed into v0.3 as design input rather than as verdict:
- F-008's expressiveness gap → `instance_container` (closed).
- F-010's verb-centric friction → `{clocks.<id>}` namespace (closed).
- F-009's structural-effect-on-output → the cost-as-amortized clarification: across a multi-session project, the per-session cost-lift amortizes against the spec's anti-drift value over time. The snapshot benchmark could not measure that either way.
- F-009's null on snapshot help-claim → the spec value proposition reframed as longitudinal (gate-correction discipline; D-021).

## Upgrade notes for in-flight trees

If you have a tree at `spec_version: 0.2.0-alpha` that wants to declare v0.3 conformance:

1. Bump `spec_version:` to `0.3.0` in the root `game-design.md` frontmatter (and in every subfile + content-schema + content-entity, if you want strict version consistency).
2. Re-run `gdmd lint <tree>` — it should stay clean. v0.3 adds vocabulary; v0.2.0-alpha trees remain valid under the v0.3 schema (the new vocab is opt-in).
3. If your tree had synthetic verbs like `verbs.advance_tick` or `verbs.advance_world_time` modeling time-passage, consider whether `{clocks.<id>}` is a cleaner expression. The retro-touches in `examples/tick-combat/` and `benchmark/games/{platformer,survival}/` are worked references.
4. If your tree had containers of mutable-state items expressed as content_collections, consider whether `instance_container` is a cleaner expression. The retro-touches in `examples/party-rpg/`, `examples/tcg/`, `benchmark/games/survival/` are worked references.
5. Run `gdmd hook install <tree>` to register the pre-commit hook; run `gdmd status <tree>` to see the aggregate health snapshot.

## What comes next

v0.3 release marks the moment the project moves from "developing the format" to "the format exists, positioned for live adoption to inform v0.4."

The next vocabulary growth is gated on observed-need feedback, which has to come from outside the methodological-discipline session that has carried v0.3. That can be the maintainer authoring spec trees for live projects, external adopters trying the format, or both. There is plenty of v0.4-adjacent work that can happen independently (cross-agent probe setup; the methodology framework's separate publication; whatever live work surfaces); the *next minor* is gated, not the project as a whole.

When v0.4 lands, it will be informed by what live adoption actually surfaced — not by what v0.3 imagined live adoption might surface. That distinction is the whole point of the discipline.
