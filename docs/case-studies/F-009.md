# Case Study — F-009

> **Help-benchmark trial zero (v0.2 Phase 5): NULL on success-lift, FAIL on cost-lift, reported by the locked rule.** A worked example of operationalizing a benchmark hypothesis honestly when the operationalization returns null. The methodology framework this case study illustrates is documented in [`docs/methodology/README.md`](../methodology/README.md); the spec-level scoping consequences are recorded in [`DECISIONS.md`](../../DECISIONS.md) D-021 and [`docs/spec.md`](../spec.md) §11.2.

## Why this case study exists

F-009 is the v0.2 Phase 5 finding that **the `game-design.md` spec format, as authored at v0.2.0-alpha, does not produce a 25pp-or-larger within-subject success-lift over flattened prose** on the Qwen3-Coder-30B-A3B-Instruct headline subject (success-lift = −5.8pp, paired McNemar p=0.42, N=120 paired instances), **and produces a 37.1% cost-lift over flattened prose** (gate: ≤25%, lower-bounded by output-cap truncation on the hard task tier).

The result was reported by the rule that had been locked in pre-registration before the trial fired. The pre-registration had been authored under the explicit constraint that *a rigged positive is worse than an honest negative*. The result returned null on the headline; the rule reported it; the project moved forward under the corrected reading of what had been built.

This case study exists because **how a result like F-009 is handled is itself the discipline worth recording**. Three readings of the data are observable from outside the project's history:

1. *F-009 refuted the claim that the standard is helpful.* Wrong reading. F-009 refuted "the spec format produces a 25pp-or-larger help-effect for Qwen-Coder on the locked task set." The MDE bounds the refutation; a smaller-but-real effect (~4–10pp) is structurally undetectable at the design's paired N=120 and is not what F-009 spoke to.
2. *F-009 was apparatus-limited; the real lift would have emerged with better gear.* Possibly true, but unprovable from F-009 alone. The right response is: report the apparatus limitations as named caveats, queue a future run with the corrected apparatus, and let the corrected run speak for itself.
3. *F-009 measured a different question than the project had been claiming to answer.* This is the reading that produced D-021 and §11.2. The benchmark hypothesis was a *snapshot-correctness* claim about the spec format ("does giving an agent the tree help them produce correct code in a single session"); the project's underlying value proposition turned out to be a *longitudinal living-doc* claim about the artifact ("does the spec stay current alongside code over weeks/months of game development"). These answer different questions, and the snapshot measurement does not bear on the longitudinal claim either way. v0.3 reframed under the longitudinal proposition with the data F-009 had generated (F-008 expressiveness gap, F-010 verb-centric friction, the structural-effect-on-output finding, the cost-as-amortized clarification) feeding back as design input rather than as verdict.

The third reading is the one this case study explicates. **It is gate correction, not gate loosening** — see D-021 for the diagnostic that distinguishes them.

## The claim under test

Pre-registered at v1 of the Phase 5 pre-reg (commit `f76f4c2`, 2026-05-25):

> Providing a coding agent with a conformant `game-design.md` tree improves first-pass implementation correctness, by enough to justify any added token/step cost, relative to providing the same information without the structure.

Operationalized as:
- **Headline metric:** within-subject success-lift A-vs-B, paired McNemar exact two-sided test, MDE ≈ 25pp at paired N=120.
- **Secondary metric:** within-subject cost-lift A-vs-B, paired bootstrap 95% CI, gate ≤25%.
- **Three conditions per task:** A (full spec tree), B (flattened prose carrying the same information without tokens/refs/structure), C (one-paragraph minimal-context baseline).
- **Refutation condition** (locked in v1, never moved): "If the full-tree condition does not beat the flattened-prose condition on success rate by more than the cost increase warrants, the standard's structure does not earn its complexity at v0.2, and that becomes F-009 with the measured numbers."

The pre-reg's lead-in named the failure mode it was built to avoid: *a rigged positive is worse than an honest negative*. The locked rule, the pre-committed refutation condition, and the never-move-the-bar discipline together carried the methodological weight.

## The supersession chain

Pre-registration evolved across sixteen commits (`f76f4c2` → `f05bf0d`) before trial zero fired. Each supersession addressed a discovered gap in the apparatus or scope; **every supersession was a commit, not an intention** (see memory `phase5-pre-registration`); each was named explicitly as superseding the prior version.

Two patterns visible in the chain:

**Disciplines name themselves at supersession boundaries.** Each version's discipline (right column) corresponds to a class of failure mode discovered during the supersession that produced it. v7 named *judge-family independence* because v6 had implicitly assumed the judge could be drawn from the same family as the test subject without bias; v9 named *train/test distribution shift on sanitizer* because v8's calibration had validated the sanitizer on its own sample without checking generalization to the real trial population; v12 named *pre-flight batch carries every load-bearing read* because v11 had shipped a calibration that fit comfortably but hadn't load-tested the heaviest expected workload; v13-D and v14-D were both apparatus corrections discovered after smoke tests revealed stubs in the active code path; v12-D was the first *constraint-driven scope reduction* — a logistical constraint forced reduction of scope without weakening the validation. Each discipline was saved to operating memory at the moment it was named so it would carry into future operations; the running discipline log is the methodology framework's foundation.

**The supersession chain terminates at trial zero, not at PASS.** v14-D was the final supersession because trial zero fired on the v14-D apparatus; the pre-reg explicitly forbade further supersessions once trials began. The terminal condition was *the chain ran out of principled fixes, not the gate passing*. This is the sanitizer-iteration stopping rule applied to the pre-reg itself: iterate while each iteration is diagnosis-driven, closed-enum or structural, and preserves the prior Phase 2 calibration; stop when no more principled fixes remain regardless of whether the gate has passed.

## What F-009 reported

The full F-009 finding lives in [`docs/v0.2-findings.md`](../v0.2-findings.md#f-009--phase-5-help-benchmark-trial-zero-qwen-coder-single-subject-under-v12-d-null-on-success-lift-fail-on-cost-lift-reported-by-the-locked-rule). The headline:

| Gate | Measured | Pass criterion | Verdict |
| --- | --- | --- | --- |
| Success-lift A-vs-B (paired McNemar) | **−5.8pp**, p=0.42 | ≥ 25pp paired | **FAIL** (null on MDE-25pp; refutes "medium-or-larger A-vs-B effect") |
| Cost-lift A-vs-B (paired bootstrap) | **37.1%** [34.5%, 39.8%] | ≤ 25% within Qwen | **FAIL** (lower-bounded; A is cap-suppressed on 60% of trials) |

Paired discordance was 55/120 = 45.8% — well above the floor the design assumed, so the McNemar test had the power its design called for. The null is meaningful within the MDE; it is not a low-power inconclusive.

The post-hoc sanitizer-generalization check (pre-reg v9 step 14) **FAILED** the lift target: judge accuracy on a uniform sample from the full sweep was 0.525, 95% Wilson CI [0.436, 0.612], chance 0.333 — a residual leak of ~19pp above chance. Per the pre-reg's stopping discipline ("post-hoc fail does NOT invalidate F-009; F-009 reports the residual leak as a measured fact rather than retroactively re-running"), the leak was reported as a named limitation, not used to reopen the trial.

Two other limitations were reported alongside the headline:
- The hard task tier hit the output cap on 92.5% of A trials (the v13-D cap was set against the medium-task workload; the load-test had not exercised the heaviest tier — see memory `canary-output-length-representativeness`). The hard-task −52.5pp A-vs-B lift reflects both whatever genuine effect exists AND the cap-truncation distortion; F-009 cannot disentangle them under the locked apparatus.
- The v12-D scope reduction (deferred Opus transfer probe) made F-009 a single-subject Qwen-Coder finding only; cross-subject claims are explicitly out of F-009's scope.

The finding was reported with all of these alongside the headline. *Whatever the numbers said.*

## The reframe

The data F-009 returned was three things at once:

1. **A null on the help-claim as operationalized.** The −5.8pp success-lift refutes "medium-or-larger A-vs-B effect for Qwen-Coder on this task set."
2. **A measured structural-effect-on-output.** Token output was 1.88× higher under A vs B; verbose schema outputs cap-truncated on hard tasks; the cost-lift gate failed at 37.1%. The spec format demonstrably produces different output than flattened prose; the question is whether the difference is *valuable* over the right time horizon.
3. **Two unresolved expressiveness gaps surfaced by fresh-game authoring** (F-008, F-010) — the per-instance owned-item state gap and the verb-centric friction in non-turn-based games. Both were authored *under the v0.2.0-alpha vocabulary* and both surfaced friction during authoring that v0.2.0-alpha couldn't express cleanly.

What the data did NOT speak to:

- **Whether the spec stays current across weeks/months of code development.** The benchmark measured single-session correctness on tasks against trees the spec authors had built; longitudinal maintenance is a different operation on a different time horizon.
- **Whether the spec helps the same agent across multiple sessions** (the case the AGENTS.md three-mode lens is built for — see [`AGENTS.md`](../../AGENTS.md)).
- **Whether the spec's discipline mechanisms** (the closed-vocabulary-grows-by-observed-use principle, the anti-drift ritual, the gate-correction-vs-loosening framework, the supersession chain pattern) **transfer to the artifact's own evolution.** v0.3 became the place this question was answered: see memory `discipline-applies-to-its-own-artifacts` and the spec §11.2 triangulation.

The reframe: v0.3 was constructed under the **longitudinal living-doc proposition** with F-009's outputs as design inputs:

- F-008's expressiveness gap → `instance_container` entity type (closed).
- F-010's verb-centric friction → `{clocks.<id>}` namespace (closed).
- F-009's structural-effect-on-output → the cost-as-amortized clarification: across a multi-session project, the per-session cost-lift amortizes against the spec's anti-drift value over time, which the snapshot benchmark could not measure either way.
- F-009's null on snapshot help-claim → the spec value proposition reframed as longitudinal (gate-correction discipline; see D-021).

The reframe is recorded as:
- **Decision-of-record:** [`DECISIONS.md`](../../DECISIONS.md) D-021 — "v0.3 deployment-surface reframe: premise-correction, not gate-loosening."
- **Spec scoping text:** [`docs/spec.md`](../spec.md) §11.2 — three-claim validation surface for v0.3 + queued-for-v0.4 longitudinal claim.
- **Methodology framing:** [`docs/methodology/README.md`](../methodology/README.md) — the three sister disciplines (gate-correction vs gate-loosening; constraint-driven scope reduction; premise-correction reframe) as a transferable family.

## Caveats inherited from the supersession chain

Each is a measured limitation, not a gate failure; each was named in the v14-D pre-reg before trial zero fired:

- **v12-D bounded-blinding caveat** — the judge has modest discriminative power (Phase 1 lower CI 0.378 = ~4.5pp above chance). Blinding-PASS at any sample size only establishes "tells of magnitude this judge can detect are stripped."
- **v13-D cap caveat** — `sampling_max_tokens=8192` is too low for A on hard tasks (92% cap-hit). The bump from 4096 was informed by the medium-task smoke; it should have load-tested heavier tasks. See memory `canary-output-length-representativeness`.
- **v14-D scoring-stack caveat** — `ChecklistGrader._grade_criterion` was wired at v14-D after smoke #2 surfaced the stub. The 30 archived 6c Claude trials carry stub-checklist values; all 330 Qwen trials in the F-009 dataset were scored under the wired v14-D ChecklistGrader. See memory `comprehensive-stub-audit-before-iterating`.
- **Cost-lift lower-bound** — the measured 37.1% is a *lower bound* on the true A-vs-B cost-lift because A's outputs are cap-clamped at 8192 on 60% of headline cells. The gate FAIL is robust either way.
- **Single-subject scope** (v12-D) — F-009 is a within-Qwen-Coder finding only. The capability-tier hypothesis (pre-reg line 133) and the "frontier-infers-structure-unaided" reading (pre-reg line 99) remain untested; the Opus transfer probe is archived for re-activation in a v0.3+/v0.4+ second wave.

## What F-009 specifically establishes (and does not)

Establishes, within scope:

- Within-Qwen-Coder, the v0.2.0-alpha spec format does NOT produce a 25pp-or-larger success advantage over flattened prose on the locked task set. The null is meaningful within the MDE; a smaller effect (~4–10pp) is structurally undetectable by this benchmark and out of scope.
- The spec format costs more tokens than flattened prose (37%, lower-bounded due to cap-truncation). This is the cost-lift gate's failure direction.
- Per-task heterogeneity is substantial; hard tasks drive the directional negativity and the cap-truncation caveat concentrates there.
- The sanitizer leaks ~19pp residually on the trial population at N=120 — the matches-intent component of trial scoring is partially contaminated; the objective checklist is immune.

Does NOT establish:

- Cross-subject claims (deferred under v12-D).
- "No residual blinding tell remains" — F-009 establishes "after stripping the IDENTIFIED tells, a judge of THIS modest sensitivity could not bring the residual leak below the N=120 detection threshold." Subtler tells remain possible.
- A small-but-real success-lift below ±25pp (structurally undetectable).
- Generality across model families (Qwen-Coder is a single coder-specialized variant).
- Generality across genres beyond two (Embergrave + Driftwood).
- Independent-implementer convergence (same-author-twice was not exercised).
- **Deployment-evidence vs benchmark-evidence non-conflation** (v12-D discipline). Real-world deployment of the spec format with any model is *observational* evidence — fitness-for-purpose, no controlled baseline. F-009 is a controlled within-Qwen-Coder finding; any later observational write-up about deployment experience must be labeled as observational and not folded into F-009's controlled finding.
- The hard-task A finding as a pure spec-format effect (cap-truncation confounded).

## Transferable lessons

The methodology framework documented at [`docs/methodology/README.md`](../methodology/README.md) draws its working examples from the F-009 supersession chain. The lessons that have generalized:

1. **Pre-registration is a commit, not an intention.** Locked thresholds + refutation conditions + apparatus pinning all go into the pre-reg commit before trials fire. The locked rule reports whatever the data says.
2. **A null result is reportable; a rigged positive is not.** Build the gate so it can report null cleanly; do not move the gate after seeing results. The supersession chain ends at trial-zero, not at PASS.
3. **Disciplines name themselves at failure boundaries.** Each supersession discovered a class of failure mode; each got saved to memory at the moment it was named. The running discipline log is what makes the framework transferable.
4. **The three sister disciplines for legitimate bar movement.** When a stated gate moves, the move is legitimate exactly when one of these conditions holds: a *constraint-driven scope reduction* (apparatus or logistical constraint fired AS DESIGNED, not result-driven), a *premise-correction reframe* (the bar was set against a factual premise that turned out false), or a *gate-correction-vs-loosening* test passes (counterfactual-adoption: would you have adopted this gate-shape if the result had been different?). See [`docs/methodology/README.md`](../methodology/README.md) for the diagnostic tests.
5. **The supersession chain is the audit trail.** Future readers should be able to reconstruct exactly why each piece of the apparatus landed where it did. The chain is not noise; it is the methodological work-in-progress made visible.
6. **Operational disciplines apply to their own artifacts.** The closed-vocabulary-grows-by-observed-use principle that closes spec vocabulary (F-008's addressing DSL, F-010's mode enum, D-020's lifecycle states) also closes the agent template's prohibitions (AGENTS.md's three-mode lens) and the per-genre starter content (`templates/starters/`). Reflexive application is the strongest signal a discipline is load-bearing rather than performative. See memory `discipline-applies-to-its-own-artifacts`.

## Refutation conditions for F-009 itself

A future independent re-run with (a) a corrected apparatus (uncapped or much-higher `sampling_max_tokens`; second judge family alongside Gemma; re-activated Opus transfer probe per v12-D's archived design-of-record), (b) a strengthened sanitizer targeting the residual tell the post-hoc judge keyed on, and (c) ideally a third fresh game beyond Embergrave + Driftwood, produces a different headline. The most informative future outcomes:

- A success-lift ≥ 25pp emerges once the cap and sanitizer-leak limitations are addressed → demonstrates the help-claim under stronger conditions; F-009 was apparatus-limited.
- The null persists across the strengthened apparatus → F-009's claim strengthens.
- Cross-subject Opus probe shows different lift direction or magnitude → resolves the deferred capability-tier question.

Any of these is a Phase-5++ or v0.3+/v0.4+ scope item, gated on live adoption surfacing the need for the re-run.

## Status

Demonstrated 2026-05-28, Phase 5 trial zero. Single-subject Qwen-Coder under v12-D scope reduction; v13-D bundle + v14-D scoring stack. Pre-registration locked at v14-D `f05bf0d` since trial zero fired; no further supersessions permitted. **Reported by the rule, whatever the numbers say.**
