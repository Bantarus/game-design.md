# v0.2 Phase 2 — Spec ambiguities surfaced by the xtreme implementation

> Per the v0.2 kickoff: "When you checkpoint Phase 2, the artifact I most want
> is still the list of spec ambiguities the xtreme implementation forced you
> to resolve — that list, not the working build, is the actual yield of the
> phase."

This is that list. Each item is a place where the `examples/tick-combat/`
spec was under-determined — silent on a question the implementation could
not avoid answering.

## Status: 14 resolved (#1–#9 at Phase 2.5, #11 at Phase 2.5+, #12–#15 at Phase 4+); #10 deferred. F-004 closed.

After the user's Phase-2 redirect — **"determinism is binary; anything that
feeds integer simulation state is cross-engine-blocking, regardless of how
minor or rare it looks"** — the triage was re-ordered. Phase 2.5 resolved
#1–#9 in the spec, re-aligned the xtreme implementation to match, and
locked the resulting seed-12345 trajectory as a golden fixture. #10 (verb↔
event emission) remains a true defer because it is a formalization slated
for v0.3 in D-005, not a determinism hole.

At Phase 2.5+, reviewing the Phase 2.5 checkpoint surfaced **#11**: D-012's
`params_from:` pinned syntax + broken-ref handling but left the *binding
moment* (action-start vs apply-time) unspecified — invisible in tick-combat
because no mid-tick attack mutations exist, but a guaranteed Phase-4 divergence
point the first time content introduces a buff/debuff/DoT-that-scales. Closed
in the same pass that produced this section header: apply-time binding pinned
in spec §3 and §4.7, D-012 extended.

The systematic root-cause finding ("rules expressed as prose labels instead
of computable procedures" — #1, #5, #8, #9 are four instances of the same
pattern) became D-011 and the new `determinism-undetermined-rule` lint
advisory. Phase-2 archaeology is now a Phase-1 automated signal — the
standard got better at its own job.

**Read with `examples/tick-combat/gdd/` open.** Tokens cited below resolve
there; the implementation cited resolves under
`examples/tick-combat/impl/xtreme/src/`.

## #1 — `distributions.action_order.sequence` is a *rule*, not a literal sequence

**Where.** `gdd/systems/distributions.md` declares:

```yaml
action_order:
  type: deterministic
  sequence:
    - "highest_speed_first"
    - "tie_break_by_deployment_order"
```

But spec §4.7 documents the `deterministic` distribution type as a "literal
sequence: `[a, b, c, a, b, c]`." The strings in tick-combat are
English-encoded rules an implementer must *interpret*, not literal token
ids. Two engines may interpret them differently — exactly the cross-engine
divergence D-009 forbids.

**Call made.** `impl/xtreme/src/distributions.rs::action_order` reads the
prose strings as the sort spec "speed desc, tie-break deploy_order asc"
and implements that.

**Proposed resolution.** Add a new distribution type to §4.7 — e.g.
`ordering_rule` with a closed vocabulary of declarable clauses (e.g.
`by_field_desc: speed`, `tie_break_by_field_asc: deploy_order`) — *or*
relax `deterministic` to accept either a literal sequence OR a
clause-string sequence drawn from a normative vocabulary. Either choice
must close the "two engines can guess differently" door.

## #2 — `distributions.damage_roll` rounding/clamping order at the boundary

**Where.** `gdd/systems/distributions.md::damage_roll` declares
`clamp: [1, 99]`, `round_mode: half_to_even` (D-010). The spec does not
specify whether clamp happens before or after rounding. At interior
samples this is invisible; at boundary samples (e.g. 99.5) it differs.

**Call made.** `impl/xtreme/src/distributions.rs::damage_roll` does
`sample → clamp(1.0, 99.0) → round_ties_even → clamp(1, 99)` —
clamp-then-round, with a belt-and-suspenders integer clamp after
rounding.

**Proposed resolution.** Spec should canonicalize the order. The
clamp-first form preserves more of the gaussian's tail mass at the
boundary; the round-first form is conceptually simpler. The D-010
addendum implies "round at apply" means the clamp is applied to the
continuous sample (since `clamp:` lives next to `mean/stddev`); make
that explicit.

## #3 — `distributions.critical_hit` threshold inequality direction

**Where.** `gdd/systems/distributions.md::critical_hit`:
`range: [0.0, 1.0]`, `threshold: 0.10`. Crit semantics: returns true iff
sample ?? threshold. Spec doesn't say `<`, `<=`, `>=`, or `>`.

**Call made.** `impl/xtreme/src/distributions.rs::critical_hit` returns
`sample <= threshold` — standard practice.

**Proposed resolution.** Spec should declare `sample <= threshold` (and
in the D-010 integer reformulation, `sample_int <= threshold_int`).
Boundary semantics are measure-zero in the continuous case but matter
in finite-precision float and trivially in the integer reformulation.

## #4 — `distributions.gold_drop` returns category labels, not gold values

**Where.** `gdd/systems/distributions.md::gold_drop` has
`options: { small: 0.6, medium: 0.3, large: 0.1 }`. The implementation
needs to know *how much gold* a "small" drop is worth. The spec is
silent.

**Call made.** `impl/xtreme/src/distributions.rs::GoldDrop::provisional_gold`
hardcodes `(small=1, medium=3, large=10)` and marks them explicitly as
"SPEC AMBIGUITY #4 — these numbers are an interpretation, not a
normative quote."

**Proposed resolution.** Either (a) extend `weighted` distributions so
options can be `{ weight: <p>, value: <int> }` pairs instead of bare
weights, or (b) add a sibling token `gold_per_drop: { small: 1, medium: 3,
large: 10 }` to the balance-targets surface and reference it from
`{rules.combat_resolution}`. (a) is more general; (b) keeps weighted
distributions probabilistic-only and pushes the values into the balance
namespace where they belong.

## #5 — `{rules.tick_resolution}` step 2 "resolve_unit_action": target selection is undefined

**Where.** `gdd/mechanics.md::rules.tick_resolution.do[1]` is the literal
string `"resolve_unit_action"`. This is the entire definition of "the
attacking unit hits something." Who does it hit?

**Call made.** `impl/xtreme/src/rules.rs::tick_resolution` selects the
first alive unit on the opposite side, by `deploy_order` ascending. This
is the simplest interpretation; "lowest hp," "highest threat,"
"random alive opposite," "closest on grid," and several others would
all also be defensible.

**Proposed resolution.** Target selection is a *design lever*, not an
implementation choice. Add a `target_selection:` field to `verbs` or
`rules` (or both) with a closed vocabulary (e.g.
`first_alive_opposite_by_deploy_order`, `lowest_hp_opposite`, `random`).
Without this, two engines may pick different targets for the same seed
and the integer trajectory diverges.

## #6 — Number of gold drops per encounter is unspecified

**Where.** `gdd/mechanics.md::rules.combat_resolution.do` says
`sample {distributions.gold_drop}` and `award_gold_to_winner`. Singular.
But `balance_targets.gold_per_encounter: 14` with tolerance `[10, 20]`
and the gold_drop weights/values above can't be satisfied with one drop
— expectation per drop is ~2.5 gold with the (1, 3, 10) values, so the
target of 14 implies ~5–6 drops per encounter on average.

**Call made.** `impl/xtreme/src/rules.rs::combat_resolution` does one
drop per encounter. The simulation hits `gold=3` typically, far below
the 14 target.

**Proposed resolution.** Declare the per-encounter drop schedule.
Options: (a) fixed number (`drops_per_encounter: 6`); (b) variable
(`drops_per_encounter: { distribution: ... }`); (c) scales with the
encounter (e.g. one drop per dead enemy). The balance target's tolerance
band implicitly mandates *some* answer; right now it just floats.

## #7 — How are enemy squads composed?

**Where.** `gdd/mechanics.md::entities.player.properties.roster_size: 5`
sets the player roster. There is no `enemy_squad` or `encounter` token.
The spec implies "Lockstep is two squads, a hex grid, and ~120 ticks"
in the root file's prose, but never says how the second squad is
specified.

**Call made.** `impl/xtreme/src/sim.rs::deploy_demo_roster` mirrors the
player roster onto the enemy side as a placeholder. This is the worst
form of "design value in the impl" — the demo invents the enemy squad.

**Proposed resolution.** Add a content type `encounters` with frontmatter
shape something like:

```yaml
encounters:
  shock_canyon_01:
    seed_offset: 0
    enemy_units:
      - { id: shock_titan, deploy_order: 0 }
      - { id: volt_marine, deploy_order: 1 }
      ...
    twist: { ... }   # optional rules modifiers
```

Then `examples/tick-combat/content/encounters/*.yaml` holds the real
encounter library. The harness picks an encounter and feeds it to the
implementation. **This is the single biggest spec gap surfaced by Phase
2.** Without it, every implementation must invent encounter content.

## #8 — `{entities.units}.attack` is not wired to `{distributions.damage_roll}`

**Where.** `gdd/content/units.md::schema.properties.attack: integer, [0, 50]`.
Unit stats include `attack`, but `{distributions.damage_roll}` is
gaussian(mean=5, stddev=1) — fixed parameters that ignore the attacking
unit's `attack` value. Volt Marine's `attack: 5` happens to match the
gaussian mean, which looks intentional, but the spec doesn't say so.

**Call made.** `impl/xtreme/src/rules.rs::tick_resolution` rolls
`damage_roll(rng)` without consulting `unit.attack`. Damage is uniform
across all unit types — Shock Titan (`attack: 12`) deals the same damage
as Spark Drone (`attack: 2`). This is almost certainly wrong; the test
runs nonetheless because the simulation terminates.

**Proposed resolution.** Either (a) the damage distribution parameters
read from the attacking unit's stats — e.g. `damage_roll: mean:
{actor.attack}, stddev: 1, clamp: [1, 99]` — which requires
templated parameters in the distribution declaration; or (b) damage is
deterministic in `actor.attack` and the gaussian is a separate
"variance" distribution multiplied or added in. Pick one; the current
spec is silent and both are defensible.

## #9 — `{entities.units}.speed` semantics: dominate or rotate?

**Where.** `gdd/systems/distributions.md::action_order` says "highest
speed first." Read literally, the highest-speed alive unit acts *every
tick* until it dies — the others never act. Read as "speed determines a
rotation," it's an Initiative-style game where speed 10 means three
acts to speed 3's one.

**Call made.** `impl/xtreme/src/distributions.rs::action_order` reads
"highest speed first" literally — the same fastest alive unit acts on
every tick. In the demo, this means the volt_marine (speed 7) hammers
the enemy volt_marine across most ticks; shock_titan (speed 3) never
attacks while volt_marine is alive. The combat resolves in ~18 ticks,
which suggests the literal reading is wrong (intended pace is ~120
ticks per the median balance target).

**Proposed resolution.** Spec must clarify "highest speed first" — is
it (a) "fastest alive acts every tick" (current literal reading), or
(b) "fastest alive acts first this tick, then everyone else takes a
turn before the round restarts," or (c) "speed determines initiative
points consumed per action and units act in initiative order." Pick
one; the balance target's `120 tick median` implies (b) or (c), not (a).

## #11 — `params_from` binding moment is unspecified (surfaced at Phase 2.5 checkpoint review)

**Where.** D-012 added `params_from: { mean: "{actor.attack}" }` on the
gaussian `damage_roll` distribution. The spec pinned syntax + broken-ref
handling for `{actor.<field>}` and `{target.<field>}` but was silent on
*when* the value is read relative to other mutations in the same firing.
Two readings produce different integers the moment any mid-firing
mutation exists:

- **Action-start binding.** When the rule firing begins, snapshot the
  relevant context fields and use the snapshot throughout the firing.
- **Apply-time binding.** At the specific `do:` step that consumes the
  reference, read live from the world.

In tick-combat as it stands today, no content can modify `{actor.attack}`
between action-start and the `sample:` step — so both readings produce
identical trajectories, and the gap stayed invisible through Phase 2.5.
The xtreme implementation happens to read `actor.attack` from a tick-start
snapshot (`impl/xtreme/src/rules.rs::tick_resolution` snapshots all units
at the top of each tick; `actor_unit.attack` reads from the snapshot at
the sample step). It also mutates `{target.hp}` *live* via
`world.get_mut::<Unit>(target)` — asymmetric, but silently correct under
tick-combat's no-mid-tick-mutation invariant.

**The risk.** Phase 4's Unreal Blueprint port (or any future tick-combat
content with mid-firing mutations — a buff applied earlier in the same
tick, a damage-over-time that scales) would pick the *other* reading
and the canonical integer trajectory would diverge, masquerading as a
spec bug. This is the exact #5-class invisible-assumption failure mode
the disambiguation pass exists to prevent — pinned now rather than
discovered in Phase 4.

**Call made (in Phase 2.5+).** Spec normatively pins **apply-time
binding** for both `{actor.<field>}` and `{target.<field>}` and any
future context-local prefix. Reads happen at the specific `do:` step,
live against the world. Engines MAY internally snapshot when they can
prove no in-firing mutations affect the reads — tick-combat's xtreme
implementation is permitted to keep its tick-start snapshot because
tick-combat has no mid-tick attack mutations and the snapshot is
provably equivalent to a live read. When future content introduces
mid-firing mutations, snapshot-based engines must refactor to live
reads.

**Why apply-time and not action-start.** Action-start has surface
appeal ("a unit's whole action uses the values it began with") but
breaks symmetry with `{target.<field>}` (which every engine reads live,
otherwise accumulated damage doesn't kill) and requires an implicit
per-firing snapshot data structure in every engine. Apply-time has the
simplest mental model (refs always read what's true right now),
composes with intra-firing mutations, and matches what every existing
engine already does for target reads.

**Spec footprint.** §3 (binding-moment paragraph under the context-local
prefixes section), §4.7 (apply-time clause under the `params_from:`
subsection). D-012 in `DECISIONS.md` extended with a "Binding moment —
apply-time" subsection covering symmetry, composability, and the
snapshot-optimization carve-out.

**Out of scope at v0.2.0-alpha.** A normative escape hatch for "use the
value as it was at action-start" (e.g. a `snapshot:` `do:` step kind
that captures a context field into a `{local.<name>}` ref) — that is a
v0.3+ concern when actual content needs it.

**Trajectory impact.** None. xtreme's existing tick-start snapshot is
permissible under the apply-time normative rule (no mid-tick mutations
exist to falsify the equivalence). The locked seed-12345 trajectory is
unchanged.

## #10 — `{verbs.start_combat}` and `{events.start_combat}` share a name without an explicit relation

**Where.** D-005 placed `events` in its own namespace, so a verb and an
event can share a bare-string id without collision. `{verbs.start_combat}`
(emitted by the player) and `{events.start_combat}` (drives the
`{states.combat_phase}.setup → ticking` transition) appear to refer to
the same logical event but the spec doesn't *say* the verb emits the
event.

**Call made.** `impl/xtreme/src/sim.rs::Simulation::start_combat` does
both jobs — issues the verb conceptually and steps the
`{states.combat_phase}` machine on `{events.start_combat}`. No
declarative trace of the connection exists in `gdd/`.

**Proposed resolution.** The v0.3 ratchet for D-005 already names this:
"introduce an `events` namespace and add the cross-check that every
event a state reacts to is *emitted* somewhere by a verb's `effects`
or a rule's `outputs`." Add an `emits:` field to verbs and rules, and
have the linter check that every `{events.<id>}` referenced by a
transition is emitted by at least one verb or rule. Phase 2 makes this
real: without `emits:`, the spec cannot declare that
`verbs.start_combat` is what produces `events.start_combat`, and two
engines could realize the link differently (or not at all).

## #12 — Spec does not pin the PRNG algorithm (surfaced by Phase 4 / Godot)

**Where.** `gdd/systems/distributions.md` declares distributions by *type* (`gaussian`, `uniform`, `weighted`, `shuffle_bag`, `pity_floor`, `deterministic`, `ordering_rule`) and `seed: deterministic_per_run`. The spec is silent on which PRNG algorithm produces the sample stream. xtreme uses ChaCha20 keyed by seed (chosen for portability per the v0.2 kickoff addendum); the engine-B Godot reference uses Godot 4's built-in `RandomNumberGenerator`, which is a PCG-family generator. Same seed value (12345) → entirely different sample streams → trajectories diverge at the very first sampling call.

**Phase-4 evidence.** At seed=12345:

| Engine | Tick-1 enemy.volt_marine hp | Total ticks | Terminal gold |
| --- | --- | --- | --- |
| xtreme (ChaCha20) | 19 (damage 1) | 23 | 26 |
| Godot (PCG-family) | 18 (damage 2) | 17 | 6 |

Both samples (1 and 2) are valid draws from `gaussian(mean=2, stddev=1, clamp=[1,99], round=half_to_even)`. The spec permits both. The implementations don't disagree on the spec; they disagree on the underlying randomness, which the spec doesn't constrain.

**The risk.** This was the dominant Phase-4 failure mode (per the discipline framing: "engine B will diverge, the question is whether the spec permits the divergence"). Without pinning the PRNG, the cross-engine integer-trajectory bar (D-009) is structurally unsatisfiable — any two engines using different PRNG libraries will diverge regardless of how carefully they implement everything else.

**Proposed resolution (deferred to a Phase 4+ commit per the discipline: surface gap first, lock spec separately).** Spec §4.7 must declare a normative PRNG algorithm + seeding contract for any `verify_target` with cross-engine trajectory expectations:

```yaml
# Proposed addition to gdd/systems/distributions.md frontmatter,
# or to per-distribution declarations
prng:
  algorithm: chacha20            # closed vocabulary at v0.3
  seeding: seed_from_u64         # how the integer seed maps to PRNG state
```

Closed vocabulary candidates (v0.3): `chacha20` (cryptographic, well-implemented in every language), `pcg32`/`pcg64` (fast, simple, widely available), `xoshiro256pp` (fast, contemporary). The spec picks one as the canonical default; per-distribution overrides are spec-author choices.

**Trajectory impact (when fixed).** The golden re-locks. Godot's adapter implements the chosen PRNG (or whichever the spec pins) instead of `RandomNumberGenerator`. xtreme's golden may or may not change depending on which PRNG is picked (if ChaCha20 is canonical, no change; if PCG is canonical, xtreme's golden re-locks).

**Adjudication.** *Spec gap, not engine-B bug.* xtreme's ChaCha20 and Godot's PCG are both valid spec-compliant choices; the spec under-constrains the PRNG. Per the discipline framing, **do not patch Godot to use ChaCha20**. Fix the spec and re-lock the golden in a separate commit.

**RESOLVED at v0.2.0-alpha Phase 4+ (D-015, 2026-05-23).** PRNG pinned to `xoshiro256_starstar + splitmix64` as the default; per-game `prng:` declaration in §4.7; closed vocabulary at v0.2.0-alpha (`xoshiro256_starstar | chacha20 | pcg32 | pcg64`); reference vector at canonical_seed=0 is the self-validation hook. Both engines now hand-roll the PRNG (no crate/library dependency) and self-check against the spec's vector at adapter startup. xtreme's ChaCha20 dependency removed; Godot's built-in `RandomNumberGenerator` no longer used.

## #13 — Spec does not pin the gaussian sampling algorithm (surfaced by Phase 4 / Godot)

**Where.** §4.7 declares `type: gaussian` with `mean`, `stddev`, `clamp`. It is silent on the algorithm that maps PRNG output to gaussian samples. Common algorithms — Box-Muller transform, Marsaglia polar method, Ziggurat — consume different numbers of PRNG outputs per gaussian sample, and the same PRNG with two different algorithms produces two different gaussian streams.

**Phase-4 evidence (transitive).** xtreme uses Rust's `rand_distr::Normal`, which is Marsaglia polar at the time of writing. Godot's `RandomNumberGenerator::randfn` is a Box-Muller-family implementation. Even if both engines used the same PRNG, the gaussian samples would still diverge because they consume PRNG state at different rates.

**Proposed resolution (deferred).** Spec §4.7 must declare a normative sampling algorithm per stochastic distribution type:

```yaml
# Proposed addition to spec §4.7 per-type sub-spec
gaussian:
  sample_method: marsaglia_polar   # closed vocabulary at v0.3
  # alternatives: box_muller, ziggurat
```

Marsaglia polar is the recommended default — well-defined, widely implemented, and consumes a deterministic number of PRNG outputs per sample (2 uniforms → 1 gaussian, with potential rejection retries that ALSO need to be specified). The ziggurat method is fastest but has algorithm-specific tables that vary across libraries.

**Adjudication.** *Spec gap*. Resolution belongs alongside #12.

**RESOLVED at v0.2.0-alpha Phase 4+ (D-016, 2026-05-23) — folded with #15.** Continuous gaussian is no longer the cross-engine path. New `type: discrete_sum` (integer-native: sum *k* PRNG draws each uniform integer in `range`, add `mean`, clamp) replaces float-gaussian-with-rounding for state-affecting use. The algorithm-choice question is eliminated entirely — no continuous sampling step exists, so there is no algorithm to pin. `tick-combat::damage_roll` migrated: `samples: 3, range: [-1, 1], params_from.mean: {actor.attack}, clamp: [1, 99]`. The continuous `gaussian` distribution type is reserved for cosmetic/non-cross-engine use.

## #14 — Spec does not pin the iteration order of `weighted.options` for cumulative-sum sampling

**Where.** `{distributions.gold_drop}` declares `options: { small: {weight: 0.6, value: 1}, medium: {weight: 0.3, value: 3}, large: {weight: 0.1, value: 10} }`. Cumulative-sum weighted sampling needs a fixed iteration order: do you walk `small → medium → large` (yielding cumulative thresholds 0.6, 0.9, 1.0) or some other order?

**Phase-4 evidence.** xtreme iterates in declaration order (Rust source code's field order). Godot's GDScript Dictionary iteration order matches insertion order, which matches the YAML map order — same answer. Both engines happened to converge, but YAML map order is preservation-dependent; a future engine using a hash map (or even xtreme in a refactor that switches to `HashMap`) would diverge silently.

**Proposed resolution (deferred).** Spec §4.7 must pin the canonical iteration order for `weighted.options`. Two candidates:

- **Declaration order:** the order keys appear in the YAML. Requires the spec to constrain the YAML loader to preserve order (PyYAML 5.1+ does; the loader at `src/game_design_md/loader.py` should be audited for guarantee).
- **Sorted-by-key order:** alphabetical iteration. Predictable, but loses author control over presentation ordering.

Declaration order is the more author-friendly default. Either choice closes the gap; the spec must pick.

**Adjudication.** *Spec gap*. Trajectory impact at seed 12345: none, because both Phase-4 engines happen to agree by accident.

**RESOLVED at v0.2.0-alpha Phase 4+ (D-017, 2026-05-23).** Normative `selection_rule: declaration_order_first_above` on every `weighted` distribution:
1. Integer weights summing to `W` (float weights forbidden for cross-engine trees).
2. Draw `d = rng.next_u64() mod W`.
3. Walk options in YAML declaration order with running cumulative sum `c`.
4. Select the first option whose `c > d` — **strict greater-than**.

The strict `>` also pins the boundary-comparison question implicit in cumulative-sum sampling (same class as #3's `<=` for crit). `tick-combat::gold_drop` migrated: integer weights `{small: 60, medium: 30, large: 10}` summing to 100, `selection_rule: declaration_order_first_above`. The accidental Phase-4 agreement is now a structural requirement.

## #15 — libm transcendental ULP drift would break Phase-4 byte-identity even with PRNG + algorithm pinned (forecast by user during #13 review)

**Where.** Continuous gaussian sampling depends on `log` / `exp` / `sin` / `cos` somewhere (Marsaglia polar uses `log`; Box-Muller uses `log` + `sin`/`cos`; Ziggurat uses `exp` for the tail). IEEE-754 does NOT mandate correctly-rounded transcendentals — real libm implementations (Rust's, Godot's, MSVC's under a hypothetical Unreal port) differ in the last ULP. The integer rounding (`round_mode: half_to_even`, D-010) occasionally flips when a sample lands within ULP-distance of an x.5 boundary. Rare, unpredictable.

**The forecast.** Made during the user's adjudication of #13: "even with the algorithm pinned, transcendentals differ in the last ULP." Naming #15 *before* Phase-4+ implementation kept the count at one re-lock — pinning `marsaglia_polar` for #13 alone would have shipped a "byte-identical except occasionally" trajectory and required a second re-lock once #15 surfaced.

**RESOLVED at v0.2.0-alpha Phase 4+ (D-016, 2026-05-23) — folded with #13.** Replacing continuous gaussian with integer-native `discrete_sum` eliminates the libm dependency at the source. No `log`, `exp`, `sin`, `cos`, `sqrt`, or `pow` in the simulation path. The `output_domain` + `round_mode` paragraph in D-010 is deprecated for state-affecting use; both remain in the schema for backward compatibility and for legitimate cosmetic uses (presentation jitter, non-cross-engine noise). Phase-4 byte-identity at seed 12345 verified across xtreme (Rust) and Godot (GDScript), no transcendentals on either path.

**Headline finding (D-011).** #1, #5, #8, #9 were four instances of the
same root cause: `rules.<id>.do[]` items written as bare prose strings
("resolve_unit_action", "award_gold_to_winner") instead of typed
computable steps. Phase 2.5 adds:

- **D-011 declaration** in `DECISIONS.md` framing the principle.
- **Spec §4.5 requirement** that every `do[]` item on a deterministic loop
  path SHOULD be a structured object, not a bare string.
- **New linter rule `determinism-undetermined-rule`** (advisory at
  v0.2.0-alpha, warning in v0.3, error in v0.4) that flags bare-string
  `do[]` steps in rules reachable from any `{loops.<id>}` with
  `timescale: moment`. The rule canNOT prove a procedure is total or
  cross-engine-stable, but it can flag the signal at the canonical
  position — the "a human must confirm" nudge.
