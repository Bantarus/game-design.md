# v0.2 — Findings

> Demonstrated properties of the `game-design.md` standard surfaced during the v0.2 pass. Each finding is a property that v0.1.1 *claimed* but had not yet *shown*, or a property the standard discovered about its own behavior in the course of being used. Findings are **demonstrations, not checkmarks** — each cites the artifact that produced it and the artifact that would refute it.

Findings track open claims too. F-001..F-003 are demonstrated at v0.2.0-alpha; F-004 is the open testable claim Phase 4 will decide.

## F-001 — Genre-agnostic claim is real, not aspirational

**Property.** The §6 content-schema machinery introduced at v0.1.1 absorbs unanticipated content types without modification to `docs/spec.md` or `schema/game-design.schema.json`. Adding a new content type to an existing example requires one new content-schema file, one or more content-entity YAMLs, and one line added to the core file's `files:` map. Lint stays clean.

**Demonstrated by Phase 2 ambiguity #7.** During Phase 2 the xtreme implementation had no way to interpret `enemy_pack_size` without an explicit declaration of *which* units compose each pack. The resolution was authored entirely within the existing §6 surface:

- New content-schema: [`examples/tick-combat/gdd/content/encounters.md`](examples/tick-combat/gdd/content/encounters.md) (12 lines of frontmatter, 4 lines of schema).
- New content-entity YAMLs: [`examples/tick-combat/content/encounters/shock_canyon_01.yaml`](examples/tick-combat/content/encounters/shock_canyon_01.yaml), [`shock_canyon_02.yaml`](examples/tick-combat/content/encounters/shock_canyon_02.yaml).
- One new line in [`examples/tick-combat/game-design.md`](examples/tick-combat/game-design.md)'s `files:` map.

No change to `docs/spec.md`. No change to `schema/game-design.schema.json`. No change to the CLI. `gdmd lint examples/tick-combat` continued to emit 0 errors and 0 warnings.

**Why this matters.** A genre-agnostic claim is cheap to assert and expensive to demonstrate. The standard claims the seven primitives + content-schema pattern are *universal*. The first real test of that claim is "show us an unanticipated content type slotting in." Encounters is exactly that — a content type the spec authors didn't pre-imagine, with composition semantics specific to tick-combat (squad composition by faction and side), absorbed by the existing surface without spec authors touching the spec.

**Refutation condition.** If a future content type needed in `examples/{deckbuilder,party-rpg,tcg}/` cannot be authored without a change to §6 or `schema/`, this finding weakens. The §11.1 success benchmark (an agent implements a new content entity in a single session ≥80% of the time) is the standing version of this test.

**Status:** demonstrated 2026-05-22, Phase 2.5.

## F-002 — The determinism advisory is a standing signal, not a one-shot

**Property.** Phase-2 archaeology — finding "rules expressed as prose labels instead of computable procedures" by hand-grepping a single example — converts into a Phase-1 automated signal via the lint rule `determinism-undetermined-rule` (D-011). The rule fires not only on the example it was discovered in (tick-combat, now resolved), but **retroactively** on every other example with the same latent pattern, including examples nobody has implemented yet.

**Demonstrated by the post-Phase-2.5 lint sweep.** With tick-combat resolved (every `do:` step now structured), the advisory falls silent on tick-combat. It still fires on three other examples that share the same root-cause pattern but were never implemented:

```
=== deckbuilder ===
  - rules.end_of_turn.do[0]:  bare-string 'tick_burning_on_each_enemy'
  - rules.end_of_turn.do[1]:  bare-string 'discard_remaining_hand'
=== party-rpg ===
  - rules.end_of_turn.do[0]:  bare-string 'tick_status_effects'
  - rules.end_of_turn.do[1]:  bare-string 'regenerate_mp'
=== tcg ===
  - rules.card_play_resolution.do[0]:  bare-string 'pay_mana_cost'
  - rules.card_play_resolution.do[1]:  bare-string 'apply_card_effects'
```

These six gaps were *latent in the spec since v0.1.1* and nobody noticed because nobody had implemented the examples. The advisory flagged them retroactively the moment it shipped, with zero additional spec-author effort.

**Why this matters.** The standard's failure mode is drift between spec and implementation. The traditional path closes drift one example at a time, after the implementation surfaces the gap. The advisory inverts the path: the spec ratchets *all* examples toward determinism-completeness without waiting for each to be implemented. This is "the standard getting better at its own job" — a one-time discovery becomes a permanent enforcement, applied uniformly across examples.

**Governance call (Phase 2.5+).** The deckbuilder is the §11.1 benchmark target and the canonical reference example — the example people copy. Its two advisories were resolved in the same pass that produced this document by restructuring the bare strings to typed `foreach` steps and adding a `discard_at_end_of_turn` event + `card_lifecycle` transition for proper state-machine wiring. The party-rpg and tcg advisories remain — they are *instructive examples* of the advisory's intended use (visible until resolved) and the project does not promise to resolve them until those examples are implemented.

**Refutation condition.** If a future spec change weakens the advisory's reach (e.g. by accepting bare strings as "implicitly structured" in some context), the standing-signal property degrades. The v0.3 / v0.4 ratchets (warning → error) are scheduled to *strengthen* it.

**Status:** demonstrated 2026-05-22, Phase 2.5+.

## F-003 — In-engine determinism is non-vacuous

**Property.** The cross-engine determinism bar (D-009) reduces to two contracts:
1. **Within an engine:** byte-identical replay across runs at the same seed.
2. **Across engines:** identical canonical integer trajectory at the same seed.

The within-engine contract risks passing *vacuously* — a single-engine determinism test will trivially pass if the engine is deterministic by construction (e.g., no async, no system clock, no thread races). The risk is that the test silently never exercised the seed at all.

**Demonstrated by the golden trajectory's negative control.** [`examples/tick-combat/impl/xtreme/tests/golden_trajectory.rs`](examples/tick-combat/impl/xtreme/tests/golden_trajectory.rs) ships three tests:

- `golden_trajectory_seed_12345` — every Rust build reproduces the locked seed-12345 trajectory line-by-line.
- `determinism_two_runs_match` — same seed twice in a single test produces byte-identical output.
- `different_seeds_diverge` — **the negative control.** Seeds 12345 and 99999 produce *different* trajectories.

The negative control is what makes the determinism test non-vacuous. Without it, a broken implementation that simply ignores the seed and emits a hardcoded trajectory would pass both positive tests. With it, ignoring the seed is detectable by the negative control failing.

**Why this matters.** Phase 3's `verify` adapter contract (§9.5) inherits this discipline: any future engine implementing the cross-engine contract must produce a trajectory that *responds* to the seed input. The negative-control idiom belongs in the `verify_targets` spec, not in each adapter's local code, so Phase 4's Unreal Blueprint port cannot accidentally ship a vacuously-passing replay test.

**Refutation condition.** If a Phase-3 adapter is written that does not include a negative control, or if the spec's `verify_targets` schema cannot express one, F-003 weakens. Phase 3 should make the negative-control pattern first-class in §9.5.

**Status:** demonstrated 2026-05-22, Phase 2.5.

## F-004 — Cross-engine integer trajectory bar is the open Phase-4 testable claim

**Open property.** Two engines, A=`xtreme` (Bevy ECS, Rust) and B=Unreal Blueprints, implementing the *same* `examples/tick-combat/gdd/` tree, produce the *same* canonical integer trajectory at the *same* seed, modulo engine-specific serialization. This is D-009's Phase-4 contract.

**State at end of v0.2.0-alpha.**

- **Engine A (xtreme):** built, locked golden at seed 12345 (23 ticks, gold=26). [`examples/tick-combat/impl/xtreme/tests/golden_seed_12345.txt`](examples/tick-combat/impl/xtreme/tests/golden_seed_12345.txt) captures the per-tick trajectory.
- **Engine B (Unreal Blueprints):** not yet started. Standing up a headless deterministic Unreal sim is itself the first big Phase 4 milestone (per D-008 cost note).
- **Trajectory serialization format:** currently *xtreme-stdout-shaped*, not engine-neutral. Phase 3 must canonicalize the format independent of how xtreme happens to print — the `VerifyResult` contract and the trajectory format are themselves engine-neutral spec, not bespoke xtreme artifacts. If xtreme-isms leak into the interface, Phase 4 inherits them.
- **All known disambiguation closed.** Phase 2.5 resolved #1..#9; Phase 2.5+ closed #11 (`params_from` binding-moment, pinned apply-time). #10 (verb↔event emission) is a true v0.3 defer per D-005, not a determinism hole.

**What would refute it.** Phase 4's Unreal port produces a different integer trajectory at seed 12345 than xtreme's locked golden. If the divergence traces to a spec ambiguity, that is the *intended* yield of Phase 4 — surface it, log it as a new ambiguity, resolve in the spec, re-align both impls, re-lock. If the divergence traces to *no* spec ambiguity (the spec fully determined the answer and Unreal got it wrong), that is an engine-side bug.

**What would weakly refute it.** Phase 4's Unreal port matches the trajectory but only because the Blueprint graph absorbed gameplay logic that should have stayed in `.md`. Per D-008, holding the line that the graph is a pure *consumer* of the spec — every number, rule, balance value, state transition stays in `.md`, the graph points back via `implemented_in:` — is the Phase-4 anti-drift result. A trajectory match plus a Blueprint-absorbed-the-design failure is a partial win.

**Status:** open. The locked golden + the adapter contract + the apply-time binding decision are the testable surface Phase 4 will probe.

## Appendix — Findings convention

Each finding has a stable ID (`F-NNN`), a property statement, a citation of the artifact that demonstrates (or would demonstrate) it, an explanation of why it matters, a refutation condition, and a status line. Findings are appended chronologically and never removed; if a finding is refuted, it is annotated `**Status: refuted at <version>, see <new finding>**` so the trail of what was *claimed* and what was *actually shown* remains legible.

`DECISIONS.md` records *what we decided*; `v0.2-findings.md` records *what we proved about those decisions*. They are complementary, not redundant.
