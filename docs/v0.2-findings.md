# v0.2 — Findings

> Demonstrated properties of the `game-design.md` standard surfaced during the v0.2 pass. Each finding is a property that v0.1.1 *claimed* but had not yet *shown*, or a property the standard discovered about its own behavior in the course of being used. Findings are **demonstrations, not checkmarks** — each cites the artifact that produced it and the artifact that would refute it.

Findings track open claims too. F-001..F-003 are demonstrated at v0.2.0-alpha; F-004 is the open testable claim Phase 4 will decide.

## F-001 — Genre-agnostic claim is real, not aspirational

**Property.** The §6 content-schema machinery introduced at v0.1.1 absorbs unanticipated content types without modification to `docs/spec.md` or `schema/game-design.schema.json`. Adding a new content type to an existing example requires one new content-schema file, one or more content-entity YAMLs, and one line added to the core file's `files:` map. Lint stays clean.

**Demonstrated by Phase 2 ambiguity #7.** During Phase 2 the xtreme implementation had no way to interpret `enemy_pack_size` without an explicit declaration of *which* units compose each pack. The resolution was authored entirely within the existing §6 surface:

- New content-schema: [`examples/tick-combat/gdd/content/encounters.md`](examples/tick-combat/gdd/content/encounters.md) (12 lines of frontmatter, 4 lines of schema).
- New content-entity YAMLs: [`examples/tick-combat/content/encounters/shock_canyon_01.yaml`](examples/tick-combat/content/encounters/shock_canyon_01.yaml), [`shock_canyon_02.yaml`](examples/tick-combat/content/encounters/shock_canyon_02.yaml).
- One new line in [`examples/tick-combat/game-design.md`](examples/tick-combat/game-design.md)'s `files:` map.

No change to `docs/spec.md`. No change to `schema/game-design.schema.json`. No change to the CLI. `gdmd lint examples/tick-combat` continued to emit 0 errors and 0 warnings.

**Why this matters.** A genre-agnostic claim is cheap to assert and expensive to demonstrate. The standard claims the seven primitives + content-schema pattern are *universal*. The first real test of that claim is "show us an unanticipated content type slotting in." Encounters is exactly that — a content type the spec authors didn't pre-imagine, with composition semantics specific to tick-combat (squad composition by faction and side), absorbed by the existing surface without spec authors touching the spec.

**Refutation condition.** If a future content type needed in `examples/{deckbuilder,party-rpg,tcg}/` cannot be authored without a change to §6 or `schema/`, this finding weakens. The §11.1 success benchmark (an agent implements a new content entity in a single session ≥80% of the time) is the standing version of this test.

**Status:** demonstrated 2026-05-22, Phase 2.5.

## F-002 — The determinism advisory is a standing signal, not a one-shot

**Property.** Phase-2 archaeology — finding "rules expressed as prose labels instead of computable procedures" by hand-grepping a single example — converts into a Phase-1 automated signal via the lint rule `determinism-undetermined-rule` (D-011). The rule fires not only on the example it was discovered in (tick-combat, now resolved), but **retroactively** on every other example with the same latent pattern, including examples nobody has implemented yet.

**Demonstrated by the post-Phase-2.5 lint sweep.** With tick-combat resolved (every `do:` step now structured), the advisory falls silent on tick-combat. It still fires on three other examples that share the same root-cause pattern but were never implemented:

```
=== deckbuilder ===
  - rules.end_of_turn.do[0]:  bare-string 'tick_burning_on_each_enemy'
  - rules.end_of_turn.do[1]:  bare-string 'discard_remaining_hand'
=== party-rpg ===
  - rules.end_of_turn.do[0]:  bare-string 'tick_status_effects'
  - rules.end_of_turn.do[1]:  bare-string 'regenerate_mp'
=== tcg ===
  - rules.card_play_resolution.do[0]:  bare-string 'pay_mana_cost'
  - rules.card_play_resolution.do[1]:  bare-string 'apply_card_effects'
```

These six gaps were *latent in the spec since v0.1.1* and nobody noticed because nobody had implemented the examples. The advisory flagged them retroactively the moment it shipped, with zero additional spec-author effort.

**Why this matters.** The standard's failure mode is drift between spec and implementation. The traditional path closes drift one example at a time, after the implementation surfaces the gap. The advisory inverts the path: the spec ratchets *all* examples toward determinism-completeness without waiting for each to be implemented. This is "the standard getting better at its own job" — a one-time discovery becomes a permanent enforcement, applied uniformly across examples.

**Governance call (Phase 2.5+).** The deckbuilder is the §11.1 benchmark target and the canonical reference example — the example people copy. Its two advisories were resolved in the same pass that produced this document by restructuring the bare strings to typed `foreach` steps and adding a `discard_at_end_of_turn` event + `card_lifecycle` transition for proper state-machine wiring. The party-rpg and tcg advisories remain — they are *instructive examples* of the advisory's intended use (visible until resolved) and the project does not promise to resolve them until those examples are implemented.

**Adjacent ratchet (Phase 3+).** A second standing signal — `broken-implementation-pointer`, D-002 — also strengthened at v0.2.0-alpha. The rule's intended severity is `error`; v0.1.1 held it at `warning` while no example shipped real source. Phase 2 (xtreme/Bevy ECS) + Phase 3 (verify-adapter binary) satisfied the ratchet condition and the rule promoted to `error`. The draft-status gate (entities below `prototyped` are silent) holds — all four examples lint 0e/0w under the ratchet — which is the structural reason the ratchet was free: design-stage trees keep working, while any future entity promoted to `prototyped+` is force-checked against the file system.

**Refutation condition.** If a future spec change weakens either standing signal's reach (e.g. by accepting bare strings as "implicitly structured" in some context, or by making `broken-implementation-pointer` permissive again), the property degrades. The v0.3 / v0.4 ratchets on the determinism advisory (warning → error) are scheduled to *strengthen* its reach further; D-002 is closed.

**Status:** demonstrated 2026-05-22 (advisory); D-002 ratchet confirmed 2026-05-23 (Phase 3+).

## F-003 — In-engine determinism is non-vacuous

**Property.** The cross-engine determinism bar (D-009) reduces to two contracts:
1. **Within an engine:** byte-identical replay across runs at the same seed.
2. **Across engines:** identical canonical integer trajectory at the same seed.

The within-engine contract risks passing *vacuously* — a single-engine determinism test will trivially pass if the engine is deterministic by construction (e.g., no async, no system clock, no thread races). The risk is that the test silently never exercised the seed at all.

**Demonstrated by the golden trajectory's negative control.** [`examples/tick-combat/impl/xtreme/tests/golden_trajectory.rs`](examples/tick-combat/impl/xtreme/tests/golden_trajectory.rs) ships three tests:

- `golden_trajectory_seed_12345` — every Rust build reproduces the locked seed-12345 trajectory line-by-line.
- `determinism_two_runs_match` — same seed twice in a single test produces byte-identical output.
- `different_seeds_diverge` — **the negative control.** Seeds 12345 and 99999 produce *different* trajectories.

The negative control is what makes the determinism test non-vacuous. Without it, a broken implementation that simply ignores the seed and emits a hardcoded trajectory would pass both positive tests. With it, ignoring the seed is detectable by the negative control failing.

**Why this matters.** Phase 3's `verify` adapter contract (§9.5) inherits this discipline: any future engine implementing the cross-engine contract must produce a trajectory that *responds* to the seed input. The negative-control idiom belongs in the `verify_targets` spec, not in each adapter's local code, so Phase 4's Unreal Blueprint port cannot accidentally ship a vacuously-passing replay test.

**Refutation condition.** If a Phase-3 adapter is written that does not include a negative control, or if the spec's `verify_targets` schema cannot express one, F-003 weakens. Phase 3 should make the negative-control pattern first-class in §9.5.

**Status:** demonstrated 2026-05-22, Phase 2.5.

## F-004 — Cross-engine integer trajectory bar is the open Phase-4 testable claim

**Open property.** Two engines, A=`xtreme` (Bevy ECS, Rust) and B=Unreal Blueprints, implementing the *same* `examples/tick-combat/gdd/` tree, produce the *same* canonical integer trajectory at the *same* seed, modulo engine-specific serialization. This is D-009's Phase-4 contract.

**State at end of v0.2.0-alpha.**

- **Engine A (xtreme):** built, locked golden at seed 12345 (23 ticks, gold=26). [`examples/tick-combat/impl/xtreme/tests/golden_seed_12345.txt`](examples/tick-combat/impl/xtreme/tests/golden_seed_12345.txt) captures the per-tick trajectory.
- **Engine B (Unreal Blueprints):** not yet started. Standing up a headless deterministic Unreal sim is itself the first big Phase 4 milestone (per D-008 cost note).
- **Trajectory serialization format:** currently *xtreme-stdout-shaped*, not engine-neutral. Phase 3 must canonicalize the format independent of how xtreme happens to print — the `VerifyResult` contract and the trajectory format are themselves engine-neutral spec, not bespoke xtreme artifacts. If xtreme-isms leak into the interface, Phase 4 inherits them.
- **All known disambiguation closed.** Phase 2.5 resolved #1..#9; Phase 2.5+ closed #11 (`params_from` binding-moment, pinned apply-time). #10 (verb↔event emission) is a true v0.3 defer per D-005, not a determinism hole.

**What would refute it.** Phase 4's Unreal port produces a different integer trajectory at seed 12345 than xtreme's locked golden. If the divergence traces to a spec ambiguity, that is the *intended* yield of Phase 4 — surface it, log it as a new ambiguity, resolve in the spec, re-align both impls, re-lock. If the divergence traces to *no* spec ambiguity (the spec fully determined the answer and Unreal got it wrong), that is an engine-side bug.

**What would weakly refute it.** Phase 4's Unreal port matches the trajectory but only because the Blueprint graph absorbed gameplay logic that should have stayed in `.md`. Per D-008, holding the line that the graph is a pure *consumer* of the spec — every number, rule, balance value, state transition stays in `.md`, the graph points back via `implemented_in:` — is the Phase-4 anti-drift result. A trajectory match plus a Blueprint-absorbed-the-design failure is a partial win.

**Status:** **partially demonstrated 2026-05-23** (Phase 4, Godot substitute for Unreal). The cross-engine bar is testable end-to-end and produces actionable spec gaps (#12–#14 — see F-006). Closing the bar to byte-identity awaits the Phase-4+ spec-lock commit (PRNG algorithm + gaussian sampling + iteration order pinned). The Blueprint-specific §6 claim about visual graphs absorbing design remains untested — that requires a real Unreal pass.

## F-005 — Pre-Phase-4 format hardening: closing cross-engine desync windows in the spec

**Property.** The trajectory format (§9.5.5) and the `verify` contract (§9.5) are tight enough at v0.2.0-alpha that Phase 4's Unreal Blueprints adapter, when it appears, will surface *simulation* divergences against the locked golden — not *format* divergences. Three latent format-level desync risks were closed at Phase 3+ review, before a second engine could expose them:

| Risk | Failure mode | Resolution at Phase 3+ |
| --- | --- | --- |
| Array element ordering | "Stable" sort wording is conventionally insertion-order-preserving; two engines build arrays in different insertion orders → byte-different trajectories | Spec §9.5.5 now requires arrays to declare a `sort_by:` key list producing a **total order**; arrays are sorted before serialization; string sort keys compare byte-wise on UTF-8 (codepoint order). Insertion order is explicitly disallowed. |
| Integer width | Blueprint defaults to `int32`; an unspecified-width integer in the trajectory could overflow at a Rust↔Unreal boundary | Spec §9.5.5 pins integer values to **int32 by default** (`−2^31 .. 2^31−1`); a game's `trajectory.schema:` may declare `width: int64` per field when wider values are needed. Engines round-trip the declared width. Float fields are forbidden outright. |
| Locale-aware string ops | Rust's `to_lowercase`, .NET `string.ToLower()`, and C++ `std::tolower(int)` under non-default locales apply Unicode case folding (Turkish `İ → i̇`, German `ß → ss`, etc.) → enum strings desync | Spec §9.5.5 requires **ASCII-only lowercasing** for enum serialization (`[A-Z] → [a-z]`, all other codepoints unchanged) and **byte-wise codepoint order** for string sorts. Locale-aware case conversion and collation are forbidden. |

**Why these matter.** Phase 4's expense is the standing-up of headless Unreal + the writing of the Blueprint adapter; Phase 4's value is whatever the cross-engine bar surfaces about the *simulation* spec. If the first Phase-4 divergence is "the array order differs" or "the lowercased enum spells differently under tr-TR locale," the most expensive phase has been spent debugging the wrong layer. F-005 is the up-front close on the failure mode Phase 4's design is meant to avoid.

**Citations.**
- Spec §9.5.5: array sort by total-order key, int32 default + per-field width opt-in, ASCII-only lowercasing, codepoint-order string sorts.
- [`examples/tick-combat/gdd/verification.md`](../examples/tick-combat/gdd/verification.md): `trajectory.schema.units.sort_by: [side, deploy_order]` is a total order (proof in spec §9.5.5 example).
- `verify` status: promoted from experimental to stable at v0.2.0-alpha in spec §9.5 lead (Phase 3 — commit 36e4ff5). No bundled runner; project-supplied adapter contract per §9.5.6.

**Refutation condition.** Phase 4's first cross-engine divergence traces to one of the three risks above (or another format-level ambiguity not yet closed). Each such finding becomes a new entry in [`docs/v0.2-phase2-spec-ambiguities.md`](v0.2-phase2-spec-ambiguities.md) (numbering continues from #11), gets resolved in the spec, and the format ratchets again. F-005 is "true at v0.2.0-alpha, refutable by Phase 4 evidence" — exactly the structure the findings doc is for.

**Status:** demonstrated 2026-05-23, Phase 3+.

## F-006 — Phase 4 outcome: cross-engine integer-trajectory bar surfaces spec gaps as designed

**Property (demonstrated).** Running a second, paradigmatically-distant engine against the same `gdd/` tree produces an executable adjudication procedure: for each cross-engine divergence, either the spec forbids engine B's behavior (engine bug, fix engine) or the spec permits both readings (spec gap, fix spec, re-lock golden). Phase 4 ran end-to-end at v0.2.0-alpha and surfaced three concrete spec gaps with proposed resolutions, with **zero divergences attributable to engine-B implementation bugs**. The bar works as designed.

**Engine-B substitution caveat.** Engine B was *substituted* from Unreal Blueprints (the D-008 pick) to **Godot 4 / GDScript** because Unreal is not installable in this execution environment. The substitution:

- **Keeps** the cross-engine integer-trajectory test (Godot ≠ Bevy ECS, Godot ≠ Rust, Godot has its own PRNG and sampling primitives — every cross-engine determinism risk is exercised).
- **Keeps** the `data_behavior_separation` test in a non-ECS, GC'd actor/object engine (addendum §6's broader question, adapted).
- **Loses** the Blueprint-specific test ("does the visual node graph stay a pure consumer of the spec, or does it absorb design as Blueprints tend to?"). GDScript is text-authored; the temptation to encode design in graph nodes that defines the Blueprint failure mode doesn't apply. **The §6 question about Blueprints as a de facto design document remains untested.** A future Phase 4-Unreal pass would close that gap; nothing in v0.2 closes it.

**Phase-4 trajectory result at seed=12345:**

| Metric | xtreme (engine A, Bevy ECS / Rust / ChaCha20) | Godot (engine B, scene-tree / GDScript / PCG-family) |
| --- | --- | --- |
| Total ticks | 23 | 17 |
| Terminal gold | 26 | 6 |
| Terminal phase | resolved | resolved |
| First divergence | n/a | tick 1 (enemy.volt_marine hp: 19 vs 18) |
| Within-engine determinism | byte-identical (3× runs) | byte-identical (2× runs) |
| Negative-control divergence (alt seed) | trajectory differs from primary | trajectory differs from primary |

Both engines pass within-engine determinism and the seed-response negative control. They diverge cross-engine at the first sampling call — and every divergence traced to one of three spec under-determinations:

**Three spec gaps surfaced (#12–#14 in [docs/v0.2-phase2-spec-ambiguities.md](v0.2-phase2-spec-ambiguities.md)).**

1. **#12 — PRNG algorithm.** xtreme uses ChaCha20; Godot uses its built-in PCG-family `RandomNumberGenerator`. Spec §4.7 declares distribution *types* but is silent on the underlying PRNG. **This was the dominant cause of the trajectory divergence.** Proposed resolution: `prng: { algorithm, seeding }` declaration in §4.7 with a closed vocabulary (chacha20 / pcg / xoshiro); deferred for Phase-4+ spec-lock commit.
2. **#13 — Gaussian sampling algorithm.** Even if both engines used the same PRNG, Marsaglia polar vs Box-Muller vs Ziggurat consume different numbers of PRNG outputs per sample. Proposed: `sample_method:` per stochastic type; recommended default `marsaglia_polar`.
3. **#14 — `weighted.options` iteration order.** Cumulative-sum sampling depends on iteration order; both engines happened to agree at seed 12345 (both use insertion-order maps), but a hash-map-based engine would diverge silently. Proposed: pin to YAML declaration order.

**The discipline held.** Per the Phase-4 framing — "the spec is the judge, the golden is the goal" — none of these were patched in Godot to match xtreme. Each was logged as a spec gap with a proposed resolution path. The actual spec change (re-pinning the PRNG, then re-locking xtreme's and Godot's goldens) is deliberately deferred to a separate commit so the user can adjudicate the closed-vocabulary choices (chacha20 vs pcg, marsaglia_polar vs ziggurat) before the goldens move.

**D-008 dual criterion (adapted to Godot).**

| Criterion | Verdict | Evidence |
| --- | --- | --- |
| 1. Trajectory matches xtreme at seed 12345 | **FAIL** — diverges at tick 1 | All three failure modes are spec gaps (#12–#14), not engine bugs. Re-runnable after Phase-4+ spec-lock. |
| 2. Engine stayed a pure consumer of the spec | **PASS** (with caveat) | [`impl/godot/src/components.gd`](../examples/tick-combat/impl/godot/src/components.gd) — Unit is pure data, no methods other than `_init` and `clone`. [`src/simulation.gd`](../examples/tick-combat/impl/godot/src/simulation.gd) + [`src/distributions.gd`](../examples/tick-combat/impl/godot/src/distributions.gd) — all logic in stateless functions, each function cites the `{spec.token}` it realizes. Hardcoded constants (unit stats, damage clamp bounds, threshold 0.10) match xtreme's pattern — both engines hardcode where the spec lacks a YAML-loader-to-impl pipeline. Treated as legitimate v0.2 impl detail; flagged for v0.3 consideration. **Caveat:** GDScript is text-authored, so the Blueprint-specific failure mode (visual graph absorbs design) wasn't probed. |

**The §6 question — does `data_behavior_separation` survive in a non-ECS, actor/object engine — verdict: yes, in Godot.** Godot's scene-tree is actor/object (`Node` extends `Object`, holds state, has methods), but the spec's discipline ("entities are pure data; logic is stateless systems") was straightforwardly authorable in GDScript by *choosing* not to add methods to Unit. The engine doesn't fight the discipline. This is weaker than what Phase 4-Unreal would have tested — Unreal Blueprints *fight back* against text-authored discipline by making graph nodes the natural authoring surface for any logic. Godot's GDScript is closer to the Rust case (a deliberate author can hold the line); Blueprints would have tested whether the discipline survives an engine that *resists* it. **F-006 cannot confirm or refute the parallel-research §6 claim about Blueprints specifically.**

- Engine-B verify wrapper: [`examples/tick-combat/tools/verify-adapter-godot`](../examples/tick-combat/tools/verify-adapter-godot).
- Engine-B reference trajectory: [`impl/godot/tests/engine_b_seed_12345.jsonl`](../examples/tick-combat/impl/godot/tests/engine_b_seed_12345.jsonl) (18 lines, terminal tick 17, gold 6).

- Adjudication detail per gap: spec-ambiguities #12, #13, #14.

**Refutation condition.** The proposed resolutions (#12–#14) close the gap in a Phase-4+ spec-lock commit: spec pins PRNG + gaussian sampling + iteration order; Godot is re-implemented to use those normative choices; xtreme is updated if the chosen normative algorithm differs from ChaCha20; both engines re-emit trajectories; goldens re-lock; Phase-4 trajectory comparison becomes byte-identical. If after that commit the trajectories STILL diverge, there's a fourth spec gap not yet surfaced — and the discipline repeats. F-006 is "true at v0.2.0-alpha as a procedure-works finding; refutable by a Phase-4+ re-run that surfaces an unrecorded gap."

**Status:** demonstrated 2026-05-23, Phase 4 (Godot substitute for Unreal). F-004 transitions from "open" to **partially demonstrated** — the cross-engine bar is *testable* and produces actionable spec gaps; closing the bar (byte-identical trajectories) awaits the Phase-4+ spec-lock commit. The §6 claim about Blueprints specifically remains untested.

## Appendix — Findings convention

Each finding has a stable ID (`F-NNN`), a property statement, a citation of the artifact that demonstrates (or would demonstrate) it, an explanation of why it matters, a refutation condition, and a status line. Findings are appended chronologically and never removed; if a finding is refuted, it is annotated `**Status: refuted at <version>, see <new finding>**` so the trail of what was *claimed* and what was *actually shown* remains legible.

`DECISIONS.md` records *what we decided*; `v0.2-findings.md` records *what we proved about those decisions*. They are complementary, not redundant.
