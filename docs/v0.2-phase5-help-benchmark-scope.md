# v0.2 Phase 5 — The Help-Benchmark (Scope)

> The deferred Phase 5 of the v0.2 kickoff. Structurally separate from the cross-engine spine: that work proved the spec is a *complete, precise contract* (byte-identity). This proves — or fails to prove — that the spec is *helpful*: that giving an agent the `game-design.md` tree produces better output than not. The ETH/LogicStar prior (developer-written context files: +4% success; LLM-written: −3%; both +20% cost) is the live possibility that the honest answer is "no, or not enough to justify the cost." Design the experiment so a negative result is reportable, not engineered around. A rigged positive is worse than an honest negative.

## The claim under test

> Providing a coding agent with a conformant `game-design.md` tree improves first-pass implementation correctness, by enough to justify any added token/step cost, relative to providing the same information without the structure.

State the **null hypothesis and refutation condition before running** (findings-doc discipline): "If the full-tree condition does not beat the flattened-prose condition on success rate by more than the cost increase warrants, the standard's structure does not earn its complexity at v0.2, and that becomes F-009 with the measured numbers." Pre-register this. Do not move it after seeing results.

## The confound that governs the whole design

You have been the agent-under-test on every spec iteration since Phase 1. The spec is therefore shaped by *your* specific failure modes, and a with-doc-vs-without comparison using you as subject would confirm nothing — it would measure "does a spec tuned to this agent help this agent," which is circular. Two controls are mandatory:

1. **Test subjects must be agents that did not co-evolve the spec.** Headline signal comes from a **different model family** (you have local Qwen inference; a non-Claude subject is the cleanest control, since "helps" must not collapse to "helps the model it was tuned to"). Secondary data points: a different Claude version, and fresh Claude instances with no exposure to the build history. The agent that built the spec may run as an *optimistic-ceiling* data point only, explicitly labelled, never as the headline.
2. **Tasks must be on games the spec was never shaped around.** The four `examples/` games (deckbuilder, tick-combat, party-rpg, tcg) are contaminated — they were co-designed with the spec; testing on them is teaching to the test. Author **2–3 fresh game specs in genres not in the example set**, used only for the benchmark. This measures generalization, not memorization.

## The three conditions

For each task, run the subject under three information conditions built from the *same underlying facts*:

- **A — full tree.** A conformant `game-design.md` tree (core + relevant subfiles + content) plus the `AGENTS.md`/`CLAUDE.md` workflow file.
- **B — flattened prose.** The identical information content of A, flattened into an unstructured prose brief: same facts, same numbers, but no frontmatter tokens, no `{ref}` syntax, no canonical structure, no cross-file pointers. This is the ETH/LogicStar-style context file.
- **C — minimal prompt.** A one-paragraph description (the "vibe coding" baseline).

The comparisons isolate different things, and **A-vs-B is the one that matters**: it tests whether the *structure* earns its complexity, holding information constant. A-vs-C measures total value; B-vs-C tells you whether unstructured context even helps (the direction the ETH prior warns can go negative). If A beats C but not B, the standard's value is the information you put in it, not the format — a result worth knowing.

## The task ladder

The v0.1.1 card task is too easy — it's format-mimicry given a canonical example, which frontier models ace regardless. Use a difficulty ladder so the benchmark can detect where structure starts to pay off:

- **Easy** — author one content entity from a schema + one example (the existing §11.1 task).
- **Medium** — implement a new verb/rule and wire it through the affected loop and state machine (touches `mechanics.md`, `loops.md`, a state machine).
- **Hard** — add a system that spans `mechanics`, `loops`, `economy-balance`, and a state machine, honoring the architecture invariants and producing a deterministic result.
- **Ambiguity probe** — a deliberately under-specified brief ("design a card that synergizes with X," no numbers). Scores whether the agent's choices are *defensible given the prose rationale* — the direct test of the "prose is the fallback when no token covers a case" principle. This is the one task where A's prose layer should most clearly beat B's flat text and C's nothing.

## Metrics — a (success, cost) pair, not a rate

Success per trial is the conjunction of objective criteria (machine-scored wherever possible): schema-valid · lints-clean (0 errors) · refs-resolve · integer-domain honored · for code tasks, builds + passes the `verify` trajectory where applicable + honors declared invariants. The subjective criterion — *matches the brief's intent* — needs a **judge that is neither the test subject nor the spec-author agent**, scoring against a pre-registered rubric (a different model, or you as human reviewer with the rubric fixed in advance).

Record **cost** every trial (tokens and tool-steps). The ETH prior is specifically that context files raise cost >20% while barely moving success; a doc that lifts success 4% at 20% cost is the negative result in disguise. The headline number is therefore success-lift *per unit cost*, not raw success.

## Protocol

- **N ≥ 20 per (condition × task × subject)** — the v0.1.1 N=5 lesson: 5/5 has a ~50–100% CI and proves nothing. Report rates with confidence intervals.
- **Pre-register** tasks, rubric, success threshold, and refutation condition before the first trial. Archive the pre-registration in the commit that starts the run.
- **Blind the judge** to which condition produced each output where feasible.
- Build the scoring harness to be re-runnable (it parameterizes subject, condition, task), and keep trial outputs out of the repo (CI artifact), per the earlier `docs/benchmark/` decision.

## Deliverable

A pre-registered protocol, the harness, and **F-009** stating the measured result with its CI and what each comparison isolates — reported the same whichever way it lands. If A genuinely beats B per-unit-cost on the medium/hard/ambiguity tasks, the standard's reason to exist is empirically supported and v0.2 closes. If it doesn't, that is the most important finding the project will produce, and it tells you exactly where the structure is and isn't carrying weight — which is the input to v0.3.

## What this benchmark still won't establish (name it in F-009)

Even a clean positive here does not prove *independent-implementer convergence* (the same-author-twice limitation, now recorded in F-007) or the §6 Blueprint-graph-absorbs-design question. Those remain open beyond v0.2. This benchmark answers "does the structure help an agent build correctly" — not "do two independent minds converge from the spec alone," and not "does the `.md` hold the line in an engine that natively absorbs design." Scope the claim to exactly what's measured.
